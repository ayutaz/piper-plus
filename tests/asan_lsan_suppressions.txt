# Leak / behaviour suppressions for AddressSanitizer / LeakSanitizer runs in
# CI. Each entry is documented so future maintainers can decide whether the
# masked report is still acceptable.
#
# Format: one rule per line.
#   leak:<substring of stacktrace frame>
#
# Add new entries only when (a) the report is not from our code, AND (b) we
# have triaged the upstream library and confirmed it is benign. Drop entries
# the moment they stop reproducing — masking real findings is worse than
# noisy CI.

# ONNX Runtime initialisation in some 1.17.x builds intentionally caches
# allocator state for the lifetime of the process and only frees it on
# atexit handlers that LSan does not see. Reported once per process and
# does not grow with G2P workload size.
#
# We match by shared-object path because the symbols inside the prebuilt
# libonnxruntime.so are stripped on the Maven Central distribution, so the
# resolved frame looks like
#     #1 0x...  (libonnxruntime.so.1.17.0+0xa4a0d9)
# and a symbol-name suppression (`leak:onnxruntime::Env::Default`) wouldn't
# match. The substring "libonnxruntime.so" pins us to ORT-originated leaks
# only, never our own code.
leak:libonnxruntime.so

# OpenJTalk's mecab dictionary loader allocates a couple of singletons that
# are released only at program exit. Same lifetime pattern as above.
leak:libopenjtalk
leak:openjtalk_create
