## 1. 目的とゴール

`test_phonemizer_registry.py` の L94-118 にある 3 つのテスト (`TestDefaultPostProcessIds.test_ja_noop`, `TestEnglishPostProcessIds.test_bos_eos`, `TestEnglishPostProcessIds.test_padding_inserted`) が、`Phonemizer` インスタンスに存在しない `post_process_ids()` メソッドを呼び出しており `AttributeError` で失敗する。

**原因**: テスト作成時の想定では `post_process_ids(ids, prosody, phoneme_id_map)` が `Phonemizer` ABC のインスタンスメソッド (デフォルト実装: no-op、英語: BOS/EOS/pad挿入) だったが、実際には BOS/EOS/padding 挿入ロジックは `PiperEncoder._post_process()` に実装されている。`Phonemizer` ABC (`base.py`) には `post_process_ids` メソッドは存在しない。

**設計思想**: `Phonemizer` ABC の責務は「テキスト -> IPA トークンリスト」変換のみ。ID 変換・BOS/EOS/padding 挿入は `PiperEncoder` (encode 層) の責務。この分離は `base.py` の docstring にも明記されている:

```python
class Phonemizer(ABC):
    """G2P abstract base class.

    phonemize() returns IPA token lists.
    BOS/EOS/padding/PUA encoding is NOT included -- that is
    the responsibility of ``piper_plus_g2p.encode.PiperEncoder``.
    """
```

**完了の定義**:
- 3 つのテストが `AttributeError` なく正常に PASS する
- BOS/EOS/padding 挿入ロジック (元テストの意図) が `PiperEncoder` 経由で正しく検証される
- 既存の他テストに影響しない

**問題点**:
1. `p.post_process_ids(...)` -- `Phonemizer` ABC にこのメソッドは存在しない
2. テストの意図は BOS/EOS/padding 挿入の検証だが、この責務は `PiperEncoder._post_process()` にある
3. 「base class は no-op」という前提が誤り -- `PiperEncoder` は言語に関係なく同一の BOS/EOS/padding ロジックを適用する

### 実際の API (`PiperEncoder`)

`PiperEncoder._post_process()` (encoder.py L166-206) が BOS/EOS/padding 挿入を行う:

```python
def _post_process(
    self,
    phoneme_ids: list[int],
    prosody_features: list[ProsodyInfo | None],
    eos_token: str,
) -> tuple[list[int], list[ProsodyInfo | None]]:
    """Insert BOS/EOS and inter-phoneme padding."""
    id_map = self._id_map
    pad_ids = id_map.get("_", [0])
    bos_ids = id_map.get("^")
    eos_ids = id_map.get(eos_token, id_map.get("$"))

    # Insert pad between every phoneme ID
    padded_ids: list[int] = []
    padded_prosody: list[ProsodyInfo | None] = []
    for phoneme_id, prosody_feature in zip(phoneme_ids, prosody_features, strict=True):
        padded_ids.append(phoneme_id)
        padded_prosody.append(prosody_feature)
        if phoneme_id not in pad_ids:
            padded_ids.extend(pad_ids)
            padded_prosody.extend([None] * len(pad_ids))

    # Wrap with BOS / EOS
    if bos_ids:
        phoneme_ids = bos_ids + [pad_ids[0]] + padded_ids
        ...
    if eos_ids:
        phoneme_ids = phoneme_ids + eos_ids
        ...

    return phoneme_ids, prosody_features
```

### 修正後のコード

`PiperEncoder` を使ってテストの意図を再現する。BOS/EOS/padding 挿入は言語非依存なので、`TestDefaultPostProcessIds` (no-op テスト) は廃止し、`PiperEncoder` の post-process テストに統一する。

```python
from piper_plus_g2p.encode import PiperEncoder


class TestPiperEncoderPostProcess:
    """PiperEncoder inserts BOS/EOS/padding around phoneme IDs."""

    def _make_encoder(
        self, phoneme_id_map: dict[str, list[int]] | None = None
    ) -> PiperEncoder:
        if phoneme_id_map is None:
            phoneme_id_map = {"_": [0], "^": [1], "$": [2]}
        return PiperEncoder(phoneme_id_map)

    def test_bos_eos(self):
        enc = self._make_encoder()
        # encode() wraps raw IDs with BOS + pad + ... + EOS
        # Using encode_with_prosody to mirror the original test's
        # (ids, prosody) return signature
        ids, prosody = enc.encode_with_prosody([], [])
        # Empty input -> BOS(1), pad(0), EOS(2)
        assert ids[0] == 1   # BOS
        assert ids[-1] == 2  # EOS

    def test_padding_inserted(self):
        """Inter-phoneme padding is inserted between every phoneme ID."""
        phoneme_id_map = {"_": [0], "^": [1], "$": [2], "a": [10], "b": [20]}
        enc = PiperEncoder(phoneme_id_map)
        ids = enc.encode(["a", "b"])
        # BOS(1), pad(0), 10, pad(0), 20, pad(0), EOS(2)
        assert ids == [1, 0, 10, 0, 20, 0, 2]

    def test_prosody_preserved(self):
        """Prosody features are aligned with padding insertion."""
        phoneme_id_map = {"_": [0], "^": [1], "$": [2], "a": [10], "b": [20]}
        enc = PiperEncoder(phoneme_id_map)
        from piper_plus_g2p import ProsodyInfo
        p1 = ProsodyInfo(a1=1, a2=2, a3=3)
        p2 = ProsodyInfo(a1=4, a2=5, a3=6)
        ids, prosody = enc.encode_with_prosody(["a", "b"], [p1, p2])
        # BOS(1), pad(0), 10, pad(0), 20, pad(0), EOS(2)
        assert ids == [1, 0, 10, 0, 20, 0, 2]
        assert len(prosody) == len(ids)
        # BOS and pad positions should be None
        assert prosody[0] is None  # BOS
        assert prosody[1] is None  # pad after BOS
        # Phoneme positions should have prosody
        assert prosody[2] == p1    # 'a'
        assert prosody[3] is None  # pad after 'a'
        assert prosody[4] == p2    # 'b'

    def test_empty_phoneme_id_map_no_bos_eos(self):
        """When BOS/EOS symbols are absent from map, they are skipped."""
        phoneme_id_map = {"_": [0], "a": [10]}
        enc = PiperEncoder(phoneme_id_map)
        ids = enc.encode(["a"])
        # No BOS/EOS, just: 10, pad(0)
        assert 10 in ids
        assert ids[0] != 1  # No BOS
```

### 旧 `TestDefaultPostProcessIds` の扱い

元テスト `test_ja_noop` は「`post_process_ids` の base 実装は no-op」を検証していたが:
- `Phonemizer` ABC に `post_process_ids` は存在しない
- `PiperEncoder._post_process()` は言語非依存で常に BOS/EOS/padding を挿入する (no-op ではない)

ただし、`PiperEncoder` に空の `phoneme_id_map` (BOS/EOS なし) を渡した場合は事実上 no-op に近い動作になるため、上記の `test_empty_phoneme_id_map_no_bos_eos` でこのエッジケースをカバーする。

## 5. 懸念事項とレビュー項目

1. **`PiperEncoder.encode()` vs `_post_process()`**: テストは公開 API (`encode()`, `encode_with_prosody()`) のみを使用する。`_post_process()` はプライベートメソッドなので直接テストしない。

2. **`PiperEncoder` のトークン変換**: `encode()` は内部で `map_token()` (PUA 変換) を経由してから `_post_process()` を呼ぶ。テストで直接 `[10, 20]` のような raw ID を渡すことはできない。代わりに `phoneme_id_map` に `"a": [10]` を定義し、トークン `["a"]` を渡すことで同等の結果を得る。

3. **元テストとの意味的差異**: 元テスト (`test_padding_inserted`) は raw ID `[10, 20]` を直接渡して `[1,0,10,0,20,0,2]` を期待していた。修正後はトークン `["a", "b"]` を `encode()` に渡し、`map_token("a")` -> `"a"` -> ID 10 の変換を経由する。PUA 変換が単一文字トークンに対しては恒等写像であることを前提としている (これは `pua.py` の仕様)。

4. **テストファイル名**: `test_phonemizer_registry.py` に `PiperEncoder` のテストが混在するのは名前として不適切。将来的には `test_encoder.py` に分離すべきだが、本チケットではファイル分割は行わない (最小限の修正に留める)。

5. **M0-1 との import 競合**: M0-1 で `from piper_plus_g2p.encode import get_phoneme_id_map` を追加し、M0-2 で `from piper_plus_g2p.encode import PiperEncoder` を追加する。両方まとめて `from piper_plus_g2p.encode import PiperEncoder, get_phoneme_id_map` とするのが望ましい。
