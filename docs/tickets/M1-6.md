## 1. 目的とゴール

`infer_onnx.py` の `main()` 内 (L397-414) に、speaker ID の解決ロジックがインラインで埋め込まれている。このロジックは「`speaker_id` が `None` かつモデルが `sid` 入力を持つ場合にデフォルト `[0]` を返す」という 3 分岐の条件判定であり、テスト可能な純粋関数として抽出すべきである。

さらに `test_multilingual_inference.py` の `TestSidDefaultForSingleSpeakerMultilingual` (L356-417) は、この本番ロジックを **テスト内にコピー&ペースト** して再実装しており、本番コードの変更に追従しない脆弱なテストになっている。

**問題点**:
1. **本番コード**: speaker ID 解決ロジックが `main()` の 400 行超のループ内に埋もれており、単体テスト不可能
2. **テストコード**: 本番ロジックのコピペ (L370-380, L393-398, L410-415) を各テストメソッドで手動再現しており、本番側が変更されてもテストは気付かずに PASS し続ける (偽陽性リスク)

**完了の定義**:
- `resolve_speaker_id()` が `infer_onnx.py` のモジュールレベル関数として抽出されている
- `main()` 内の該当ロジックが `resolve_speaker_id()` の呼び出しに置換されている
- `TestSidDefaultForSingleSpeakerMultilingual` の 3 テストが `resolve_speaker_id()` を直接呼び出す形に書き換えられている
- 既存の推論動作に影響がない (回帰なし)

## 2. 実装の詳細

### 2.1 関数抽出: `resolve_speaker_id()`

**変更ファイル**: `src/python/piper_train/infer_onnx.py`

**抽出する関数** (モジュールレベルに配置、`_DominantLanguageDetector` クラスの後あたり):

```python
def resolve_speaker_id(
    speaker_id: int | None,
    has_sid: bool,
) -> "np.ndarray | None":
    """Resolve speaker_id into an ONNX-compatible numpy array.

    Parameters
    ----------
    speaker_id : int or None
        Explicit speaker ID from the utterance / CLI argument.
    has_sid : bool
        Whether the ONNX model has a ``sid`` input tensor.

    Returns
    -------
    numpy.ndarray or None
        - ``np.array([speaker_id], dtype=np.int64)`` if *speaker_id* is not None
        - ``np.array([0], dtype=np.int64)`` if *speaker_id* is None and *has_sid* is True
          (single-speaker multilingual models require sid even though there is only 1 speaker)
        - ``None`` if *has_sid* is False and *speaker_id* is None
    """
    import numpy as np

    if speaker_id is not None:
        return np.array([speaker_id], dtype=np.int64)
    if has_sid:
        return np.array([0], dtype=np.int64)
    return None
```

> **注**: `import numpy as np` はファイル先頭に既にあるため、関数内 import は不要。上記は docstring の明示のために記載。実際のコードではファイル先頭の import を利用する。

```python
from piper_train.infer_onnx import resolve_speaker_id


@pytest.mark.unit
class TestSidDefaultForSingleSpeakerMultilingual:
    """Verify that resolve_speaker_id() defaults sid to [0] when has_sid=True
    but speaker_id is None (single-speaker multilingual model)."""

    def test_sid_default_for_single_speaker_multilingual(self):
        """has_sid=True で speaker_id=None の場合、sid がデフォルト [0] になる."""
        sid = resolve_speaker_id(speaker_id=None, has_sid=True)

        assert sid is not None, "sid must not be None when has_sid=True"
        assert sid.tolist() == [0], f"sid should default to [0], got {sid.tolist()}"
        assert sid.dtype.name == "int64"

    def test_sid_preserved_when_explicit(self):
        """has_sid=True で speaker_id=5 の場合、sid は [5] のまま."""
        sid = resolve_speaker_id(speaker_id=5, has_sid=True)

        assert sid is not None
        assert sid.tolist() == [5], f"sid should be [5], got {sid.tolist()}"

    def test_sid_none_when_no_sid_input(self):
        """has_sid=False で speaker_id=None の場合、sid は None のまま."""
        sid = resolve_speaker_id(speaker_id=None, has_sid=False)

        assert sid is None, "sid must remain None when has_sid=False"
```

## 5. 懸念事項とレビュー項目

1. **公開 API への昇格**: `resolve_speaker_id()` をモジュールレベル関数にすることで、外部コードから `from piper_train.infer_onnx import resolve_speaker_id` が可能になる。現状 `infer_onnx.py` は CLI エントリーポイントであり、ライブラリとしての公開 API は想定していない。`_resolve_speaker_id` (アンダースコアプレフィックス) にして内部関数扱いとする選択肢もあるが、テストから import する都合上プレフィックスなしが妥当。テスト対象の関数を `_` プレフィックスにするのは慣習に反する。

2. **`has_sid=False, speaker_id=3` のケース**: 現行の `main()` ロジックでは `speaker_id is not None` → `sid = np.array([3])` → `has_sid=False` だが `sid is not None` → `inputs["sid"] = sid` となり、モデルが受け付けない `sid` が渡される可能性がある。しかし `resolve_speaker_id()` は `inputs` 辞書への追加を責務としないため、このガードは呼び出し側 (`if sid is not None: inputs["sid"] = sid`) に残る。**関数の責務を「sid の値の解決」に限定し、「inputs への追加判断」は呼び出し側に委ねる** 設計が正しい。

3. **numpy import の遅延**: `resolve_speaker_id()` は `numpy` に依存する。`infer_onnx.py` はファイル先頭で `import numpy as np` しているため追加の import は不要。テスト側も `numpy` は既にテスト環境の依存に含まれている。

4. **docstring の言語**: 既存コードベースの docstring は英語で統一されている。`resolve_speaker_id()` の docstring も英語で記述する。
