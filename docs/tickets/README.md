## フェーズ横断: 一から作り直すとしたら

各チケットのセクション6に詳細な振り返りがありますが、フェーズ全体の設計判断をここにまとめます。

### M1: オンボーディング戦略
現在の設計は「README + HF + ベンチマーク」の3軸。もしゼロから設計するなら、HuggingFace Spaces のインタラクティブデモを最優先にし、README はデモへの誘導に徹する方が「30秒で音を出す」体験に直結する。ベンチマーク表はデモ内に埋め込む形が理想。

### M2: API + エコシステム
Python API (M2a) と Wyoming (M2b) の並列構造は正しい。ただし、もしやり直すなら Python API を薄いファサードではなく `openai` SDK 互換のクライアントとして設計する選択肢もある。既存の OpenAI 互換 API (`inference.py`) をバックエンドとし、ローカル/リモート両対応のクライアントにすれば、LLM エコシステムとの親和性がさらに高まる。

### M3: 音質 + エコシステム
Go/No-Go ゲートの設計は正しい。ただし、もしゼロから設計するなら VITS2 移行を M2 に前倒しし、M3 では Voice Cloning に全リソースを集中する方が効率的。VITS2 は既存アーキテクチャの改善であり研究リスクが低いため、早期実施でリスクを分散できる。
