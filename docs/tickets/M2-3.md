## 1. 目的とゴール

`test_performance.py` 内の 4 つのテストメソッドが、それぞれ**同一の `phonemize_japanese()` ローカル関数を独立に定義**している。4 箇所のコピペはすべて以下の同一パターン:

```python
from piper_plus_g2p.japanese import JapanesePhonemizer as _JaPhonemizer
from piper_plus_g2p.encode.pua import map_token as _mt

def phonemize_japanese(text):
    p = _JaPhonemizer()
    tokens = p.phonemize(text)
    return [_mt(t) for t in ["^"] + tokens + ["$"]]
```

**問題点**:
1. **DRY 違反**: 同一の 4 行関数が 4 箇所に存在し、ロジック変更時に 4 箇所の同期が必要
2. **import の冗長性**: `from piper_plus_g2p.japanese import ...` と `from piper_plus_g2p.encode.pua import ...` が各メソッドで重複
3. **`JapanesePhonemizer()` の毎回インスタンス化**: `phonemize_japanese()` が呼ばれるたびに新しい `JapanesePhonemizer` を作成しており、ベンチマークの安定性に影響 (初期化コストがベンチマーク結果に混入)

**完了の定義**:
- モジュールレベルに 1 つの `_phonemize_japanese()` ヘルパー関数が定義される
- 4 つのテストメソッドがすべてこのヘルパーを使用する
- 各テストメソッド内の `try/except ImportError` によるスキップ動作が維持される
- 既存の全テストが PASS する

### 2.2 修正後のコード

#### モジュールレベルヘルパーの追加 (ファイル先頭付近、import 直後)

```python
"""
Performance benchmarks for phonemization and synthesis
"""

import os
import statistics
import time

import pytest


try:
    import psutil
except ImportError:
    psutil = None


# ---------------------------------------------------------------------------
# Shared helper — replaces 4 identical local definitions
# ---------------------------------------------------------------------------

try:
    from piper_plus_g2p.japanese import JapanesePhonemizer as _JaPhonemizer
    from piper_plus_g2p.encode.pua import map_token as _map_token

    _HAS_G2P = True
except ImportError:
    _HAS_G2P = False


def _phonemize_japanese(text: str) -> list[str]:
    """Phonemize Japanese text with BOS/EOS markers.

    Raises ImportError if piper_plus_g2p is not installed.
    """
    if not _HAS_G2P:
        raise ImportError("piper_plus_g2p is not installed")
    p = _JaPhonemizer()
    tokens = p.phonemize(text)
    return [_map_token(t) for t in ["^"] + tokens + ["$"]]
```

### 2.3 `try/except` パターンの維持

現在の各テストメソッドは `try: ... except ImportError: pytest.skip(...)` でガードされている。モジュールレベルでの import 失敗時に `_HAS_G2P = False` を設定し、ヘルパー関数で `ImportError` を raise することで、既存の skip 動作を維持する。

代替案として `pytest.importorskip` を使う方法もある:

```python
# ファイル先頭
_g2p_available = pytest.importorskip is not None  # 常に True だが意図を明示

# 各テストにデコレータ
@pytest.mark.skipif(not _HAS_G2P, reason="piper_plus_g2p not installed")
def test_phonemization_single_conversion(self):
    ...
```

ただし既存コードが `try/except` パターンを使っているため、最小差分を優先して同パターンを維持する。

## 5. 懸念事項とレビュー項目

1. **モジュールレベル import の副作用 (重要度: 中)**: `from piper_plus_g2p.japanese import JapanesePhonemizer` をモジュールレベルで実行すると、`test_performance.py` のモジュール収集時に `pyopenjtalk` 等の依存が読み込まれる。現在は `try/except ImportError` でガードしているため、依存がなくてもモジュール自体の import は失敗しない。ただし `piper_plus_g2p` がインストール済みで `pyopenjtalk` が未インストールの場合、モジュールレベルで `ImportError` が発生するか確認すること。

2. **`JapanesePhonemizer()` の毎回インスタンス化**: 現在のコードは `phonemize_japanese()` 呼び出しごとに `JapanesePhonemizer()` を新規作成している。これはベンチマークとしては初期化コストも含めた計測になっており、意図的な可能性がある。ヘルパー関数でもこの挙動を維持する (インスタンスをキャッシュしない)。

3. **テストの独立性**: 4 つのテストメソッドはそれぞれ `try/except` で独立にスキップ可能である。モジュールレベルヘルパーにしても、各メソッドが個別にスキップできる構造を維持すること。

## 6. 一から作り直すとしたら

1. **`JapanesePhonemizer` のインスタンスをフィクスチャ化**: `@pytest.fixture(scope="class")` で `JapanesePhonemizer` インスタンスを 1 回だけ作成し、ベンチマークでは初期化コストを除外する。初期化コスト自体のベンチマークは別テストで行う。

2. **`pytest-benchmark` の導入**: 手動の `time.perf_counter()` 計測を `pytest-benchmark` に置き換えれば、統計処理 (mean, stddev, outlier detection) が自動化される。

3. **`conftest.py` への移動**: `_phonemize_japanese` を `conftest.py` に配置すれば、他のテストファイルからも再利用可能。ただし `test_performance.py` 以外に利用箇所がなければ、ファイル内ヘルパーで十分。

4. **`@pytest.mark.skipif` デコレータによる統一**: `try/except` をやめて `@pytest.mark.skipif(not _HAS_G2P, reason="...")` で統一すれば、テストメソッド冒頭のボイラープレートがさらに削減される。
