## 1. 目的とゴール

`test_integration.py` L29-85 に定義された 2 つのテストケースが、本番コードを一切呼び出さず、Python 標準ライブラリの機能のみをテストしている。テストスイートの信頼性と意味を損なうため、本番コード呼び出しに書き換えるか削除する。

**問題のテスト 1** — `test_model_config_validation` (L29-54):

```python
@pytest.mark.integration
def test_model_config_validation(self):
    """Test that model configs are valid"""
    test_configs = [
        {
            "audio": {"sample_rate": 22050},
            "num_symbols": 100,
            "phoneme_id_map": {"_": 0},
        },
        {
            "audio": {"sample_rate": 22050},
            "phoneme_type": "openjtalk",
            "language": {"code": "ja"},
            "num_symbols": 150,
            "phoneme_id_map": {"_": 0, "\ue00e": 30},
        },
    ]

    for config in test_configs:
        # Required fields
        assert "audio" in config
        assert "sample_rate" in config["audio"]
        assert "phoneme_id_map" in config

        # Japanese specific
        if config.get("language", {}).get("code") == "ja":
            assert config.get("phoneme_type") == "openjtalk"
```

**問題点**: テスト内でハードコードした辞書リテラルに対して `"audio" in config` 等を assert している。これはテストデータ自身の構造を検証しているだけで、本番コードの config バリデーションロジックは呼び出されていない。

**問題のテスト 2** — `test_wav_file_generation` (L57-85):

```python
@pytest.mark.integration
def test_wav_file_generation(self):
    """Test that we can generate valid WAV files"""
    import wave

    sample_rate = 22050
    duration = 0.5
    samples = int(sample_rate * duration)
    audio_data = np.zeros(samples, dtype=np.int16)

    with tempfile.NamedTemporaryFile(suffix=".wav", delete=False) as tmp:
        tmp_path = Path(tmp.name)

    try:
        with wave.open(str(tmp_path), "wb") as wav_file:
            wav_file.setnchannels(1)
            wav_file.setsampwidth(2)
            wav_file.setframerate(sample_rate)
            wav_file.writeframes(audio_data.tobytes())

        with wave.open(str(tmp_path), "rb") as wav_file:
            assert wav_file.getnchannels() == 1
            assert wav_file.getsampwidth() == 2
            assert wav_file.getframerate() == sample_rate
            assert wav_file.getnframes() == samples
    finally:
        tmp_path.unlink()
```

**問題点**: Python 標準 `wave` モジュールの read/write が正常動作するかをテストしているだけ。piper-plus の本番コードは一切呼び出されていない。

**完了の定義** (以下のいずれか):
- **Option A (本番コード呼び出し)**: 本番の config パース / WAV 生成ロジックを呼び出すテストに書き換え
- **Option B (削除)**: 無意味なテストを削除

## 2. 実装の詳細

### 2.1 推奨: Option B (削除)

**理由**:
- `test_model_config_validation`: 本番の config バリデーションは `infer_onnx.py` や C#/Rust ランタイムの config 読み込みで行われており、Python 側に独立した config バリデーション関数が存在しない。テストを書き換える対象の本番コードがない。
- `test_wav_file_generation`: piper-plus の WAV 出力は `infer_onnx.py` 内で `wave` モジュールを直接使用しており、独立した WAV ユーティリティ関数は存在しない。標準ライブラリの動作検証はプロジェクトのテストスイートに含めるべきではない。

### 2.4 代替案: Option A の場合

もし削除ではなく書き換えを選ぶ場合:

**test_model_config_validation の書き換え案**: `infer_onnx.py` の config 読み込みパスを呼び出す。ただし現状 `infer_onnx.py` は config を `json.load()` で直接読むだけで独立したバリデーション関数がないため、まず本番コードに `validate_config()` を追加する必要がある (M2 の範囲を超える)。

**test_wav_file_generation の書き換え案**: `infer_onnx.py` の WAV 出力パスを呼び出す。ただし ONNX モデルが必要なため `@pytest.mark.requires_model` が必要で、CI では実行できない。

いずれも本番コードの変更が前提となるため、M2 (テストのみの修正) では Option B (削除) を推奨する。
