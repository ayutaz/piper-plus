## 1. 目的とゴール

piper-plus の推論性能を定量的に測定し、競合 TTS システムと比較可能なベンチマークスクリプトを作成する。README やブログで使える客観的な数値を提供し、piper-plus の速度・効率性を訴求する。

完了時の状態:
- `scripts/benchmark.py` が作成されている
- RTF, モデルサイズ, ピークメモリ, 初回レイテンシ, パラメータ数を自動計測できる
- JSON/Markdown 形式で結果を出力できる
- 再現可能な計測条件が文書化されている

### 計測指標

| 指標 | 単位 | 説明 |
|------|------|------|
| RTF (Real-Time Factor) | ratio | 音声長 / 推論時間。< 1.0 ならリアルタイム以上 |
| モデルサイズ | MB | ONNX ファイルサイズ |
| ピークメモリ | MB | 推論中の最大 RSS (Resident Set Size) |
| 初回レイテンシ | ms | コールドスタート (warmup なし) での初回推論時間 |
| Warmed レイテンシ | ms | warmup 2回後の推論時間 (P50/P95) |
| 対応言語数 | int | モデルが対応する言語数 |
| パラメータ数 | M | ONNX モデルのパラメータ数 |

def measure_latency(session, input_dict, n_warmup=2, n_runs=10):
    """warmup 後に n_runs 回推論し、レイテンシ統計を返す"""
    # warmup
    for _ in range(n_warmup):
        session.run(None, input_dict)

    latencies = []
    for _ in range(n_runs):
        start = time.perf_counter()
        session.run(None, input_dict)
        elapsed = time.perf_counter() - start
        latencies.append(elapsed * 1000)  # ms

    return {
        "mean_ms": np.mean(latencies),
        "p50_ms": np.percentile(latencies, 50),
        "p95_ms": np.percentile(latencies, 95),
        "min_ms": np.min(latencies),
        "max_ms": np.max(latencies),
    }


def measure_cold_start(model_path: str, input_dict: dict) -> float:
    """セッション作成 + 初回推論のレイテンシ (ms)"""
    start = time.perf_counter()

    session.run(None, input_dict)
    return (time.perf_counter() - start) * 1000

## 5. 懸念事項とレビュー項目

### 懸念事項
- ピークメモリ計測は OS 依存 (`resource` モジュールは Unix のみ)。macOS と Linux で `ru_maxrss` の単位が異なる (macOS: bytes, Linux: KB)。**Windows では `resource` が利用不可のため、`psutil` をフォールバックとして使用する**
- ORT セッションのスレッド数がベンチマーク結果に大きく影響する。`ort-session-contract.toml` の設定を使うか、固定値 (1スレッド) で統一するか決める必要がある
- 競合モデルのベンチマークは各モデルのセットアップが必要で、スクリプトのスコープを超える可能性がある
- `time.perf_counter()` の精度は通常十分だが、非常に短い推論 (<1ms) では誤差が大きい

### レビューチェックリスト
- [ ] RTF 計算が正しい (audio_duration / inference_time ではなく inference_time / audio_duration)
- [ ] warmup が確実に実行されている
- [ ] メモリ計測が OS 差を適切に処理している
- [ ] 出力フォーマットが README に埋め込み可能
- [ ] 既存の `scripts/benchmark_streaming_comparison.py` との機能重複がないこと
- [ ] 依存パッケージが `pyproject.toml` に含まれている

## 6. 一から作り直すとしたら

- Python スクリプトではなく、pytest-benchmark や ASV (Airspeed Velocity) のような専用フレームワークを使うほうが、結果の追跡・比較・可視化が容易。ただしセットアップコストが高い
- 各プラットフォーム (Python/Rust/C#/Go) のベンチマークを統一フォーマットで出力する仕組みが理想。現状は Python のみ
- 競合モデルのベンチマークは `scripts/benchmark_competitors/` として別ディレクトリに分離し、各競合の Dockerfile を用意するほうがクリーン
- CI で定期的にベンチマークを実行し、パフォーマンスリグレッションを検出する仕組み (GitHub Actions + benchmark 結果の artifact 保存) まで見据えるべきだった
