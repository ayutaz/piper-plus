## 6. 一から作り直すとしたら

- shields.io の静的バッジではなく、カスタム SVG バッジ (`etc/` にアセットとして格納) のほうがデザインの自由度が高い。ただし外部サービス依存を減らせる反面、メンテコストが増える
- 「Try in Browser」よりも具体的な「Listen to Sample」のほうがクリック率が高いかもしれない。A/B テストはできないが、他の OSS TTS プロジェクトのバッジ文言を参考にすべき
- そもそもバッジよりも README 冒頭にインライン audio タグでサンプル音声を埋め込むほうがインパクトが大きい (GitHub Markdown は audio タグ非対応だが、HF Spaces の iframe embed は可能)
- GitHub Social Preview 画像を最適化して、リンク共有時のビジュアルインパクトを高めるべきだった。1280x640px の OGP 画像に多言語対応とサンプル波形を含めることで、SNS でのクリック率向上が期待できる
- Pinned Repository を活用し、piper-plus を GitHub プロフィールの目立つ位置に固定表示することで、オーガニック流入を増やせる
