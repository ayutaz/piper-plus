# M1-5 tsukuyomi-chan にサンプル音声 wav 追加 (6言語)

**マイルストーン**: M1 (v1.12.0)
**カテゴリ**: モデル配布
**工数**: 0.5d
**依存**: なし
**ステータス**: 未着手

---

## 1. 目的とゴール

HF モデルリポジトリ `ayousanz/piper-plus-tsukuyomi-chan` に6言語のサンプル音声ファイルを追加する。モデルの品質を事前に確認できるようにし、ダウンロード前の判断材料を提供する。README やモデルカードからの参照用音声としても機能する。

完了時の状態:
- `ayousanz/piper-plus-tsukuyomi-chan` の `samples/` ディレクトリに6言語の wav ファイルが格納されている
- 各 wav ファイルは高品質 (22050Hz, 16bit, mono) で、適切な長さ (2-5秒) のサンプル文を収録
- モデルカードから各サンプル音声を再生/ダウンロードできる

## 2. 実装内容の詳細

### サンプル音声仕様

| ファイル名 | 言語 | テキスト | 想定長 |
|-----------|------|---------|--------|
| `samples/ja.wav` | 日本語 | こんにちは、つくよみちゃんです。 | ~3s |
| `samples/en.wav` | 英語 | Hello, how are you today? | ~2.5s |
| `samples/zh.wav` | 中国語 | 你好，今天天气很好。 | ~1.2s |
| `samples/es.wav` | スペイン語 | ¿Hola, cómo estás hoy? | ~2.9s |
| `samples/fr.wav` | フランス語 | Bonjour, comment allez-vous? | ~2.1s |
| `samples/pt.wav` | ポルトガル語 | Olá, como você está hoje? | ~2.2s |

### 音声生成コマンド

既存のつくよみちゃん 6lang モデルを使用:

```bash
# 各言語のサンプル生成
CUDA_VISIBLE_DEVICES="" uv run python -m piper_train.infer_onnx \
  --model /data/piper/output-tsukuyomi-finetune-6lang-v2/tsukuyomi-6lang-v2-fixed.onnx \
  --config /data/piper/dataset-tsukuyomi-finetune-6lang/config.json \
  --output-dir ./samples \
  --text "こんにちは、つくよみちゃんです。" \
  --language ja-en-zh-es-fr-pt --speaker-id 0
```

各言語について個別に実行し、出力ファイルを `ja.wav`, `en.wav` 等にリネームする。

## 5. 懸念事項とレビュー項目

### 懸念事項
- ZH の duration が短い (1.21s) ことが既知の問題。凍結された DP の ZH パラメータ特性による。サンプルとしては許容するが、注記を付けるか検討
- HF の音声プレビュー機能は wav フォーマットに対応しているが、ファイルサイズが大きいとプレビューが遅い。必要なら mp3 変換も検討
- 推論時の `noise_scale` パラメータによって音質が変わる。複数パターン生成して最良のものを選ぶ

### レビューチェックリスト
- [ ] 6言語すべてのサンプルが生成されている
- [ ] 音声品質が許容レベルである (明瞭な発話、適切な速度)
- [ ] wav フォーマットが統一されている (22050Hz, 16bit, mono)
- [ ] ファイルサイズが適切 (各 100KB-500KB 程度)
- [ ] モデルカードの音声リンクが正しく機能する

## 6. 一から作り直すとしたら

- wav だけでなく mp3 も並行して提供すべき。HF のプレビューは wav 対応だが、ダウンロードサイズの観点では mp3 が有利
- 1文だけでなく、3-5文のバリエーション (ニュース文、会話文、感情を含む文) を用意するほうがモデルの能力を正確に伝えられる
- 音声生成をスクリプト化 (`scripts/generate_samples.sh`) してリポジトリに含めれば、モデル更新時にサンプルの再生成が容易になる
- `noise_scale` のバリエーション (0.3, 0.5, 0.667) で生成してベストを選ぶプロセスを定型化すべき
