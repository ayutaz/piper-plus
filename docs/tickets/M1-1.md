## 1. 目的とゴール

`export_onnx.py` の `main()` 内に定義されたクロージャ `infer_forward()` (L278-331) は、ONNX エクスポート用の推論フォワード関数である。この同一ロジックが `conftest.py` 内で **3 箇所** (L117-166, L217-258, L365-405) にコピペ再実装されている。各コピーは微妙な差異 (stochastic/deterministic、single-speaker/multi-speaker) があるものの、コアロジック (Duration Predictor → Attention → Flow → Decoder) は同一である。

**問題点**:
1. **DRY 違反**: 同一ロジックが 4 箇所に存在し、DP や Flow の変更時に全箇所の同期が必要
2. **テストの脆弱性**: conftest.py のコピーが本番コードと乖離するリスクが常に存在
3. **バグの温床**: conftest.py L164 の `audio = o.unsqueeze(1)` は本番コード (L329 `return o`) と形状が異なる。テスト側が独自に `unsqueeze` を追加しており、ONNX グラフの出力形状と不整合を起こす可能性がある

**完了の定義**:
- `export_onnx.py` に `build_infer_forward(model, stochastic=True)` 関数が公開 API として存在する
- `conftest.py` の 3 つの `infer_forward` 定義がすべて `build_infer_forward()` の呼び出しに置換されている
- 既存の全テスト (`test_export_onnx.py`, `test_onnx_parity.py` 等) が変更なく PASS する
- `build_infer_forward()` 自体のユニットテストが追加されている

### 2.2 現在のコード: export_onnx.py (L276-333)

`main()` のローカルスコープで `model_g` と `stochastic` をキャプチャするクロージャ:

```python
stochastic = args.stochastic

def infer_forward(
    text, text_lengths, scales, sid=None, lid=None, prosody_features=None
):
    length_scale = scales[1]
    noise_scale_w = scales[2]

    g = model_g._get_global_conditioning(sid, lid)
    x, m_p, logs_p, x_mask = model_g.enc_p(text, text_lengths, g=g)

    x_dp = model_g._prepare_prosody_input(x, x_mask, prosody_features, lid=lid)
    if model_g.use_sdp:
        logw = model_g.dp(x_dp, x_mask, g=g, reverse=True, noise_scale=noise_scale_w)
    else:
        logw = model_g.dp(x_dp, x_mask, g=g)

    w = torch.exp(logw) * x_mask * length_scale
    durations = w.squeeze(1)

    w_ceil = torch.ceil(w)
    y_lengths = torch.clamp_min(torch.sum(w_ceil, [1, 2]), 1).long()
    y_mask = torch.unsqueeze(
        commons.sequence_mask(y_lengths, y_lengths.max()), 1
    ).type_as(x_mask)
    attn_mask = torch.unsqueeze(x_mask, 2) * torch.unsqueeze(y_mask, -1)
    attn = commons.generate_path(w_ceil, attn_mask)

    m_p = torch.matmul(attn.squeeze(1), m_p.transpose(1, 2)).transpose(1, 2)
    logs_p = torch.matmul(attn.squeeze(1), logs_p.transpose(1, 2)).transpose(1, 2)

    if stochastic:
        noise_scale = scales[0]
        z_p = m_p + torch.randn_like(m_p) * torch.exp(logs_p) * noise_scale
    else:
        z_p = m_p

    z = model_g.flow(z_p, y_mask, g=g, reverse=True)
    o = model_g.dec((z * y_mask), g=g)

    return o, durations

model_g.forward = infer_forward
```

### 2.4 修正後のコード

#### export_onnx.py に追加する `build_infer_forward()` 関数

`main()` の外側、モジュールレベルに配置する。`should_unify_emb_lang()` 等と同じ位置 (L28 付近)。

```python
def build_infer_forward(
    model: "SynthesizerTrn",
    *,
    stochastic: bool = True,
) -> callable:
    """Build an inference-only forward function for ONNX export.

    The returned callable replaces ``model.forward`` before
    ``torch.onnx.export()`` so that the exported graph contains only
    the inference path (no discriminator, no training loss).

    Parameters
    ----------
    model : SynthesizerTrn
        The generator model (``model_g``). Must already have
        ``remove_weight_norm()`` applied on ``model.dec``.
    stochastic : bool
        If True, sample z_p with noise (production default).
        If False, use the mean (deterministic / debug).

    Returns
    -------
    callable
        A forward function with signature::

            infer_forward(text, text_lengths, scales,
                          sid=None, lid=None, prosody_features=None)
            -> (audio: Tensor, durations: Tensor)
    """

    def infer_forward(
        text, text_lengths, scales, sid=None, lid=None, prosody_features=None
    ):
        length_scale = scales[1]
        noise_scale_w = scales[2]

        g = model._get_global_conditioning(sid, lid)
        x, m_p, logs_p, x_mask = model.enc_p(text, text_lengths, g=g)

        x_dp = model._prepare_prosody_input(x, x_mask, prosody_features, lid=lid)
        if model.use_sdp:
            logw = model.dp(
                x_dp, x_mask, g=g, reverse=True, noise_scale=noise_scale_w
            )
        else:
            logw = model.dp(x_dp, x_mask, g=g)

        w = torch.exp(logw) * x_mask * length_scale
        durations = w.squeeze(1)

        w_ceil = torch.ceil(w)
        y_lengths = torch.clamp_min(torch.sum(w_ceil, [1, 2]), 1).long()
        y_mask = torch.unsqueeze(
            commons.sequence_mask(y_lengths, y_lengths.max()), 1
        ).type_as(x_mask)
        attn_mask = torch.unsqueeze(x_mask, 2) * torch.unsqueeze(y_mask, -1)
        attn = commons.generate_path(w_ceil, attn_mask)

        m_p = torch.matmul(attn.squeeze(1), m_p.transpose(1, 2)).transpose(1, 2)
        logs_p = torch.matmul(attn.squeeze(1), logs_p.transpose(1, 2)).transpose(
            1, 2
        )

        if stochastic:
            noise_scale = scales[0]
            z_p = m_p + torch.randn_like(m_p) * torch.exp(logs_p) * noise_scale
        else:
            z_p = m_p

        z = model.flow(z_p, y_mask, g=g, reverse=True)
        o = model.dec((z * y_mask), g=g)

        return o, durations

    return infer_forward
```

### 2.5 `_get_global_conditioning` による single-speaker/multi-speaker 統一

`build_infer_forward()` は常に `model._get_global_conditioning(sid, lid)` を呼ぶ。single-speaker モデル (`n_speakers=1`, `n_languages=1`) で `sid=None`, `lid=None` を渡した場合、`_get_global_conditioning` は `g=None` を返すため、conftest.py (A)(B) の `g = None` ハードコードと等価になる。

ただし `torch.onnx.export` に渡す `dummy_input` のタプル構造が変わる。現在のフィクスチャ (A)(B) では `(sequences, sequence_lengths, scales, prosody_features)` の 4 要素で `sid`, `lid` を含まない。`build_infer_forward()` の戻り関数は `sid=None`, `lid=None` をデフォルトで受け取るため、4 要素タプルでも正常にエクスポートできるが、ONNX の `input_names` と位置引数の対応が正しいことを確認する必要がある。

**推奨**: フィクスチャ側では明示的に `sid=None`, `lid=None` を渡すか、single-speaker 用のラッパーを用意する。最もシンプルなのは `functools.partial` ではなく、フィクスチャ内で thin wrapper を定義すること:

```python
_infer = build_infer_forward(mock_vits_model, stochastic=False)

def infer_forward_single(input_tensor, input_lengths, scales_tensor, prosody_features_tensor):
    return _infer(input_tensor, input_lengths, scales_tensor, prosody_features=prosody_features_tensor)

mock_vits_model.forward = infer_forward_single
```

この thin wrapper は 4 行のみであり、50 行のロジック重複と比較して許容範囲内。マルチリンガルフィクスチャ (C) では thin wrapper 不要 (引数順序が一致)。

### 2.6 `unsqueeze` の差異の解決

conftest.py (A)(B) には `audio = o.unsqueeze(1)` がある。これは本番 `export_onnx.py` には存在しない。

調査ポイント:
1. `test_onnx_parity.py` 等で `output` の ndim を検証しているか確認
2. もし ndim=3 を期待するテストがあれば、`build_infer_forward()` の戻り値に合わせてテスト側の assertion を修正
3. ONNX グラフの `output` ノードの shape が `[batch, time]` (本番) vs `[batch, 1, time]` (テスト) で異なるかチェック

**方針**: 本番コードが正しい。テスト側を本番に合わせる。

## 5. 懸念事項とレビュー項目

1. **`unsqueeze` の不整合 (重要度: 高)**: conftest.py (A)(B) の `o.unsqueeze(1)` が本番コードに存在しない。`build_infer_forward()` 統一後にテストが失敗する可能性がある。事前に `test_onnx_parity.py` の出力形状 assertion を全数確認すること。ONNX グラフの output shape が `[batch, channels, time]` (3D) であることを確認し、`unsqueeze` が不要であれば除去、必要であれば `build_infer_forward()` に含める。

2. **`torch.onnx.export` の位置引数マッチング (重要度: 高)**: `build_infer_forward()` の戻り関数は `(text, text_lengths, scales, sid=None, lid=None, prosody_features=None)` のシグネチャ。single-speaker エクスポートで `(sequences, sequence_lengths, scales, prosody_features)` の 4 要素タプルを渡すと、`prosody_features` が `sid` に位置対応してしまう。thin wrapper で引数を正しくマッピングする必要がある。

3. **`commons` の import**: `build_infer_forward()` が `commons.sequence_mask` と `commons.generate_path` を使用するため、`export_onnx.py` のモジュールスコープで `from .vits import commons` が必要。現在 L11 で既に import 済みであることを確認済み。

4. **`model` の型ヒント**: `SynthesizerTrn` は `export_onnx.py` で直接 import されていない (`VitsModel.load_from_checkpoint` 経由で取得)。型ヒントは文字列リテラル `"SynthesizerTrn"` またはプロトコルで記述し、循環 import を回避する。

5. **テストフィクスチャのスコープ**: `temp_onnx_model` 等は `scope="module"` で定義されている。`build_infer_forward` の import がモジュールスコープで解決される必要がある (テスト収集時のエラー回避)。

## 6. 一から作り直すとしたら

1. **`SynthesizerTrn.infer()` メソッドの活用**: 本来 `SynthesizerTrn` に `infer()` メソッドが存在すれば、`build_infer_forward()` は不要。`model_g.infer(text, text_lengths, scales, sid, lid, prosody_features)` のような推論専用メソッドを `models.py` に追加し、`export_onnx.py` では `model_g.forward = model_g.infer` とするだけで済む。ただしこの変更は `models.py` への侵入が大きいため、M1 の範囲外とし M2 以降で検討する。

2. **ONNX エクスポートヘルパークラス**: `build_infer_forward()`, ダミー入力生成, `input_names`/`dynamic_axes` 構築をまとめた `OnnxExportHelper` クラスを設ける。conftest.py では `helper = OnnxExportHelper(model, stochastic=False); helper.export(path)` と 1 行で済む。

3. **テストフィクスチャの簡素化**: conftest.py の 3 つのフィクスチャは「モデル作成 → forward 差し替え → ONNX エクスポート → forward 復元」の定型パターン。`@pytest.fixture` を生成する `make_onnx_fixture(model_factory, stochastic, multilingual)` メタフィクスチャを導入すれば、各フィクスチャは 3-5 行で定義可能。

4. **`unsqueeze` の根本解決**: conftest.py (A)(B) の `unsqueeze` は本番コードとの乖離であり、そもそもこの差分が生まれた経緯を調査すべき。テスト作成者が `o` の shape を `[batch, 1, time]` と誤認した可能性がある。根本原因を特定し、テスト全体で統一的な出力形状期待値を設定する。

## 7. 後続タスクへの連絡事項

- **API 公開**: `build_infer_forward()` は `export_onnx.py` のモジュールレベル関数として公開される。`__init__.py` への追加は不要 (テストからは `from piper_train.export_onnx import build_infer_forward` で直接 import)。
- **出力形状の統一**: `unsqueeze` の差異を解決した場合、後続の M2 タスクで ONNX ランタイム推論側 (`infer_onnx.py`, `ort_utils.py`) の出力形状処理も影響を受ける可能性がある。修正内容を M2 チケットに明記すること。
- **thin wrapper パターン**: conftest.py に残る thin wrapper (4 行) は「ONNX export の位置引数と `build_infer_forward` のキーワード引数のブリッジ」であり、本番コードのロジック重複ではない。M2 で `OnnxExportHelper` を導入する際にこの wrapper も統合可能。
- **`models.py` への `infer()` メソッド追加**: 「一から作り直すとしたら」で提案した `SynthesizerTrn.infer()` の追加は、Duration Predictor 周りの変更 (M2 予定) と合わせて検討すると効率的。
