## 1. 目的とゴール

`lightning.py` の `on_validation_epoch_end()` (L706-730) に、`speaker_id` / `language_id` を正規化するインラインロジックが存在する。このロジックは `int` / 0-D Tensor / 1-D Tensor / `None` の 4 パターンを処理して `shape [1]` の `LongTensor` (または `None`) に統一する。同一のロジックが `speaker_id` と `language_id` で完全に重複しており (L706-717 と L719-730)、さらに `test_freeze_dp.py` の `test_speaker_id_tensor_handling` (L108-167) がこのロジックをコピー&ペーストで再実装してテストしている。

**問題点**:
1. **DRY 違反**: `lightning.py` 内で `speaker_id` と `language_id` に対して全く同じ 11 行のコードが 2 回存在する
2. **テストがコピペ**: `test_freeze_dp.py` が本番コードの関数ではなく、コピペしたロジックを直接テストしている。本番コードを変更してもテストは壊れない (テストの意味がない)
3. **可読性**: `on_validation_epoch_end()` が長大 (100行超) で、ID 正規化のような汎用処理がインラインにあると本質的な音声生成ロジックが読みにくい

**完了の定義**:
- `_normalize_speaker_id()` をモジュールレベル関数 (またはスタティックメソッド) として抽出し、`lightning.py` の `on_validation_epoch_end()` から呼び出す
- `test_freeze_dp.py` の `test_speaker_id_tensor_handling` がコピペではなく抽出した関数を直接呼び出すように書き換える
- 既存の全テストが PASS する
- 抽出前後で `on_validation_epoch_end()` の振る舞いが変わらない

### 抽出する関数

`lightning.py` にモジュールレベル関数として追加する (`VitsModel` クラスの外、ファイル先頭のユーティリティ領域):

```python
def _normalize_id(
    raw_value: int | torch.Tensor | None,
    device: torch.device | None = None,
) -> torch.Tensor | None:
    """Normalize a speaker_id or language_id to a 1-D LongTensor of shape [1].

    Handles four input patterns produced by the dataset layer:
    - ``int``         -> ``torch.LongTensor([value])``
    - 0-D ``Tensor``  -> ``value.unsqueeze(0)``  (scalar from ``random_split`` Subset)
    - 1-D ``Tensor``  -> pass-through            (already shape [1])
    - ``None``        -> ``None``

    Optionally moves the result to *device*.
    """
    if raw_value is None:
        return None
    if isinstance(raw_value, torch.Tensor):
        t = raw_value.unsqueeze(0) if raw_value.dim() == 0 else raw_value
    else:
        t = torch.LongTensor([raw_value])
    if device is not None:
        t = t.to(device)
    return t
```

```python
from piper_train.vits.lightning import _normalize_id


@pytest.mark.unit
def test_speaker_id_tensor_handling():
    """speaker_id が int でも Tensor でも正しく shape [1] の LongTensor になる.

    Regression test: on_validation_epoch_end の audio logging で
    speaker_id が int (test_utterances.jsonl 経由) の場合と

    assert sid.dtype == torch.long
    assert sid.item() == 3

    assert sid.item() == 5

    assert sid.dtype == torch.long
    assert sid.item() == 7

    # Case 4: None -> stays None
    sid = _normalize_id(None)
    assert sid is None
```

## 5. 懸念事項とレビュー項目

1. **関数名 `_normalize_id` vs `_normalize_speaker_id`**: `speaker_id` と `language_id` の両方に使うため、汎用名 `_normalize_id` を採用する。ユーザーの指示では `_normalize_speaker_id()` だが、`language_id` にも同じ関数を使うのであれば `_normalize_id` が適切。チケットタイトルでは `_normalize_speaker_id()` を維持するが、実装は `_normalize_id` とする。もし `_normalize_speaker_id` を厳密に使いたい場合は `language_id` 用に `_normalize_language_id` を別途作るか、エイリアスを定義する必要があるが、ロジックが完全に同一なので冗長。

2. **関数の配置場所**: モジュールレベル関数 (`lightning.py` ファイルのトップレベル) として定義する。`VitsModel` のスタティックメソッドにする案もあるが、テスト側で `_normalize_id` を import する際にモジュールレベル関数の方が簡潔 (`from piper_train.vits.lightning import _normalize_id`)。スタティックメソッドの場合は `VitsModel._normalize_id` を呼ぶ必要があり、`VitsModel` の import で PyTorch Lightning 等の依存が芋蔓式に引かれる可能性がある (ただし `test_freeze_dp.py` は既に `VitsModel` を import しているためこの懸念は限定的)。

3. **`.to(device)` の統合**: 現在のインラインコードでは `.to(self.device)` が `Tensor` パスと `LongTensor` パスの両方に存在する。抽出後は `_normalize_id(raw_value, device)` で統一。`device=None` の場合は `.to()` を呼ばない (テスト時に device を指定する必要がないため)。

4. **型ヒント**: `raw_value: int | torch.Tensor | None` は Python 3.10+ の union syntax。プロジェクトが Python 3.10+ を前提としているか確認 (`pyproject.toml` の `python_requires`)。3.9 以下の場合は `Union[int, torch.Tensor, None]` を使う。

5. **アンダースコアプレフィックス**: `_normalize_id` は先頭にアンダースコアを付ける (モジュール内部の非公開ユーティリティ)。テストからは直接 import するが、これは Python の慣習として許容される (テストは内部 API にアクセスしてよい)。

6. **テストから `VitsModel` の import が不要になる**: 修正後の `test_speaker_id_tensor_handling` は `_normalize_id` のみを使用し、`VitsModel` を作成しない。そのため PyTorch Lightning や VITS の重い依存なしでテストが実行できる。ただし、同ファイルの他テスト (`test_freeze_dp_params_frozen` 等) は引き続き `VitsModel` を使うため、ファイルレベルの依存は変わらない。

2. **データセット側での正規化**: `Utterance` NamedTuple / dataclass の `speaker_id` / `language_id` を常に `Optional[torch.LongTensor]` (shape [1]) で返すように `PiperDataset.__getitem__()` を修正する。そうすれば `on_validation_epoch_end()` で正規化する必要がなくなる。ただし `random_split` の Subset ラッパーが 0-D Tensor を生成する PyTorch の挙動に依存するため、完全には防げない。

```python

    "raw_value, expected_item",
    [
        (3, 3),
        (torch.LongTensor([5]), 5),
        (torch.tensor(7, dtype=torch.long), 7),
```

5. **device テストの追加**: `device` 引数のテストを明示的に追加する。CPU 環境では `torch.device("cpu")` のみだが、CI に GPU がある場合は `@pytest.mark.skipUnless` で CUDA テストも追加できる。
