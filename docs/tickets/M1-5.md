## 1. 目的とゴール

### ステップ 1: `infer_onnx.py` に `resolve_config_path()` を抽出

#### 抽出する関数 (新規追加)

`infer_onnx.py` のモジュールレベル (クラス外、`main()` の前) に以下を追加する:

```python
def resolve_config_path(model: str, config: str | None) -> Path:
    """Resolve the config.json path for a given model.

    Fallback order:
      1. If *config* is explicitly given, use that path directly.
      2. Try ``{model}.json`` (e.g. ``model.onnx.json`` — C++ CLI convention).
      3. Fall back to ``{model_dir}/config.json``.

    Returns the resolved :class:`Path`.  The caller is responsible for
    checking whether the returned path actually exists.
    """
    if config:
        return Path(config)

    model_path = Path(model)
    onnx_json = model_path.with_suffix(model_path.suffix + ".json")
    if onnx_json.exists():
        return onnx_json
    return model_path.parent / "config.json"
```

#### `main()` 内の書き換え (L309-320)

**Before** (現在の `main()` 内インライン実装):

### Rust 実装との対応

Python 版と Rust 版 (`src/rust/piper-core/src/config.rs` L114-144 `VoiceConfig::resolve_config_path()`) のフォールバック順序が同一であることを確認する:

| 優先度 | Python (`infer_onnx.py`) | Rust (`config.rs`) |
|--------|------------------------|--------------------|
| 1 | `config` 引数あり -> `Path(config)` | `explicit_config` あり -> `Ok(p.to_path_buf())` |
| 2 | `{model}.onnx.json` 存在チェック | `model_path.with_extension("onnx.json")` 存在チェック |
| 3 | `{model_dir}/config.json` | `dir.join("config.json")` |

**差異**: Rust 版はファイル不在時に `Err(PiperError::ConfigNotFound)` を返すが、Python 版は存在しないパスをそのまま返し、呼び出し側 (`main()` L322-327) で `sys.exit(1)` する。この責務分離は維持する (Python 版で例外を投げる変更は行わない)。

## 5. 懸念事項とレビュー項目

1. **循環 import リスク**: `infer_onnx.py` は `ort_utils`, `vits.utils`, `vits.wavfile` を import している。`resolve_config_path()` 自体は `pathlib.Path` のみに依存するため、テストファイルから `from piper_train.infer_onnx import resolve_config_path` しても重い依存 (`onnxruntime` 等) がモジュールレベルで import されるリスクがある。現状 `onnxruntime` は `from .ort_utils import ...` 経由でモジュールレベル import されている。テスト環境に `onnxruntime` がインストールされていない場合は `ImportError` になる。

   **対策**: テスト環境 (`pyproject.toml` の test dependencies) に `onnxruntime` が含まれていることを確認する。もし含まれていない場合は `pytest.importorskip("onnxruntime")` を追加するか、`resolve_config_path()` を別モジュール (`config_utils.py` 等) に切り出す。ただし、現状のテストファイル (`test_infer_onnx_config.py`) は CI で実行されていると想定されるため、`onnxruntime` は存在するはず。

2. **関数シグネチャの型**: `config: str | None` は Python 3.10+ の union 構文。プロジェクトの Python バージョン要件 (3.10+) と合致することを確認する。

3. **`with_suffix()` の挙動**: `Path("model.onnx").with_suffix(".onnx.json")` ではなく `Path("model.onnx").with_suffix(Path("model.onnx").suffix + ".json")` を使用している。これは `with_suffix(".onnx.json")` が ValueError を投げる (ドットが2つ) ためであり、意図的な実装である。Rust 版は `with_extension("onnx.json")` でこの問題を回避している。

4. **テストファイル冒頭コメントの更新**: L1-8 のモジュール docstring がまだ「lines 143-152」を参照しており、行番号が古い。コメントを更新して「`infer_onnx.resolve_config_path()` を直接テスト」に変更する。

5. **`__all__` への追加**: `infer_onnx.py` に `__all__` が定義されている場合は `resolve_config_path` を追加する。定義されていない場合は不要。

## 6. 一から作り直すとしたら

もし config パス解決を今から設計するなら:

1. **専用モジュールへの配置**: `resolve_config_path()` を `infer_onnx.py` ではなく `piper_train/config_utils.py` (新規) に配置する。理由:
   - `infer_onnx.py` は ONNX 推論に特化したモジュールであり、config パス解決は汎用ロジック
   - `export_onnx.py` や将来の CLI ツールからも同じロジックを再利用できる
   - テスト時に `onnxruntime` 等の重い依存を import せずに済む

2. **存在確認の統合**: Rust 版のように、関数内でファイル存在確認を行い、見つからない場合は例外を投げる設計にする。現在の「パスを返すだけ + 呼び出し側で存在確認」は責務が分散しており、呼び出し側で確認漏れが起きやすい。

   ```python
   class ConfigNotFoundError(FileNotFoundError):
       """Config file not found after trying all fallback locations."""

   def resolve_config_path(model: str, config: str | None) -> Path:
       if config:
           p = Path(config)
           if p.exists():
               return p
           raise ConfigNotFoundError(f"Explicit config not found: {p}")

       model_path = Path(model)
       onnx_json = model_path.with_suffix(model_path.suffix + ".json")
       if onnx_json.exists():
           return onnx_json

       dir_config = model_path.parent / "config.json"
       if dir_config.exists():
           return dir_config

       raise ConfigNotFoundError(
           f"No config found for {model_path} "
           f"(tried {onnx_json}, {dir_config})"
       )
   ```

3. **パラメトライズテスト**: テストケースを `@pytest.mark.parametrize` でテーブル駆動にする:

   ```python
   class TestResolveConfigPath:
       @pytest.mark.parametrize(
           "setup, model_name, config_arg, expected_name",
           [
               (["model.onnx.json", "config.json"], "model.onnx", None, "model.onnx.json"),
               (["config.json"], "model.onnx", None, "config.json"),
               ([], "model.onnx", None, "config.json"),  # 不在パスを返す
               (["model.onnx.json"], "model.onnx", "custom/my.json", "custom/my.json"),
           ],
           ids=["onnx_json_preferred", "fallback_config", "neither_exists", "explicit_override"],
       )
       def test_resolve(self, tmp_path, setup, model_name, config_arg, expected_name):
           ...
   ```

4. **4言語実装の統一仕様**: Python / Rust / C# / Go の全実装で `resolve_config_path()` を持ち、`docs/spec/ort-session-contract.toml` と同様に `docs/spec/config-resolution.toml` で仕様を統一定義する。
