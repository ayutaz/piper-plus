## タスク目的とゴール

M2 マイルストーン全体の品質保証として、テストの通過確認・CI ワークフローの更新・ビルド時間の検証を行う。piper-g2p への移行により feature フラグの構成が変わるため、CI マトリクスが正しく動作することを確認する。

**ゴール**: CI (3 OS でそれぞれ 2 feature テストを直列実行) が全てグリーンで、既存のテスト資産 (24 integration テスト、96 PUA マッピングテスト含む) が全件パスする状態。

**注意**: 実際の CI 構造は 3 OS (ubuntu-24.04, macos-latest, windows-latest) でそれぞれ 2 つの feature テスト (`default`, `naist-jdic`) を**直列実行**する形式。3 x 2 のマトリクスではない。

## 実装する内容の詳細

### テスト確認事項

#### 1. PUA マッピングテスト (`test_token_map_parity.rs`)

`test_token_map_parity.rs` は 96 件の PUA (Private Use Area) マッピングの一致を検証するテスト。piper-g2p への移行後も、同一のマッピングが維持されることを確認する。

- piper-core の旧 `token_map.rs` で定義されていたマッピングが piper-g2p の `encode` モジュールに移行されているため、import パスの更新が必要な場合がある
- 96 件全てのマッピングが一致することを確認

#### 2. Integration テスト (24 ファイル)

`src/rust/piper-core/tests/` 配下の全 24 integration テストファイルがパスすることを確認する。特に以下に注目:

- `test_custom_dict_integration.rs` -- M2-6 の custom_dict 統合後のテスト
- `test_default_output.rs` -- 推論パイプラインの E2E テスト
- 各言語固有のテスト -- piper-g2p 経由の音素化が正しく動作すること

#### 3. CI ワークフロー (`rust-tests.yml`)

**確認事項:**
- feature フラグの変更 (`all-languages` の追加) が CI マトリクスに反映されていること
- 3 OS (Linux, macOS, Windows) x 2 feature コンボで全てビルド・テストが成功すること

## 懸念事項とレビュー項目

### 懸念事項

1. **ビルド時間の増加**: jpreprocess のコンパイルにより CI のビルド時間が大幅に増加する可能性がある。特に Windows のクリーンビルドは遅い傾向にあるため、タイムアウトの調整が必要になる場合がある
2. **Windows 固有の問題**: jpreprocess や lindera の依存が Windows で正しくビルドできることを確認する。過去に Windows 固有のビルドエラーが発生した実績がある場合は注意
3. **feature フラグの組み合わせ爆発**: `all-languages` + `naist-jdic` + `dict-download` の全組み合わせをテストすると CI 時間が膨大になる。実用的な 2 コンボに絞ることが妥当
4. **テストの flaky 化**: 辞書ダウンロードを伴うテストが CI 環境のネットワーク状態に依存して flaky になる可能性。`PIPER_OFFLINE_MODE=1` の設定を確認する

### レビュー項目

1. CI マトリクスの feature コンボが適切であること
2. `test_token_map_parity.rs` が piper-g2p のマッピングを参照するよう更新されていること
3. ビルド時間の計測結果が許容範囲内であること
4. 全 24 integration テストの結果が確認されていること
5. CI タイムアウトの設定が適切であること

## 一から作り直すとしたら

CI マトリクスの設計を最初から feature フラグの粒度に合わせて定義し、「デフォルト構成」と「最小構成」の 2 パターンに固定する。feature の組み合わせテストは定期ジョブ (nightly) で実行し、PR ジョブでは実用的な構成のみをテストする設計が効率的。
