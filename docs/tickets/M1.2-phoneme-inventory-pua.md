# M1.2 音素インベントリ + PUA割り当て

## マイルストーン紐づけ
- **マイルストーン**: M1.2
- **Phase**: Phase 1
- **対応要件**: FR-04
- **依存先チケット**: なし (M1.1 と並行可)
- **ブロックするチケット**: M1.3

## 達成したい内容
- スウェーデン語の音素インベントリ (19音素: 単一コードポイント10 + PUA必要な長母音9) を `sv_id_map.py` として定義する
- 9個の長母音 (`iː`, `yː`, `eː`, `ɛː`, `øː`, `ɑː`, `oː`, `uː`, `ʉː`) を `token_mapper.py` の `FIXED_PUA_MAPPING` に固定PUAとして追加する (0xE059-0xE061)
- `_PUA_START` を 0xE059 から 0xE064 に更新する (0xE062-0xE063 はSV将来拡張用の予備枠)
- 既存6言語 (JA/EN/ZH/KO/ES/FR/PT) のPUA割り当てと衝突しないことを検証する
- `register()` による双方向マッピング (`TOKEN2CHAR` / `CHAR2TOKEN`) が正しく動作することを確認する

## 影響範囲
- **新規ファイル**: `src/python/piper_train/phonemize/sv_id_map.py`
- **変更ファイル**: `src/python/piper_train/phonemize/token_mapper.py` (`FIXED_PUA_MAPPING` に9エントリ追加, `_PUA_START` 更新)
- **既存学習済みモデルへの影響**: なし (既存の固定PUA 0xE000-0xE058 は一切変更しない)
- **動的割り当てへの影響**: `_PUA_START` が 0xE059 → 0xE064 に移動するため、新規学習時の動的割り当て開始位置が変わる。既存モデルの `config.json` に保存された `phoneme_id_map` には影響しない
- **Phase 2 への影響**: C++/Rust/C# 実装に新規PUA定数を追加する必要がある (Phase 2 スコープ)
- **注意**: `FIXED_PUA_MAPPING` に追加した9エントリのコードポイントは、一度学習に使用されたら変更禁止 (ファイル先頭の注意書き準拠)

## 実装タスク

1. **`sv_id_map.py` の作成**: `src/python/piper_train/phonemize/sv_id_map.py` を新規作成する。`SWEDISH_PHONEMES: list[str]` に19音素 (単一CP: `ɖ`, `ʈ`, `ɳ`, `ɭ`, `ɧ`, `ɵ`, `ʏ`, `œ`, `ɑ`, `ø` の10個 + 長母音PUA: `iː`, `yː`, `eː`, `ɛː`, `øː`, `ɑː`, `oː`, `uː`, `ʉː` の9個) を定義する。モジュールロード時に `register()` を呼び出してPUAマッピングを登録する
2. **`FIXED_PUA_MAPPING` への9エントリ追加**: `token_mapper.py` の `FIXED_PUA_MAPPING` 辞書末尾 (FR `ɔ̃` 0xE058 の後) に、SV長母音9個のマッピングを追加する。Unicodeエスケープ表記で記述する (`"i\u02D0": 0xE059` 等)
3. **`_PUA_START` の更新**: `token_mapper.py` の `_PUA_START = 0xE059` を `_PUA_START = 0xE064` に変更する。コメントも更新し、0xE062-0xE063 が SV 将来拡張用の予備枠であることを明記する
4. **衝突検証の実装**: 全 `FIXED_PUA_MAPPING` の値 (コードポイント) に重複がないこと、全キー (トークン文字列) に重複がないことをテストで検証する

## 懸念点と影響範囲

| # | 懸念点 | 影響度 | 緩和策 |
|---|--------|--------|--------|
| 1 | PUAコリジョン: PR #294 等の並行開発で同じPUA範囲が使用される可能性 | 高 | `_PUA_START` の現在値 (0xE059) を確認し、他のPR/ブランチで 0xE059-0xE063 が使用されていないことをマージ前に検証する。`FIXED_PUA_MAPPING` の値の重複テストを必ず実施する |
| 2 | `_PUA_START` 更新による動的割り当ての変更 | 中 | 動的割り当ては学習時に毎回実行されるため、`_PUA_START` の変更は新規学習にのみ影響する。既存モデルの `config.json` に保存された `phoneme_id_map` はそのまま有効。テストで `_PUA_START == 0xE064` を検証する |
| 3 | 長母音の1トークン vs 2トークン設計判断の不可逆性 | 高 | 方式A (1トークンPUA) を採用済み。一度学習に使用されたPUAコードポイントは変更禁止。この設計判断はスウェーデン語の Complementary Quantity (母音長と子音長の補完的関係) が音韻的に重要であることに基づく |
| 4 | `sv_id_map.py` のインポート順序によるPUA登録タイミング | 低 | `sv_id_map.py` のモジュールレベルで `register()` を呼び出すため、インポート時に自動的にPUAが登録される。既存の `zh_id_map.py`, `ko_id_map.py` 等と同じパターン |
| 5 | 予備枠 (0xE062-0xE063) の誤使用 | 低 | 予備枠は `FIXED_PUA_MAPPING` に追加せず、コメントのみで明記する。`_PUA_START = 0xE064` で動的割り当てからも保護する |

## 次に進めるための内容
- `sv_id_map.py` が存在し、`SWEDISH_PHONEMES` リストに19音素が定義されている
- `token_mapper.py` の `FIXED_PUA_MAPPING` にSV長母音9エントリ (0xE059-0xE061) が追加されている
- `_PUA_START` が `0xE064` に更新されている
- 全PUA割り当て (既存 + SV) にコードポイントの重複がない
- 全PUA割り当てにトークンキーの重複がない
- `TOKEN2CHAR` と `CHAR2TOKEN` の双方向マッピングが整合している
- 既存6言語のテストが全て PASS する (非回帰)
- テスト16件が全て PASS する
- M1.3 (Rule-based G2P) が `sv_id_map.py` の音素インベントリを参照して G2P 出力を生成できる状態である

## レビューする内容
- `SWEDISH_PHONEMES` リストが仕様書 (FR-04 Section 1.2.1) の19音素と完全一致しているか
- 単一CPの音素10個が正しいUnicodeコードポイントか (特に: `ɖ` U+0256, `ʈ` U+0288, `ɳ` U+0273, `ɭ` U+026D, `ɧ` U+0267, `ɵ` U+0275, `ʏ` U+028F)
- `FIXED_PUA_MAPPING` の9エントリのキー文字列がUnicodeエスケープ表記で正確か (特に: `"\u025B\u02D0"` for `ɛː`, `"\u0289\u02D0"` for `ʉː`)
- PUAコードポイントが仕様の 0xE059-0xE061 と完全一致しているか
- `_PUA_START` のコメントが更新されているか (「0xE061 is the last used fixed codepoint (SV ʉː), so dynamic starts at 0xE064」)
- 既存の `FIXED_PUA_MAPPING` エントリ (0xE000-0xE058) に一切変更がないか
- `sv_id_map.py` のコーディングスタイルが既存の `zh_id_map.py`, `ko_id_map.py` 等と一貫しているか
- ファイル先頭の「Do NOT change assigned codepoints」の注意書きが保持されているか
