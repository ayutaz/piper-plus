# M1.1 NST辞書データ準備 (SAMPA→IPA変換)

## マイルストーン紐づけ
- **マイルストーン**: M1.1
- **Phase**: Phase 1
- **対応要件**: FR-02
- **依存先チケット**: なし
- **ブロックするチケット**: M1.3, M1.4

## 達成したい内容
- NST辞書 (OpenSLR 29, CC0ライセンス, 822K語) をダウンロードし、piper-plus で使用可能な IPA 形式の JSON 辞書に変換する
- 43 SAMPA音素 (長母音9, 短母音9, 基本子音16, 特殊子音2, レトロフレックス5, 二重母音2) の IPA マッピングテーブルを実装する
- CLI ツール `convert_nst_dictionary.py` を作成し、`--input`, `--output`, `--tier core|full`, `--gzip`, `--validate` オプションを提供する
- フィルタリング (F-1: !SIL, F-2: `<UNK>`, F-3: ハイフン接頭断片, F-4: 空発音, F-5: 重複エントリ) を適用し、Core辞書 (~238K語, 副ストレスなし=単純語) と Full辞書 (~821K語) を生成する
- スポットチェック20語の IPA 正確性を100%検証する

## 影響範囲
- **新規ファイル**: `src/python/piper_train/tools/convert_nst_dictionary.py`
- **新規成果物**: `sv_lexicon_core.json.gz` (~2.3 MB), `sv_lexicon_full.json.gz` (~10.6 MB) -- 成果物はコミット対象外 (ビルド前処理で生成)
- **既存ファイルへの影響**: なし (独立した前処理ツール)
- **依存パッケージ**: 追加なし (標準ライブラリの `json`, `gzip`, `argparse`, `unicodedata` のみ使用)
- **影響を受けるチーム**: M1.3 (Rule-based G2P) と M1.4 (辞書ルックアップ統合) がこの成果物に依存する

## 実装タスク

1. **NST辞書ダウンロード機構**: OpenSLR 29 から `lexicon-sv.tgz` を取得する手順をドキュメント化する。ツール自体にはダウンロード機能を含めず、ローカルの `lexicon.txt` を入力とする
2. **`NST_SAMPA_TO_IPA` マッピング定数の実装**: 仕様書 (FR-02 Section 2.2) に従い、長母音9, 短母音9, 基本子音16, 特殊子音2, レトロフレックス5, 二重母音2 の計43エントリを定義する。SAMPA `g` → IPA `ɡ` (U+0261, ASCII g ではない) に特に注意
3. **行パース関数 `parse_nst_line()`**: TSV形式 (`WORD\tSAMPA`) をパースし、小文字変換済み単語とSAMPA文字列のタプルを返す。不正行は None を返却し警告ログを出力する
4. **SAMPA→IPA変換関数 `convert_sampa_to_ipa()`**: スペース区切りのSAMPAトークンを順次変換。ストレス接頭辞 (`"` → `ˈ`, `%` → `ˌ`, `%"` → `ˌˈ`) を先に処理し、残りをマッピングテーブルで変換する。未知トークンは警告付きでパススルーする
5. **フィルタリング関数 `should_skip_entry()`**: F-1 (`!SIL`), F-2 (`<UNK>`), F-3 (ハイフン接頭断片), F-4 (空発音), F-5 (重複) の5条件を判定する
6. **Coreティアフィルタ `is_simple_word()`**: SAMPAに副ストレス (`%`) を含まない語を単純語として判定する
7. **CLI `main()` の実装**: `argparse` で `--input`, `--output`, `--gzip`, `--tier core|full`, `--validate`, `--stats`, `--quiet` を受け付ける。出力はキーでソート済みJSON (再現性確保)、gzip圧縮はオプション
8. **スポットチェック検証 `run_spot_check()`**: 仕様書の20語テーブル (V-1〜V-20) を `--validate` 時に実行する。全語一致で成功、1語でも不一致で終了コード4を返す
9. **統計情報出力**: 変換完了時に入力行数, 有効エントリ数, 各フィルタ理由別スキップ数, 未知SAMPAトークン数, 出力エントリ数, 出力ファイルサイズを stderr に出力する

## 懸念点と影響範囲

| # | 懸念点 | 影響度 | 緩和策 |
|---|--------|--------|--------|
| 1 | NST辞書の品質 (機械生成エントリの存在) | 中 | Core ティア (~238K語) は副ストレスなし=単純語に限定することで複合語の誤変換を回避。スポットチェック20語で代表的な音素カテゴリをカバー |
| 2 | エンコーディング問題 (NST原本のUTF-8以外の可能性) | 低 | OpenSLR 29 版は UTF-8。`open()` で `encoding="utf-8"` を明示し、`UnicodeDecodeError` をキャッチして致命的エラーとする |
| 3 | SAMPAの曖昧性 (`s'` vs `s` + `'`, `u0` vs `u` + `0`) | 低 | NST辞書はスペース区切りで各SAMPAトークンが独立しているため、部分一致の曖昧性は発生しない。仕様書 Section 2.3.4 で確認済み |
| 4 | 複数発音エントリ (2,105語) の扱い | 低 | 最初のバリアント (最も標準的な発音) のみ採用する方針。Phase 2以降でカスタム辞書による上書きサポートを検討 |
| 5 | OpenSLR 29 のファイル形式変更リスク | 低 | ダウンロードURLとフォーマットをドキュメント化し、変更時は `parse_nst_line()` を調整。ハッシュチェックを追加して整合性を検証 |
| 6 | 辞書サイズによる CI の遅延 | 低 | Core辞書 (gzip ~2.3 MB) は成果物としてコミット対象外。テストはモック辞書 (数十エントリ) を使用する |

## 次に進めるための内容
- `convert_nst_dictionary.py` が `--tier core` と `--tier full` の両方で正常に JSON 辞書を出力できる
- `--validate` オプションでスポットチェック20語 (V-1〜V-20) が全問正解 (終了コード 0)
- `NST_SAMPA_TO_IPA` テーブルの43エントリ全てが正しい IPA にマッピングされている
- フィルタリング (F-1〜F-5) が正しく動作し、統計情報が期待値と整合する
- テスト26件が全て PASS する
- M1.4 (辞書ルックアップ統合) が生成された JSON 辞書をロードして使用できる状態である

## レビューする内容
- `NST_SAMPA_TO_IPA` マッピングテーブルが仕様書 (FR-02 Section 2.2) と完全一致しているか (特に: SAMPA `g` → IPA `ɡ` U+0261, SAMPA `u0` → IPA `ʉ` U+0289)
- ストレス接頭辞の処理順序が正しいか (`%"` → `ˌˈ` を `"` → `ˈ` より先に判定しているか)
- フィルタリング条件が仕様の5パターン (F-1〜F-5) を網羅しているか
- Core ティアフィルタの判定基準 (`%` の有無) が仕様と一致するか
- 出力JSONのキーソート、UTF-8エンコーディング (BOMなし)、gzip圧縮が仕様通りか
- スポットチェックテーブルの20語が仕様書 (FR-02 Section 2.7.1) と一致しているか
- CLI の終了コード (0/1/2/3/4) が仕様通りか
- 既存コード (他の `tools/` スクリプト) とのコーディングスタイル整合性
