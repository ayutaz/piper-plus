## 1. 目的とゴール

`test_prosody_extraction.py` L18-38 と `test_phonemize.py` L26-31 に、ほぼ同一のBOS/EOSラッパー関数 `phonemize_japanese()` / `phonemize_japanese_with_prosody()` が重複定義されている。これらを共通ユーティリティとして抽出し、DRY 原則に従う。

**`test_prosody_extraction.py` L18-38 のラッパー**:

```python
_EOS_TOKENS = {"$", "?", "?!", "?.", "?~"}

def phonemize_japanese(text):
    """Wrapper that matches old piper_train API: returns PUA-mapped tokens with BOS/EOS."""
    p = JapanesePhonemizer()
    tokens = p.phonemize(text)
    # piper_plus_g2p returns raw tokens; add BOS and EOS
    full_tokens = ["^"] + list(tokens)
    if not tokens or tokens[-1] not in _EOS_TOKENS:
        full_tokens.append("$")
    return [_map_token(t) for t in full_tokens]

def phonemize_japanese_with_prosody(text):
    """Wrapper that matches old piper_train API: returns PUA-mapped tokens+prosody with BOS/EOS."""
    p = JapanesePhonemizer()
    tokens, prosody = p.phonemize_with_prosody(text)
    full_tokens = ["^"] + list(tokens)
    full_prosody = [None] + list(prosody)
    if not tokens or tokens[-1] not in _EOS_TOKENS:
        full_tokens.append("$")
        full_prosody.append(None)
    mapped_tokens = [_map_token(t) for t in full_tokens]
    return mapped_tokens, full_prosody
```

**`test_phonemize.py` L26-31 のラッパー**:

```python
def phonemize_japanese(text):
    """Wrapper that matches old piper_train API: returns PUA-mapped tokens with BOS/EOS."""
    p = JapanesePhonemizer()
    tokens = p.phonemize(text)
    full_tokens = ["^"] + tokens + ["$"]
    return [_map_token(t) for t in full_tokens]
```

**問題点**:
1. **DRY 違反**: 2 ファイルに同一パターンのラッパーが重複定義
2. **微妙な差異**: `test_prosody_extraction.py` 版は EOS トークン (`?`, `?!` 等) の末尾チェックを行い、既に EOS がある場合は `$` を追加しない。`test_phonemize.py` 版は無条件に `$` を追加する。この差異がバグなのか意図的なのか不明確
3. **BOS/EOS の追加ロジック**: `piper_plus_g2p` のG2P出力にBOS (`^`) / EOS (`$`) を追加するロジックは、本番ランタイム (`src/python_run/piper/phonemize/`) でも必要であり、テスト内に閉じ込めるべきではない

**完了の定義**:
- `phonemize_japanese()` と `phonemize_japanese_with_prosody()` を共通テストユーティリティとして `conftest.py` または専用ヘルパーモジュールに抽出
- 2 ファイル間の EOS 処理の差異を統一 (EOS トークンチェック版に統一)
- 両テストファイルが共通ユーティリティを import して使用

    def phonemize_japanese_with_prosody(text):

        p = JapanesePhonemizer()

        full_prosody = [None] + list(prosody)

            full_tokens.append("$")
            full_prosody.append(None)

    HAS_JAPANESE = True
except ImportError:

### 2.4 EOS 処理の差異統一

`test_phonemize.py` 版は `full_tokens = ["^"] + tokens + ["$"]` と無条件に `$` を追加する。疑問文 (`?`, `?!` 等) で終わるテキストの場合、`["?", "$"]` のように EOS が二重になる。`test_prosody_extraction.py` 版の EOS チェックロジックに統一する。

## 5. 懸念事項とレビュー項目

1. **EOS 二重付与のバグ修正**: `test_phonemize.py` 版の無条件 `$` 追加を EOS チェック版に統一すると、一部テストの期待値が変わる可能性がある。具体的には、疑問文のテストケースで末尾が `["?", "$"]` から `["?"]` に変わる。影響を受けるテストを全数確認すること。

2. **conftest.py の肥大化**: 既存の `conftest.py` にはモデルフィクスチャ等が定義されている。BOS/EOS ヘルパーを追加することで conftest.py の責務が広がる。代替案として `tests/helpers/japanese.py` のような専用モジュールを作成してもよい。

3. **`HAS_JAPANESE` フラグの統一**: 両テストファイルで独立に `HAS_JAPANESE` を定義している。conftest.py で一元管理することで、pyopenjtalk がインストールされていない環境でのスキップ処理が統一される。

4. **pytest の conftest.py 自動インポート**: pytest は `conftest.py` のトップレベル定義を自動で利用可能にするが、明示的に `from conftest import ...` するかフィクスチャとして提供するかは設計判断。関数として直接呼び出すなら `from conftest import` が明快。

## 6. 一から作り直すとしたら

BOS/EOS の追加はG2Pパッケージ (`piper_plus_g2p`) の責務とすべき。`JapanesePhonemizer.phonemize()` が最初から BOS/EOS 付きのトークン列を返すようにすれば、テスト側のラッパーは不要になる。ただし `piper_plus_g2p` は独立パッケージ (PyPI 公開済み) であり、BOS/EOS はランタイム固有の概念であるため、G2P 側に含めるかは議論が必要。代替案として、ランタイム側 (`piper/phonemize/`) に `wrap_with_bos_eos()` ユーティリティを追加し、テストはそれを import する設計が望ましい。
