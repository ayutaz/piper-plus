# [M3-4] Rust/C#/Go/WASM ランタイムに Voice Cloning 統合

**マイルストーン**: M3a (v2.0.0)
**カテゴリ**: Voice Cloning
**工数**: 10d
**依存**: M3-2, M3-3
**ステータス**: 未着手

---

## 1. 目的とゴール

M3-3 で学習した Voice Cloning モデル (VITS2 + speaker_embedding) を、Python / Rust / C# / Go / WASM の全5ランタイムで利用可能にする。参照音声から話者 embedding を抽出し、VITS 推論に注入する一連のパイプラインを各言語で実装する。

**ゴール:**
- 全5ランタイムで `synthesize(text, reference_audio)` 相当の API を提供
- Speaker Encoder ONNX (音声波形入力 -> embedding 出力の single ONNX) の推論を各ランタイムで実装
- 全ランタイムで同一参照音声 -> 同一 embedding (誤差 L2 < 1e-3) を保証

## 2. 実装内容の詳細

### 2.1 Speaker Encoder ONNX の設計 (本案)

**本案: メルスペクトログラム変換を Speaker Encoder ONNX に内包**

音声波形入力 -> embedding 出力の single ONNX モデルとして提供する。これにより各ランタイムでのメルスペクトログラム実装が不要となり、実装差異による embedding の不一致問題を根本的に解消する。

```
[音声波形 (16kHz, float32)] -> [Speaker Encoder ONNX] -> [256次元 embedding]
                                (メル変換 + ECAPA-TDNN を内包)
```

| 入力テンソル | 型 | 形状 | 説明 |
|------------|-----|------|------|
| `audio` | float32 | (1, samples) | 16kHz モノラル音声波形 |

| 出力テンソル | 型 | 形状 | 説明 |
|------------|-----|------|------|
| `embedding` | float32 | (1, 256) | L2 正規化済み話者 embedding |

ONNX opset 17 の DFT/STFT オペレータを活用してメルスペクトログラム変換をグラフに含める。各ランタイムは音声のリサンプリング (16kHz 化) のみ実装すれば良い。

**代替案: 各ランタイムでメル実装**

メルスペクトログラム変換をランタイム側で実装し、Speaker Encoder ONNX はメル入力のみ受け付ける方式。

| ランタイム | 工数 | 音声前処理 |
|-----------|------|-----------|
| Python | 1.5d | numpy + scipy |
| Rust | 2.5d | ndarray + rubato |
| C# | 2.5d | NAudio + custom mel |
| Go | 2d | gonum + custom mel |
| WASM | 1.5d | AudioContext API |

5ランタイムでメルスペクトログラム出力の一致を保証する必要があり (L2 < 1e-3)、テスト・デバッグコストが高い。本案 (ONNX 内包) が実現困難な場合のフォールバックとして位置付ける。

### 2.3 共通仕様 (docs/spec/speaker-encoder-preprocess.toml)

```toml
[audio]
sample_rate = 16000   # 入力音声は各ランタイムで 16kHz にリサンプリング

[encoder]
model_name = "speaker_encoder.onnx"
# 本案: 音声波形入力 -> embedding 出力 (メル変換は ONNX 内包)
input_tensor = "audio"        # (1, samples) float32, 16kHz mono
output_tensor = "embedding"   # (1, 256) float32, L2 正規化済み
l2_normalize = true

[encoder.mel_params]
# ONNX グラフ内のメルスペクトログラム仕様 (参考情報、ランタイム実装不要)
n_fft = 400        # 25ms @ 16kHz
hop_length = 160   # 10ms @ 16kHz
n_mels = 80
fmin = 0
fmax = 8000
window = "hann"
normalize = true   # per-utterance mean/var normalization

[vits_integration]
input_tensor = "speaker_embedding"       # (1, 256)
mask_tensor = "speaker_embedding_mask"   # (1,) int64, 0=disabled, 1=enabled
```

### 2.4 Python 実装

```python
# api.py に追加
class PiperPlus:
    def synthesize(self, text: str, *, reference_audio: str | np.ndarray | None = None,
                   speaker_id: int = 0, **kwargs) -> AudioResult:
        if reference_audio is not None:
            embedding = self._speaker_encoder.encode(reference_audio)
            return self._infer_with_embedding(text, embedding, **kwargs)
        return self._infer_with_sid(text, speaker_id, **kwargs)
```

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **メルスペクトログラムの実装差異**: 本案 (ONNX 内包) を採用することで、メル実装の言語間差異問題を解消。各ランタイムはリサンプリング (16kHz 化) のみ実装すれば良い。ONNX opset 17 の DFT/STFT オペレータを使用するため、全ランタイムの ONNX Runtime が opset 17 以上に対応している必要がある。WASM の ONNX Runtime バージョン対応を事前確認すること
2. **Speaker Encoder モデルのダウンロード**: 各ランタイムで既存のモデル自動DL機構に Speaker Encoder を追加する必要がある。`ayousanz/piper-plus-speaker-encoder` からの DL パスを統一
3. **WASM のメモリ制約**: Speaker Encoder (~5-8MB) + VITS モデル (~75MB) をブラウザメモリに同時ロード。WebGPU 利用で緩和できるが、モバイルブラウザでは OOM のリスク
4. **全実装の同期リリース**: 5ランタイム全ての実装が揃わないとリリースできない。並行開発でボトルネックが発生しやすい

### レビューチェックリスト

- [ ] 全ランタイムのメルスペクトログラム出力が Python 基準で L2 < 1e-3
- [ ] 全ランタイムの embedding 出力が Python 基準で cosine similarity > 0.999
- [ ] 全ランタイムの CLI に `--reference-audio` オプションが追加されている

## 6. 一から作り直すとしたら

**マルチランタイム同期開発の課題:**

5ランタイムへの同時展開は piper-plus の強みだが、Voice Cloning のような複雑な機能を全ランタイムに同時実装するのはリスクが高い。ゼロから設計するなら:

1. **Phase 1**: Python + Rust のみで Voice Cloning をリリース (v2.0.0-alpha)
2. **Phase 2**: C# / Go を追加 (v2.0.0-beta)
3. **Phase 3**: WASM を追加 (v2.0.0)

この段階的リリースにより、Phase 1 で API 設計を固め、Phase 2-3 での手戻りを最小化できる。

**メルスペクトログラム実装の統一について:**

本案ではメルスペクトログラム変換を Speaker Encoder ONNX に内包する方式を採用した。ONNX opset 17 の DFT/STFT オペレータを活用することで、ランタイム側の前処理を音声リサンプリングのみに削減できる。代替案 (各ランタイムでメル実装) と比較して実装コストが低く、実装差異によるバグのリスクも排除できる。

**Voice Cloning のアーキテクチャ選択の振り返り:**

ZSE-VITS (Speaker Encoder + VITS) は「既存 VITS に最小限の変更で Voice Cloning を追加する」アプローチとして適切。ただし、品質の上限は Speaker Encoder の性能に制約される。ゼロから設計するなら、VALL-E のような codec-based approach や CosyVoice の Flow Matching approach のほうが品質上限は高い。しかし、これらはモデルサイズが大きく (500MB+)、piper-plus の軽量性コンセプトと矛盾する。

M3a 全体として「軽量 Voice Cloning」という方向性は正しく、ECAPA-TDNN + VITS2 はその制約下での最適解に近い。
