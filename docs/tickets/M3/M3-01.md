# [M3-1] Speaker Encoder (ECAPA-TDNN) ONNX 統合

**マイルストーン**: M3a (v2.0.0)
**カテゴリ**: Voice Cloning
**工数**: 10d
**依存**: M2完了
**ステータス**: 未着手

---

## 1. 目的とゴール

参照音声 (3-10秒) から話者 embedding を抽出する Speaker Encoder を ONNX モデルとして統合する。ECAPA-TDNN アーキテクチャを採用し、既存の piper-plus エコシステム (Python / Rust / C# / Go / WASM) 全てで利用可能な形式で提供する。

**ゴール:**
- ECAPA-TDNN の事前学習済みモデルを ONNX 形式 (FP16, ~5-15MB) でエクスポート
- Python で `SpeakerEncoder` クラスを実装し、WAV 入力 -> 256次元 embedding 出力のパイプラインを構築
- 話者類似度メトリクス (cosine similarity) の評価ユーティリティを同梱
- HuggingFace にエンコーダモデルをアップロードし、各ランタイムから自動 DL 可能にする

## 2. 実装内容の詳細

### 2.1 ファイル構成

```
src/python/piper_train/speaker_encoder/
  __init__.py              # SpeakerEncoder クラスを re-export

  export_encoder.py        # PyTorch -> ONNX 変換スクリプト
  evaluate.py              # cosine similarity 評価ユーティリティ
```

### 2.2 ECAPA-TDNN モデル仕様

| パラメータ | 値 | 根拠 |
|-----------|-----|------|
| アーキテクチャ | ECAPA-TDNN (1024 channels) | SpeechBrain の検証済みモデル |
| 入力 | 80-dim メルスペクトログラム (16kHz, 25ms窓, 10msシフト) | VITS と同一のサンプリングレート |
| 出力 | 256次元 embedding (L2正規化済み) | emb_g と同次元にマッピング可能 (※下記 Day 1 確認事項参照) |
| 事前学習 | VoxCeleb1+2 (~7,000話者) | 話者弁別性能の実績 (※下記ライセンス確認参照) |
| ONNX サイズ | FP16: ~5-8MB, FP32: ~15MB | RPi/ブラウザでも実用的 |

### 2.3 SpeakerEncoder クラス

```python
class SpeakerEncoder:
    def __init__(self, model_path: str | None = None, *, device: str = "cpu"):
        """ONNX Speaker Encoder を初期化。model_path=None で自動DL。"""

    def encode(self, audio: np.ndarray, sr: int = 16000) -> np.ndarray:
        """音声波形 -> 256次元 embedding (L2正規化済み)。"""

    def encode_file(self, wav_path: str) -> np.ndarray:
        """WAV ファイル -> embedding。リサンプリング自動。"""

    def similarity(self, emb1: np.ndarray, emb2: np.ndarray) -> float:
        """2つの embedding の cosine similarity を返す。"""
```

### 2.4 音声前処理パイプライン

1. WAV 読み込み (任意サンプリングレート)
2. 16kHz にリサンプリング (librosa or scipy)
3. エネルギー VAD で無音区間除去 (既存 `norm_audio` の VAD を再利用)
4. 80-dim メルスペクトログラム変換
5. ONNX 推論 -> 256次元 embedding
6. L2 正規化

### 2.5 Day 1 事前確認タスク

本チケット着手初日に以下を確認し、設計パラメータを確定する。

**a) ECAPA-TDNN 出力次元の確認:**
SpeechBrain の `spkrec-ecapa-voxceleb` モデルの出力次元を実測する (192次元: SpeechBrain standard / 256次元: large variant)。

```bash
uv run python -c "
from speechbrain.pretrained import EncoderClassifier
model = EncoderClassifier.from_hparams(source='speechbrain/spkrec-ecapa-voxceleb')
import torch
dummy = torch.randn(1, 16000)
emb = model.encode_batch(dummy)
print(f'Output dim: {emb.shape[-1]}')
"
```

- **256次元の場合**: VITS の `gin_channels=256` と一致。射影層不要
- **192次元の場合**: 線形射影層 `nn.Linear(192, 256)` を M3-2 で追加。M3-02 連携事項に射影層の学習戦略 (frozen encoder + trainable projection) を含める

**b) VoxCeleb 事前学習モデルのライセンス確認:**
SpeechBrain `spkrec-ecapa-voxceleb` の重み再配布ライセンスを確認する。Creative Commons / VoxCeleb のデータ利用規約が推論用途の重み再配布を許容するか検証。

- **許容される場合**: そのまま HuggingFace にアップロード
- **不可の場合 (フォールバック)**: WeSpeaker ECAPA-TDNN (Apache-2.0 ライセンス、VoxCeleb 学習済み) に切り替え。WeSpeaker モデルの出力次元も同時に確認する

### 2.6 ONNX エクスポート

## 4. テスト計画

### 提供範囲（スコープ）
- IN: ECAPA-TDNN ONNX モデルのエクスポート、SpeakerEncoder クラス、音声前処理、cosine similarity 評価
- OUT: VITS への embedding 注入 (M3-2)、各ランタイム統合 (M3-4)、学習パイプライン (M3-3)

### ユニットテスト

- `test_encoder_init`: モデルパス / 自動DL での初期化
- `test_encode_shape`: 出力が (256,) の L2 正規化ベクトルであること
- `test_encode_deterministic`: 同一入力で同一出力 (再現性)
- `test_encode_different_speakers`: 異なる話者で cosine similarity < 0.7
- `test_encode_same_speaker`: 同一話者の異なる発話で cosine similarity > 0.85
- `test_audio_preprocessing`: リサンプリング、VAD、メルスペクトログラム変換の正当性
- `test_variable_length`: 3秒、5秒、10秒の音声で全て正常動作

### E2Eテスト

- VoxCeleb1-O テストセット (40話者) で EER (Equal Error Rate) を測定
- 目標: EER < 3.0% (SpeechBrain 公式: 0.8%)
- ONNX vs PyTorch の出力差が L2 norm < 1e-4 であること

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **SpeechBrain 依存**: エクスポート時のみ SpeechBrain が必要。ランタイムでは ONNX のみ。ただしエクスポートスクリプトの `requirements-export.txt` を分離する必要あり
2. **メルスペクトログラム実装の一貫性**: Python (numpy/scipy) と Rust/C#/Go/WASM で同一の前処理結果を保証する必要がある。仕様を `docs/spec/speaker-encoder-preprocess.toml` に定義
3. **モデルサイズと RPi 性能**: FP16 で ~5-8MB はモバイル/RPi で許容範囲だが、初回推論の JIT 遅延が ~500ms 追加される可能性。warmup で対処
4. **VoxCeleb ライセンス**: VoxCeleb データセット自体は研究目的。事前学習済みモデルの重みの再配布ライセンスを確認する必要あり

### レビューチェックリスト

- [ ] ONNX モデルの入出力テンソル名・形状が仕様書と一致
- [ ] FP16 変換後の精度劣化が EER +0.5% 以内
- [ ] 音声前処理仕様が `docs/spec/` にドキュメント化されている
- [ ] GPL 依存 (espeak-ng 等) が混入していない
- [ ] HuggingFace リポジトリのライセンス表記が正しい
- [ ] 自動DL がオフラインモード (`PIPER_OFFLINE_MODE=1`) で無効化される

## 6. 一から作り直すとしたら

**Speaker Encoder のアーキテクチャ選択について:** ECAPA-TDNN は話者検証タスクでの実績が豊富だが、2024年以降は WavLM + X-vector や DINO-based speaker encoder など Transformer ベースのアプローチが台頭している。ゼロから設計するなら、以下の選択肢を比較検証してから決定すべき:

1. **ECAPA-TDNN (現案)**: 軽量 (5-15MB)、高速推論、ONNX 変換が安定。実績豊富
2. **WavLM-based**: 音響特徴の汎化性能が高いが、モデルサイズが大きい (~100MB+)。ブラウザ/RPi には非現実的
3. **CAM++ (Alibaba)**: ECAPA-TDNN より高精度で同等サイズ。ただし事前学習済みモデルの再配布ライセンスが不明確

M3a 全体の設計としては、Speaker Encoder の選定に Go/No-Go ゲート (M3-GATE) を設けているのは正しいアプローチ。ただし、ゲートの判定基準を cosine similarity だけでなく、実際の VITS embedding 空間へのマッピング後の MOS 評価まで含めるべきだった。cosine similarity が高くても、VITS の `emb_g` 空間に射影した際に音質が劣化するケースがある。

また、Speaker Encoder のメルスペクトログラム仕様を最初から全ランタイムで統一するための仕様書 (TOML) を M3-1 の最初の成果物にすべき。後から仕様の不一致が発覚すると、M3-4 (全ランタイム統合) で手戻りが大きい。

## 7. 後続タスクへの連絡事項

- **M3-GATE**: 本チケットの成果物 (SpeakerEncoder + ONNX モデル) を使って話者類似度を評価する。評価データセットと基準値は M3-GATE で定義
- **M3-2**: `SpeakerEncoder.encode()` の出力形状と VITS の `emb_g` (gin_channels=256) の次元マッチングが必要。Day 1 確認で ECAPA-TDNN 出力が 192次元の場合、M3-2 で線形射影層 `nn.Linear(192, 256)` を追加する。射影層の学習戦略: Speaker Encoder は frozen、射影層のみ trainable として Voice Cloning 学習 (M3-3) で最適化
- **M3-4**: 各ランタイムでの音声前処理仕様を `docs/spec/speaker-encoder-preprocess.toml` に定義する。Rust/C#/Go/WASM はこの仕様に準拠して実装
- HuggingFace リポジトリ URL: `ayousanz/piper-plus-speaker-encoder`
