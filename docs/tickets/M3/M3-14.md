# [M3-14] モデル投稿ガイド + Issue テンプレート

**マイルストーン**: M3b (v2.1.0)
**カテゴリ**: モデルZoo
**工数**: 1d
**依存**: M3-9
**ステータス**: 未着手

---

## 1. 目的とゴール

コミュニティメンバーが自分の音声データやデータセットを使って piper-plus モデルを学習し、モデルZoo に投稿できるようにするためのガイドとテンプレートを整備する。M3-9 (LJSpeech) の実例をベースに、再現可能な手順を文書化する。

**ゴール:**
- `CONTRIBUTING_MODELS.md` を作成 (モデル投稿の手順書)
- `.github/ISSUE_TEMPLATE/model-request.yml` を作成 (モデルリクエスト Issue テンプレート)
- `.github/ISSUE_TEMPLATE/model-submission.yml` を作成 (モデル投稿 Issue テンプレート)
- 品質チェックリスト + ライセンス確認フローを定義

## 2. 実装内容の詳細

### 2.1 CONTRIBUTING_MODELS.md

```
CONTRIBUTING_MODELS.md
├── 1. 概要
├── 2. 前提条件
│   ├── piper-plus のインストール
│   ├── GPU 環境 (推奨: V100 以上)
│   └── 必要な Python パッケージ
├── 3. データセット準備
│   ├── サポートされるデータ形式
│   ├── LJSpeech 形式への変換
│   └── 品質要件 (SNR, サンプリングレート)
├── 4. 学習
│   ├── Template B (シングルスピーカー) の使い方
│   ├── パラメータの調整ガイド
│   └── WandB でのモニタリング
├── 5. ONNX エクスポート
│   ├── FP16 エクスポート手順
│   └── 品質検証コマンド
├── 6. 投稿手順
│   ├── HuggingFace リポジトリの作成
│   ├── voices.json へのエントリ追加
│   └── Issue での投稿申請
├── 7. ライセンスガイドライン
│   ├── 許可されるライセンス
│   ├── Model Card への記載事項
│   └── 非推奨ライセンス (GPL 系)
└── 8. 品質チェックリスト
```

### 2.2 品質チェックリスト

投稿モデルが満たすべき最低基準:

| # | チェック項目 | 必須/推奨 |
|---|------------|----------|
| 1 | 6言語各1文で推論成功 | 必須 |
| 2 | 音声長が 1-10s の範囲 (無音/ピー音でない) | 必須 |
| 3 | FP16 ONNX エクスポート成功 | 必須 |
| 4 | config.json の num_symbols がベースモデルと一致すること | 必須 |
| 5 | ライセンスが明記されている | 必須 |
| 6 | Model Card が存在する | 必須 |
| 7 | `md5_digest` が計算済み | 必須 |
| 8 | MOS スコア >= 3.0 (主観評価) | 推奨 |
| 9 | 長文 (100文字超) で安定 | 推奨 |
| 10 | WandB のトレーニングログ共有 | 推奨 |

### 2.3 ライセンスガイドライン

| ライセンス | 投稿可否 | 備考 |
|-----------|---------|------|
| CC0 / Public Domain | 推奨 | 制約なし |
| CC BY 4.0 | 可 | 帰属表記必須 |
| CC BY-SA 4.0 | 可 | 派生物も同一ライセンス |
| CC BY-NC 4.0 | 条件付き可 | 非商用限定であることを明記 |
| Apache 2.0 / MIT | 可 | |
| GPL 系 | 不可 | エコシステム全体への影響 |
| 独自ライセンス | 要審査 | 投稿時にライセンス全文を添付 |

### 2.4 Issue テンプレート

**model-request.yml** (モデルリクエスト):
```yaml
name: "Model Request"
description: "Request a new voice model for piper-plus"
labels: ["model-request"]
body:
  - type: dropdown
    id: language
    attributes:
      label: Language
      options: [Japanese, English, Chinese, Korean, Spanish, French, Portuguese, Swedish, Other]
  - type: input
    id: dataset
    attributes:
      label: Dataset name and URL
  - type: input
    id: license
    attributes:
      label: Dataset license
  - type: textarea
    id: reason
    attributes:
      label: Why this model would be valuable
```

**model-submission.yml** (モデル投稿):
```yaml
name: "Model Submission"
description: "Submit a trained voice model to piper-plus model zoo"
labels: ["model-submission"]
body:
  - type: input
    id: model_name
    attributes:
      label: Model name (e.g., "en_US-ljspeech-medium")
  - type: input
    id: huggingface_repo
    attributes:
      label: HuggingFace repository URL
  - type: input
    id: license
    attributes:
      label: License
  - type: checkboxes
    id: quality_checks
    attributes:
      label: Quality checklist
      options:
        - label: "6言語各1文で推論成功"
        - label: "音声長 1-10s (無音/ピー音なし)"
        - label: "FP16 ONNX エクスポート成功"
        - label: "config.json num_symbols がベースモデルと一致"
        - label: "ライセンス明記済み"
        - label: "Model Card 作成済み"
        - label: "md5_digest 計算済み"
```

### 2.5 voices.json エントリテンプレート

投稿者が `piper_plus_voices.json` に追加するエントリのテンプレート:

```json
{
    "<lang>_<REGION>-<name>-medium": {
        "key": "<lang>_<REGION>-<name>-medium",
        "name": "<name>",
        "language": {
            "code": "<lang>_<REGION>",
            "family": "<lang>",
            "region": "<REGION>",
            "name_native": "...",
            "name_english": "...",
            "country_english": "..."
        },
        "quality": "medium",
        "num_speakers": 1,
        "speaker_id_map": {},
        "source": "community",
        "repo": "<username>/<repo-name>",
        "files": {
            "<model-name>.onnx": { "size_bytes": 0, "md5_digest": "" },
            "config.json": { "size_bytes": 0, "md5_digest": "" }
        },
        "aliases": [],
        "description": "..."
    }
}
```

## 4. テスト計画

### 提供範囲（スコープ）
- IN: CONTRIBUTING_MODELS.md、Issue テンプレート 2種、品質チェックリスト、ライセンスガイドライン
- OUT: 自動品質検証 CI (将来課題)、投稿モデルの実際のレビュー

### ユニットテスト

- Issue テンプレート YAML の構文検証
- voices.json テンプレートが既存スキーマと整合するか

### E2Eテスト

- CONTRIBUTING_MODELS.md の手順に従って LJSpeech モデル (M3-9) を再現できるか (ドキュメントのウォークスルー)
- Issue テンプレートを使って実際に Issue を作成し、必要情報が収集できるか

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **投稿モデルの品質管理**: 自動検証 CI がないため、投稿されたモデルの品質はメンテナが手動で確認する必要がある。投稿数が増えた場合のスケーラビリティが課題
2. **ライセンスの多様性**: コミュニティ投稿のライセンスが多岐にわたる可能性。審査の負荷が高い
3. **voices.json の競合**: 複数の投稿が同時に voices.json を変更する場合のマージ競合

### レビューチェックリスト

- [ ] CONTRIBUTING_MODELS.md の手順が M3-9 の実際の作業と一致しているか
- [ ] Issue テンプレートに必要十分な情報が含まれているか
- [ ] ライセンスガイドラインが GPL 排除方針と整合しているか
- [ ] voices.json テンプレートが既存エントリと同一スキーマか
- [ ] 英語と日本語の両方でガイドが提供されているか (最低限英語)

## 6. 一から作り直すとしたら

コミュニティモデル投稿フローを最初から設計するなら、GitHub Issue ベースではなく専用の Web フォーム + 自動検証パイプラインを構築する。投稿者が HuggingFace リポジトリ URL を入力すると、CI が自動でモデルをダウンロードし、品質チェックリストを実行し、合格すれば自動で voices.json に追加する。ただし現段階ではコミュニティの規模が小さいため、Issue ベースの手動フローで十分。投稿数が月 5 件を超えた時点で自動化を検討する。
