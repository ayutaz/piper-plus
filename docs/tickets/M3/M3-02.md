# [M3-2] SynthesizerTrn に `speaker_embedding` 入力パス追加

**マイルストーン**: M3a (v2.0.0)
**カテゴリ**: Voice Cloning
**工数**: 5d
**依存**: M3-GATE 通過
**ステータス**: 未着手

---

## 1. 目的とゴール

VITS の SynthesizerTrn モデルに、外部から speaker embedding を直接注入する入力パスを追加する。既存の `speaker_id` (整数 -> embedding lookup) に加え、Speaker Encoder が出力する連続ベクトルを受け取る `speaker_embedding` テンソルを ONNX グラフに追加する。既存モデルとの後方互換性を維持する。

**ゴール:**
- `SynthesizerTrn` の forward パスに `speaker_embedding: Optional[Tensor]` 引数を追加
- ONNX エクスポートで `speaker_embedding` を Optional 入力テンソルとして出力
- `speaker_embedding` が与えられた場合は `emb_g` lookup をスキップし、直接使用
- 既存の `speaker_id` ベースの推論が全く影響を受けないこと

## 2. 実装内容の詳細

### 2.1 models.py の変更

```python
# SynthesizerTrn.forward() / infer() に引数追加
def infer(self, x, x_lengths, sid=None, lid=None,
          noise_scale=0.667, length_scale=1.0, noise_w=0.8,
          prosody_features=None,
          speaker_embedding=None,       # NEW
          speaker_embedding_mask=None):  # NEW
    # ...
    g = self._get_global_conditioning(sid, lid, speaker_embedding, speaker_embedding_mask)
```

### 2.2 ONNX エクスポートの変更 (export_onnx.py)

ONNX グラフに以下の Optional 入力を追加:

| テンソル名 | 型 | 形状 | 必須 | 説明 |
|-----------|-----|------|------|------|
| `speaker_embedding` | float32 | (1, gin_channels) | No | 外部 Speaker Encoder の出力 |
| `speaker_embedding_mask` | int64 | (1,) | No | 0: 無効 (sid fallback), 1: 有効 |

既存入力テンソル (`input`, `input_lengths`, `scales`, `sid`, `lid`) は変更なし。`speaker_embedding_mask` が 0 または未提供の場合は既存の `sid` -> `emb_g` lookup で動作する。

### 2.3 推論パス分岐のロジック

**本案: `speaker_embedding_mask` 方式 (推奨)**

ゼロベクトルが有効な embedding と区別できない問題を回避するため、明示的なマスクテンソルを追加する。

| テンソル名 | 型 | 形状 | 必須 | 説明 |
|-----------|-----|------|------|------|
| `speaker_embedding` | float32 | (1, gin_channels) | No | 外部 Speaker Encoder の出力 |
| `speaker_embedding_mask` | int64 | (1,) | No | 0: speaker_embedding 無効 (sid fallback), 1: speaker_embedding 有効 |

```
speaker_embedding_mask == 1?
  ├─ Yes → g = speaker_embedding + emb_lang(lid) (Voice Cloning モード)
  └─ No
       └─ sid が有効?
            ├─ Yes → g = emb_g(sid) + emb_lang(lid) (既存の multi-speaker モード)
            └─ No → g = emb_lang(lid) (single-speaker モード, emb_lang のみ)
```

`speaker_embedding_mask` は常に入力テンソルとして存在し、デフォルト値は 0 (無効)。旧ランタイムでは未指定 = 0 として扱われるため後方互換を維持する。

**代替案: ONNX opset 16+ の Optional 型**

ONNX opset 16 以降の `Optional` 入力型を使い、`speaker_embedding` の有無をグラフレベルで判定する方式。ONNX Runtime 1.14+ で対応。ただし WASM 版 ONNX Runtime のバージョン対応を事前確認する必要がある。

### 2.3.1 `_get_global_conditioning()` の合成方法

Voice Cloning モード時の global conditioning は以下のように合成する:

```python
def _get_global_conditioning(self, sid, lid, speaker_embedding, speaker_embedding_mask):
    if speaker_embedding_mask is not None and speaker_embedding_mask.item() == 1:
        g = speaker_embedding.unsqueeze(-1)  # (B, gin_channels, 1)
    elif sid is not None:
        g = self.emb_g(sid)  # (B, gin_channels, 1)
    else:
        g = None

    if lid is not None and hasattr(self, 'emb_lang'):
        lang_emb = self.emb_lang(lid)  # (B, lang_channels, 1)
        if g is not None:
            g = g + lang_emb
        else:
            g = lang_emb

    return g
```

つまり: `g = speaker_embedding + emb_lang(lid)` (Voice Cloning 時) または `g = emb_g(sid) + emb_lang(lid)` (既存パス)。emb_lang は言語条件付けを担い、speaker_embedding / emb_g とは独立に加算される。

### 2.4 次元マッチング

| ソース | 次元 | 一致 |
|--------|------|------|
| ECAPA-TDNN 出力 (M3-1) | 256 | gin_channels=256 と一致 |
| 既存 emb_g | gin_channels (256) | - |

M3-GATE で「条件付き Go」の場合は線形射影層を追加:
```python
self.speaker_proj = nn.Linear(encoder_dim, gin_channels)
```

### 2.5 後方互換性の保証

| シナリオ | 動作 | 互換性 |
|---------|------|--------|
| 既存 VITS ONNX (v1.x) + 新ランタイム | `speaker_embedding` 入力なし → `sid` 使用 | 完全互換 |
| 新 VITS ONNX (v2.0) + 旧ランタイム | `speaker_embedding` 無視 → `sid` 使用 | 完全互換 |
| 新 VITS ONNX + 新ランタイム + speaker_embedding | Voice Cloning モード | 新機能 |

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **ONNX Optional 入力の互換性**: `speaker_embedding_mask` (int64, 0 or 1) を追加入力テンソルとして採用し、ゼロベクトルと有効 embedding の区別問題を解決済み。旧ランタイムでは mask 未指定 = 0 として扱われるため後方互換を維持
2. **既存モデルの非互換リスク**: ONNX グラフに新しい入力が追加されると、入力名のハードコードに依存している旧ランタイムがクラッシュする可能性。ONNX の Optional 入力 (opset 16+) を使えばグラフレベルで互換性を担保できるが、WASM の ONNX Runtime バージョンを確認する必要がある
3. **学習パイプラインへの影響**: `lightning.py` の training step で `speaker_embedding` をどう扱うか。学習時は `sid` -> `emb_g` を使い、推論時のみ `speaker_embedding` を使う設計が最もシンプル

## 6. 一から作り直すとしたら

**VITS の speaker conditioning アーキテクチャの再設計:**

現在の VITS は `emb_g(sid)` で離散的な speaker embedding を lookup し、各モジュール (encoder, decoder, flow, dp) の条件付けに使用している。Voice Cloning で外部 embedding を注入する場合、この lookup を bypass するだけなので変更は最小限。

しかし、ゼロから設計するなら以下を検討すべき:

1. **Adaptive Layer Norm (AdaLN)**: speaker embedding を LayerNorm のスケール/バイアスとして注入 (StyleTTS 方式)。`emb_g` の単純な concat よりも表現力が高い
2. **Cross-Attention**: speaker embedding を key/value として各層で cross-attention (Tortoise-TTS 方式)。最も柔軟だが計算コスト大
3. **FiLM (Feature-wise Linear Modulation)**: 各層で `gamma * x + beta` の変調 (StyleSpeech 方式)。AdaLN より軽量

VITS の既存設計 (concat + 1x1 conv) は最もシンプルで、piper-plus の軽量性コンセプトに合致している。AdaLN への移行は M3-5 (VITS2) と合わせて検討すべきだったが、破壊的変更が大きすぎるため v3.0.0 以降のロードマップとする判断は妥当。

**VITS2 移行とのタイミングについて:** M3-2 (speaker_embedding) を VITS1 で実装してから M3-5 (VITS2) に移行するため、M3-5 で models.py を再度変更する必要がある。ゼロから設計するなら M3-5 を先に実施し、VITS2 の models.py に直接 speaker_embedding を追加すべき。ただし M3-5 は M3-GATE に依存しないため並行作業可能であり、現在の順序でも手戻りは models.py の差分マージ程度で済む。

## 7. 後続タスクへの連絡事項

- **M3-3**: 学習時に `speaker_embedding` を使う場合のデータパイプライン設計が必要。事前に全話者の embedding を Speaker Encoder で抽出してキャッシュする方式を推奨
- **M3-4**: 各ランタイムの ONNX 入力テンソル構築で `speaker_embedding` と `speaker_embedding_mask` を追加。テンソル名は `"speaker_embedding"` (float32, `(1, gin_channels)`) と `"speaker_embedding_mask"` (int64, `(1,)`)。mask=1 で Voice Cloning モード、mask=0 で sid fallback
