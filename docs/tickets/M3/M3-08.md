# [M3-8] MOS ベンチマーク実施 + 結果公開

**マイルストーン**: M3a (v2.0.0)
**カテゴリ**: 訴求力
**工数**: 3d
**依存**: M3-3, M3-5
**ステータス**: 未着手

---

## 1. 目的とゴール

piper-plus v2.0.0 の音声品質を MOS (Mean Opinion Score) で定量評価し、結果を公開する。競合 TTS エンジン (Kokoro, Coqui/XTTS, Edge TTS 等) との比較ベンチマークを含め、プロジェクトの訴求力を高める。

**ゴール:**
- 6言語 x 3モデル (6lang v1, 6lang v2/ZH増量, Voice Cloning) の MOS 評価
- 競合との比較ベンチマーク (少なくとも 3 エンジン)
- 結果を `docs/benchmark-mos.md` として公開
- HuggingFace Space でサンプル音声の試聴を提供

## 2. 実装内容の詳細

### 2.1 評価対象モデル

| モデル | 説明 | 条件 |
|--------|------|------|
| piper-plus 6lang v1 | 既存マルチリンガルベース (75 epoch) | ベースライン |
| piper-plus 6lang v2 | ZH 増量版 (M3-6) | ZH 品質改善の検証 |
| piper-plus VITS2 | adversarial DP (M3-5) | 韻律改善の検証 |
| piper-plus Voice Cloning | ZSE-VITS (M3-3) | Voice Cloning 品質 |
| piper-plus つくよみちゃん | ファインチューニング済み | 個別話者の品質 |

### 2.2 競合ベンチマーク対象

| エンジン | ライセンス | 言語 | 備考 |
|---------|----------|------|------|
| Kokoro v1.0 | Apache-2.0 | JA/EN/ZH/KO/FR/ES | 最も近い競合 |
| Edge TTS | プロプライエタリ | 全言語 | クラウド TTS の品質上限参考 |
| Piper (original) | MIT | EN 中心 | piper-plus の origin |
| gTTS | プロプライエタリ | 全言語 | Google TTS のベースライン |

### 2.3 評価テキストセット

各言語 10 文、合計 60 文。テキストのカテゴリ:

| カテゴリ | 文数/言語 | 目的 |
|---------|----------|------|
| 短文 (5-15文字) | 3 | 基本的な音質 |
| 長文 (30-50文字) | 3 | 自然なプロソディ |
| 疑問文 | 2 | イントネーション |
| 感嘆文 / 数字 | 2 | 特殊パターン |

### 2.4 MOS 評価プロトコル

| 項目 | 値 |
|------|-----|
| 評価者数 | 20人 (クラウドソーシング) |
| 評価尺度 | 5段階 (1: 非常に悪い ~ 5: 非常に良い) |
| 評価観点 | 音質 MOS + 自然性 MOS + 話者類似度 MOS (Voice Cloning のみ) |
| 音声フォーマット | WAV 22.05kHz mono |
| 提示順序 | ランダム化 (評価者間でも) |
| 評価プラットフォーム | Amazon Mechanical Turk or Prolific |

### 2.5 追加メトリクス (自動評価)

| メトリクス | ツール | 目的 |
|-----------|--------|------|
| UTMOS | SpeechMOS | 自動 MOS 推定 (人間評価との補完) |
| PESQ | pesq ライブラリ | 客観的音声品質 |
| WER | Whisper large-v3 | 合成音声の明瞭性 (STT での認識率) |
| RTF | 各ランタイム | リアルタイム係数 (推論速度) |
| Speaker similarity | Speaker Encoder | Voice Cloning の話者類似度 |

### 2.6 ベンチマーク実行スクリプト

```bash
# 全モデルの音声生成
uv run python tools/benchmark/generate_samples.py \
  --models-config tools/benchmark/models.yaml \
  --texts-dir tools/benchmark/texts/ \
  --output-dir /data/piper/benchmark-samples/

# 自動メトリクス計算
uv run python tools/benchmark/compute_metrics.py \
  --samples-dir /data/piper/benchmark-samples/ \
  --output tools/benchmark/results/auto_metrics.json

# MOS 評価用 HTML 生成
uv run python tools/benchmark/generate_mos_survey.py \
  --samples-dir /data/piper/benchmark-samples/ \
  --output tools/benchmark/mos_survey/
```

## 4. テスト計画

### 提供範囲（スコープ）
- IN: MOS 評価の設計・実施・結果公開、自動メトリクス計算、ドキュメント作成
- OUT: 品質改善の実施 (結果に基づく改善は別チケット)

### ユニットテスト

- `test_generate_samples_script`: 音声生成スクリプトがダミーモデルで正常動作
- `test_compute_metrics_script`: メトリクス計算スクリプトがダミーデータで正常動作
- `test_benchmark_texts_complete`: 全言語 x 全カテゴリのテキストが揃っていること

### E2Eテスト

- 実際のモデルで 1 言語 (JA) の全テキストを生成し、音声ファイルが正常であること
- UTMOS スコアが合理的な範囲 (2.0-5.0) にあること
- RTF が 1.0 未満 (リアルタイムより高速) であること (CPU)

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **MOS 評価のコスト**: クラウドソーシングでの MOS 評価は 1 セッション $200-500。20人 x 60文 x 5モデル = 6,000 評価。予算確認が必要
2. **評価者の言語能力**: 日本語/中国語の MOS 評価にはネイティブスピーカーが必要。Prolific のフィルタ機能でスクリーニング
3. **競合モデルの公平性**: Edge TTS はクラウド推論 (高品質 GPU)、piper-plus は CPU 推論。推論環境の違いがフェアネスに影響。RTF は CPU 環境で統一
4. **MOS のバイアス**: 評価者の主観に依存。UTMOS (自動評価) との相関を確認し、外れ値を除外

### レビューチェックリスト

- [ ] 評価テキストが各言語のネイティブチェックを受けている
- [ ] MOS 評価のプロトコルが再現可能 (評価者への指示書が明確)
- [ ] 統計的有意性の検定 (t-test / Wilcoxon) が含まれている
- [ ] docs/benchmark-mos.md のフォーマットが読みやすい
- [ ] サンプル音声が HuggingFace Space で試聴可能
- [ ] 競合モデルのバージョン・設定が明記されている
- [ ] 再現手順が `tools/benchmark/` に含まれている

## 6. 一から作り直すとしたら

**ベンチマーク設計の再考:**

MOS 評価は「品質の絶対値」を測るのに適しているが、piper-plus の差別化ポイント (軽量性、マルチランタイム、オフライン動作、Apache-2.0) は MOS では測れない。ゼロから設計するなら、以下の多軸評価を行うべき:

1. **品質軸 (MOS)**: 音質、自然性、話者類似度 -- 現案でカバー
2. **実用性軸 (RTF, メモリ, モデルサイズ)**: 各エンジンの推論コスト比較。RPi / ブラウザでの動作可否
3. **開発者体験軸 (DX)**: API の使いやすさ、ドキュメントの充実度、セットアップ時間。定量化が難しいが、GitHub Star 増加率やアンケートで近似
4. **エコシステム軸**: 対応言語数、ランタイム数、パッケージマネージャ対応。一覧表で比較

MOS だけでは Kokoro や Edge TTS に勝てない可能性が高い (クラウド推論は品質上限が高い)。しかし「同等の品質をオフライン・5ランタイム・Apache-2.0で提供」という文脈で見れば piper-plus の価値は明確。benchmark-mos.md にはこの文脈を含めるべき。

**自動 MOS (UTMOS) の活用:** 人間 MOS は高コスト・低頻度でしか実施できない。CI に UTMOS を組み込み、PR ごとに品質の回帰をチェックする仕組みを構築すべきだった。これは v2.1.0 以降の CI 改善として検討。

## 7. 後続タスクへの連絡事項

### No-Go 時のスコープ縮小版

M3-GATE が No-Go (選択肢B: Voice Cloning 延期) の場合、M3-08 のスコープを以下に縮小する:

| 評価対象 | 含む | 備考 |
|---------|------|------|
| piper-plus 6lang v1 (ベースライン) | IN | - |
| piper-plus VITS2 (M3-5) | IN | adversarial DP の効果検証 |
| piper-plus 6lang v2 ZH増量 (M3-6) | IN | ZH 品質改善の検証 |
| SSML (M3-7) | IN | prosody 制御のデモ (定性評価) |
| piper-plus Voice Cloning (M3-3) | **OUT** | Voice Cloning は v2.1.0 に延期 |
| 競合比較 (Kokoro, Edge TTS 等) | IN | 縮小版でも実施 |

Voice Cloning 関連の評価 (話者類似度 MOS, Speaker similarity) は除外し、音質 MOS + 自然性 MOS + RTF のみで評価する。

- MOS 結果が期待を下回る言語/モデルがある場合、v2.0.1 以降の品質改善チケットを作成
- HuggingFace Space の URL を README.md のベンチマークセクションに追加 (M1-8 の更新)
- benchmark-mos.md は GitHub Pages でも公開。`docs/` ディレクトリに配置するため自動的に Pages に含まれる
- 評価テキストセット (`tools/benchmark/texts/`) は将来のベンチマーク更新で再利用するためリポジトリに含める
- UTMOS の CI 統合は v2.1.0 のチケットとして別途作成を推奨
