## 1. 目的とゴール

Unity 開発者が `piper-plus` を UPM (Unity Package Manager) 経由でインストールし、C# スクリプトから TTS 音声を合成できるようにする。既存の C API 共有ライブラリ (libpiper_plus) を P/Invoke で呼び出し、生成された PCM データを Unity の `AudioClip` に変換する高レベル API を提供する。

**ゴール:**
- UPM パッケージ `com.piper-plus.tts` の作成
- `PiperTTS.Synthesize(text)` -> `AudioClip` のシンプルな API
- `PiperTTSAsync` でバックグラウンド合成 (メインスレッドをブロックしない)
- Windows (x86_64), macOS (arm64/x86_64), Linux (x86_64) のネイティブプラグイン同梱
- iOS (arm64, 静的リンク), Android (arm64-v8a) への対応準備

### 2.2 PiperPlusNative.cs (P/Invoke 宣言)

```csharp
internal static class PiperPlusNative
{
    private const string DllName =
#if UNITY_IOS && !UNITY_EDITOR
        "__Internal";
#else
        "piper_plus";
#endif

    [DllImport(DllName)] internal static extern IntPtr piper_plus_version();
    [DllImport(DllName)] internal static extern int piper_plus_api_version();
    [DllImport(DllName)] internal static extern PiperPlusStatus piper_plus_create(
        ref PiperPlusConfig config, out IntPtr engine);
    [DllImport(DllName)] internal static extern PiperPlusStatus piper_plus_synthesize(
        IntPtr engine,
        [MarshalAs(UnmanagedType.LPUTF8Str)] string text,
        ref PiperPlusSynthOptions opts,   /* NULL = defaults */
        out IntPtr out_samples,
        out int out_num_samples,
        out int out_sample_rate);
    [DllImport(DllName)] internal static extern void piper_plus_free(IntPtr engine);
    [DllImport(DllName)] internal static extern void piper_plus_free_audio(IntPtr samples);
    // ... 全 C API 関数をマッピング (piper_plus.h 参照)
}
```

### 2.3 PiperTTS.cs (高レベル API)

```csharp
public class PiperTTS : IDisposable
{
    public PiperTTS(string modelPath, string configPath = null);

    /// <summary>テキストから AudioClip を合成</summary>
    public AudioClip Synthesize(string text, int speakerId = 0, string language = null);

    /// <summary>テキストから raw PCM float[] を合成</summary>
    public float[] SynthesizeRaw(string text, int speakerId = 0, string language = null);

    /// <summary>サンプリングレート (通常 22050)</summary>
    public int SampleRate { get; }

    /// <summary>対応言語一覧</summary>
    public string[] Languages { get; }

    public void Dispose();
}
```

### 2.4 PiperTTSAsync.cs (非同期 API)

```csharp
public class PiperTTSAsync
{
    public PiperTTSAsync(string modelPath, string configPath = null);

    /// <summary>バックグラウンドで合成し、AudioClip を返す</summary>
    public async Task<AudioClip> SynthesizeAsync(string text, ...);

    /// <summary>文単位ストリーミング (コールバック)</summary>
    public void SynthesizeStreaming(string text, Action<AudioClip> onChunk, ...);
}
```

**重要**: `AudioClip.Create()` はメインスレッドでのみ呼び出し可能。`Task.Run` でPCM生成後、メインスレッドに戻して `AudioClip` 化する。

### 2.5 PCM -> AudioClip 変換

```csharp
public static class AudioClipExtensions
{
    public static AudioClip ToAudioClip(float[] pcmData, int sampleRate, string name = "piper_tts")
    {
        var clip = AudioClip.Create(name, pcmData.Length, 1, sampleRate, false);
        clip.SetData(pcmData, 0);
        return clip;
    }
}
```

### 2.8 事前調査: ネイティブバイナリ配布方式の決定 (実装前に実施)

libpiper_plus は ONNX Runtime 込みで 50-100MB になるため、UPM パッケージへの同梱方式を事前に調査・決定する:

| 方式 | メリット | デメリット |
|------|---------|----------|
| Git LFS で UPM リポジトリに同梱 | セットアップが簡単、UPM の git URL で完結 | リポジトリサイズ肥大化、LFS 帯域制限 |
| GitHub Releases + 初回 DL | リポジトリが軽量、帯域制限なし | Editor スクリプトで DL 自動化が必要、オフライン環境で不便 |
| NuGet ネイティブパッケージ | .NET エコシステムと親和性が高い | Unity での NuGet 統合が非標準 |

**推奨**: 初期段階では Git LFS でシンプルに進め、バイナリサイズやユーザーの帯域が問題になった時点で GitHub Releases 方式に移行。

## 5. 懸念事項とレビュー項目

### 既存 C# 実装 (`PiperPlus.Core`) との関係

`PiperPlus.Core` は純粋な C# (.NET) 実装で、OnnxRuntime C# バインディングを直接使用する。本 UPM パッケージ (`com.piper-plus.tts`) は C API 共有ライブラリ (`libpiper_plus`) を P/Invoke で呼び出す方式であり、両者は以下の点で差別化される:

| 項目 | PiperPlus.Core (.NET) | UPM パッケージ (P/Invoke) |
|------|----------------------|--------------------------|
| 推論方式 | OnnxRuntime C# Managed | libpiper_plus (C API) 経由 |
| Unity 互換性 | Mono では動作可能だが IL2CPP で制限あり | P/Invoke は IL2CPP と完全互換 |
| プラットフォーム | Desktop (.NET 8/9) | Desktop + iOS + Android |
| G2P | C# 再実装 (DotNetG2P) | C++ 実装 (libpiper_plus 内蔵) |
| 用途 | サーバーサイド、デスクトップアプリ | ゲーム開発 (Unity) |

**Unity で P/Invoke を選択する理由**: Unity の IL2CPP (Ahead-of-Time コンパイル) では、OnnxRuntime C# バインディングの一部が動作しない可能性がある。P/Invoke によるネイティブライブラリ呼び出しは IL2CPP と完全に互換性があり、iOS/Android を含む全プラットフォームで安定動作する。

### 懸念事項

1. **iOS 静的リンク (中リスク)**: iOS では dylib が使えないため `.a` 静的ライブラリが必要。`__Internal` DllImport + IL2CPP ビルドの検証が必要
2. **Android NDK ビルド (中リスク)**: arm64-v8a 向けの `.so` を NDK でクロスコンパイルする必要。CMake toolchain file の設定
3. **ONNX Runtime 依存**: libpiper_plus は onnxruntime に静的リンクしている前提。onnxruntime の Unity GPU プロバイダとの競合がないか確認
4. **モデルファイルの配置**: `.onnx` ファイルは数十 MB。StreamingAssets に配置するか、ランタイムDL するかの設計判断
5. **Unity バージョン互換**: 2021.3 LTS 以降をサポート対象とするが、IL2CPP/Mono の両方で動作確認が必要
6. **メインスレッド制約**: `AudioClip.Create()` はメインスレッドのみ。非同期 API ではスレッド切り替えのオーバーヘッドが発生

### レビューチェックリスト

- [ ] P/Invoke 宣言が `piper_plus.h` の全パブリック関数をカバーしているか
- [ ] P/Invoke の構造体レイアウトが C 構造体と一致しているか (`[StructLayout(LayoutKind.Sequential)]`)
- [ ] `IDisposable` パターンが正しく実装されているか (ファイナライザ含む)
- [ ] iOS の `__Internal` 分岐が正しいか
- [ ] `.meta` ファイルのプラットフォーム設定が正しいか

## 6. 一から作り直すとしたら

UPM パッケージ vs Unity Asset Store vs NuGet の選択が最初の分岐点。UPM は Unity 公式の配布チャネルだが、ネイティブプラグインの配布に制約がある (git URL でのインストール時にネイティブバイナリが大きい)。Asset Store は有料配布の道が開けるが審査プロセスが重い。NuGet (NuGetForUnity) は .NET エコシステムとの親和性が高いが、ネイティブプラグインの配置が非標準。

理想的には UPM + GitHub Releases でネイティブバイナリを別途配布し、UPM パッケージ側は C# ラッパーのみ含める二層構造。ネイティブプラグインは初回使用時に自動DL するか、ユーザーが手動で Plugins/ に配置する。ただし初期段階ではシンプルさを優先し、全プラットフォームのバイナリを同梱する方式で進める。

## 7. 後続タスクへの連絡事項

- **M3-16**: サンプルシーン (`Samples~/BasicTTS/`) の実装。本チケットで API が固まった後に着手
- **M3-17**: iOS/Android ビルド CI。本チケットで P/Invoke 宣言と `.meta` ファイルの基盤を作成するが、モバイルネイティブビルドは M3-17 で対応
- C API (`piper_plus.h`) に変更があった場合、`PiperPlusNative.cs` を同期更新する必要がある。API バージョン (`PIPER_PLUS_API_VERSION`) でバージョン不一致を検出する仕組みを検討
- ネイティブバイナリのサイズ: libpiper_plus は onnxruntime 込みで 50-100MB。UPM パッケージのサイズが git リポジトリに与える影響に注意 (Git LFS or 外部配布を検討)
