# [M3-6] ZH データ増量 (63K -> 150K+) + 再学習

**マイルストーン**: M3a (v2.0.0)
**カテゴリ**: 音質
**工数**: 5d
**依存**: なし
**ステータス**: 未着手

---

## 1. 目的とゴール

現在の中国語データ (AISHELL-3, 63,223発話, 142話者) を追加データソースで 150K+ 発話に増量し、ZH の音声品質 (特に duration の短さ問題) を改善する。6言語マルチリンガルモデルを再学習し、ZH の MOS を他言語と同等レベルに引き上げる。

**背景:**
- つくよみちゃん 6lang ファインチューニングで ZH の duration が 1.21s と他言語 (2-3s) の半分以下
- 原因: AISHELL-3 のデータ量が他言語 (ES 168K, FR 107K) と比べて少なく、DP の ZH パラメータが不十分
- 目標: ZH 発話数を 150K+ に増量し、言語間のデータバランスを改善

## 2. 実装内容の詳細

### 2.1 追加データソースの選定

| データセット | 発話数 | 話者数 | ライセンス | 品質 |
|------------|--------|--------|----------|------|
| AISHELL-3 (既存) | 63,223 | 142 | Apache-2.0 | 高 (スタジオ録音) |
| DiDiSpeech-2 | ~50,000 | 100+ | CC-BY-SA-4.0 | 高 |
| WenetSpeech (clean subset) | ~40,000 | 200+ | CC-BY-4.0 | 中-高 (選別必要) |
| 合計目標 | **~153,000** | **~440+** | - | - |

**代替候補 (ライセンス要確認):**
- Baker (中国語単一話者, ~12h): 単一話者のため見送り
- THCHS-30 (~10K): 古いデータセット、品質不足


### 2.3 前処理スクリプト (新規)

`src/python/piper_train/tools/prepare_zh_additional.py`:
- DiDiSpeech-2 のディレクトリ構造を LJSpeech 形式に変換
- WenetSpeech の clean subset を抽出・変換
- エネルギー VAD で無音除去
- 音素化 (ChinesePhonemizer) + phoneme_ids 生成
- 話者ごとの発話数フィルタ (>= 30)

### 2.4 データ品質管理

| チェック項目 | 基準 | ツール |
|------------|------|--------|
| SNR | > 15dB | waveform-based SNR 推定 |
| 発話長 | 1-30秒 | 自動フィルタ |
| 音素化成功率 | > 99% | ChinesePhonemizer のエラー率 |
| 話者あたり発話数 | >= 30 | 自動フィルタ |

### 2.5 再学習コマンド

```bash
export WANDB_API_KEY=$(grep WANDB_API_KEY /data/piper/.env | cut -d= -f2) && \
NCCL_DEBUG=WARN NCCL_P2P_DISABLE=1 NCCL_IB_DISABLE=1 \
nohup /data/piper/.venv/bin/python -m piper_train \
  --dataset-dir /data/piper/dataset-multilingual-6lang-v2 \
  --prosody-dim 16 \
  --accelerator gpu --devices 4 --precision 32-true \
  --max_epochs 75 --batch-size 20 --samples-per-speaker 2 \
  --checkpoint-epochs 5 --quality medium \
  --base_lr 2e-4 --disable_auto_lr_scaling \
  --ema-decay 0.9995 \
  --max-phoneme-ids 400 \
  --no-wavlm \
  --audio-log-epochs 5 \
  --default_root_dir /data/piper/output-multilingual-6lang-v2 \
  > /data/piper/training_multilingual_6lang_v2.log 2>&1 &
```

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **データソースのライセンス**: DiDiSpeech-2 は CC-BY-SA-4.0 (ShareAlike)。モデルの重みに SA 条件が伝播するかはグレーゾーン。法務確認が必要
2. **データ品質の不均一**: WenetSpeech は YouTube 由来のため、スタジオ録音の AISHELL-3 と品質差がある。SNR フィルタで品質担保するが、完全ではない
3. **他言語への悪影響**: ZH データ増量により、学習時のバッチ内 ZH 比率が上がる。`--language-balanced-sampling` で緩和されるが、他言語の品質劣化がないか注意
4. **GPU 時間**: ~100h (4x V100) はコスト大。M3-3 (Voice Cloning 学習) と合わせると ~200h。GPU リソースのスケジューリングが必要

## 6. 一から作り直すとしたら

**ZH データ不足問題の根本対処:**

そもそも 6lang v1 の設計段階で、言語ごとのデータ量バランスを考慮すべきだった。ES (168K) / FR (107K) に対して ZH (63K) は明らかに少ない。ゼロから設計するなら:

1. **言語ごとの目標発話数を事前設定**: 各言語 100K 前後を目標とし、データが足りない言語は事前にデータ収集
2. **データ増量よりもサンプリング戦略**: `--language-balanced-sampling` で各言語のバッチ比率を均等にする (既に実装済み)。ただし、ZH のデータ量自体が少ないと、同じデータを繰り返し使うことになりオーバーフィットのリスク
3. **合成データ (TTS augmentation)**: 既存の ZH TTS モデル (他プロジェクト) で合成データを生成し、学習データに追加。データ量問題の根本解決にはならないが、多様性の改善に有効

**DiDiSpeech-2 のライセンスリスクについて:** CC-BY-SA-4.0 の ShareAlike 条件がモデル重みに伝播するかは法的に不明確。リスク回避を優先するなら、DiDiSpeech-2 を除外し、CC-BY-4.0 の WenetSpeech clean subset のみで 100K を目指すべき。品質は落ちるが、ライセンスリスクがゼロになる。
