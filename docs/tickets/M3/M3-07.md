# [M3-7] SSML 基本サポート (break, prosody rate/pitch)

**マイルストーン**: M3a (v2.0.0)
**カテゴリ**: 機能
**工数**: 5d
**依存**: なし
**ステータス**: 未着手

---

## 1. 目的とゴール

SSML (Speech Synthesis Markup Language) の基本タグ (`<break>`, `<prosody>`) をサポートし、音声合成の韻律制御を可能にする。W3C SSML 1.1 仕様のサブセットを全8言語の phonemizer に統合する。

**ゴール:**
- `<break time="500ms"/>` で指定時間の無音を挿入
- `<prosody rate="slow" pitch="+2st">` で話速・ピッチを制御
- 全8言語 (JA/EN/ZH/KO/ES/FR/PT/SV) の phonemizer で SSML パーシングを共通化
- 全5ランタイム (Python/Rust/C#/Go/WASM) で SSML 入力を受け付け

## 2. 実装内容の詳細

### 2.1 サポートする SSML タグ

| タグ | 属性 | 値の例 | v2.0.0 スコープ |
|------|------|--------|---------------|
| `<speak>` | - | - | IN (ルート要素) |
| `<break>` | `time` | `250ms`, `1s` | IN |
| `<break>` | `strength` | `weak`, `medium`, `strong` | IN |
| `<prosody>` | `rate` | `slow`, `medium`, `fast`, `x-slow`, `x-fast`, `80%`, `120%` | IN |
| `<prosody>` | `pitch` | `low`, `high`, `+2st`, `-1st`, `+10%` | IN |
| `<prosody>` | `volume` | - | OUT (v2.1.0) |
| `<say-as>` | - | - | OUT (v2.1.0) |
| `<phoneme>` | - | - | OUT (既存 `[[ ]]` 記法でカバー) |
| `<sub>` | - | - | OUT (v2.1.0) |
| `<lang>` | - | - | OUT (既存 MultilingualPhonemizer でカバー) |

### 2.2 SSML パーサー (共通モジュール)

各言語の phonemizer に SSML パーシングを直接実装するのではなく、共通の SSML パーサーを先に適用し、結果を phonemizer に渡す方式を採用。

```python
# src/python/g2p/piper_plus_g2p/ssml.py (新規)
@dataclass
class SSMLSegment:
    text: str
    break_ms: int = 0           # <break> の無音時間 (ms)
    rate_scale: float = 1.0     # <prosody rate> のスケール (0.5-2.0)
    pitch_shift_st: float = 0.0 # <prosody pitch> のシフト (半音)

class SSMLParser:
    def parse(self, ssml: str) -> list[SSMLSegment]:
        """SSML テキスト -> セグメントリスト。
        非SSML テキストはそのまま1セグメントとして返す。"""

    def is_ssml(self, text: str) -> bool:
        """テキストが SSML かどうかを判定 (<speak> タグの有無)。"""
```

### 2.3 break の実装

`<break>` は phoneme_ids に特殊な無音トークンを挿入して実現する。

| strength | 無音時間 (ms) | 無音トークン数 |
|---------|-------------|-------------|
| `x-weak` | 100 | 1 |
| `weak` | 250 | 2 |
| `medium` | 500 | 4 |
| `strong` | 750 | 6 |
| `x-strong` | 1000 | 8 |
| `time="Nms"` | N | N / 125 (端数切り上げ) |

無音トークン: 既存の silence phoneme (`_`, id=0) を使用。1トークンあたり ~125ms の無音に相当 (DP の学習データに基づく近似値)。

### 2.4 prosody rate の実装

`<prosody rate>` は VITS の `length_scale` パラメータにマッピングする。

| rate | length_scale |
|------|-------------|
| `x-slow` | 1.5 |
| `slow` | 1.25 |
| `medium` | 1.0 |
| `fast` | 0.8 |
| `x-fast` | 0.6 |
| `N%` | 1.0 / (N / 100) |

セグメント単位で `length_scale` を変更するため、SSML テキストは複数の推論呼び出しに分割される。

### 2.5 prosody pitch の実装

`<prosody pitch>` は推論パラメータとして直接制御が難しい。VITS の `noise_scale` を調整することで近似的にピッチ変化を実現する。

| pitch | noise_scale 補正 |
|-------|-----------------|
| `x-low` | +0.3 |
| `low` | +0.15 |
| `medium` | 0 |
| `high` | -0.15 |
| `x-high` | -0.3 |
| `+Nst` | -N * 0.05 |

**注意:** VITS のピッチ制御は noise_scale の調整による近似であり、精密な半音制御ではない。v2.1.0 で pitch conditioning (F0 入力) を追加することで精密制御を実現予定。

### 2.6 各ランタイムでの実装

| ランタイム | SSML パーサー | 実装方針 |
|-----------|-------------|---------|
| Python (G2P) | `piper_plus_g2p/ssml.py` | xml.etree.ElementTree |
| Python (Runtime) | `piper/phonemize/ssml.py` | G2P の ssml.py を参照 |
| Rust | `piper-core/src/ssml.rs` | quick-xml クレート |
| C# | `PiperPlus.Core/Ssml/SsmlParser.cs` | System.Xml.Linq |
| Go | `piperplus/ssml/parser.go` | encoding/xml |
| WASM | JS 側で DOMParser | ブラウザネイティブ XML パーサー |

### 2.7 API 変更

各ランタイムの `synthesize()` メソッドが SSML 入力を自動判定して処理:

```python
# Python
piper.synthesize("<speak>こんにちは<break time='500ms'/>お元気ですか</speak>")

# Rust
model.synthesize_text("<speak>Hello<break time=\"500ms\"/>world</speak>", &params)?;
```

非 SSML テキスト (= `<speak>` タグなし) はこれまで通りプレーンテキストとして処理。

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **prosody pitch の近似精度**: noise_scale による pitch 制御は近似であり、セミトーン単位の精密制御は不可能。ドキュメントに制約を明記し、v2.1.0 の F0 conditioning で改善予定であることを記載
2. **break の無音トークン方式の限界**: 1トークン=125ms は近似値であり、DP の予測によって実際の無音時間がばらつく。長い break (> 2s) は波形レベルでの無音挿入に切り替えるべき
3. **セグメント分割による音声の不連続**: prosody rate が変わるセグメント境界で音声が不自然に途切れるリスク。クロスフェード処理の追加を検討
4. **SSML 仕様の範囲**: W3C SSML 1.1 は広範な仕様。v2.0.0 でサポートするサブセットを明確にし、未サポートタグの処理 (無視してテキストを抽出) を実装

### レビューチェックリスト

- [ ] SSML パーサーが W3C SSML 1.1 のサブセット仕様に準拠
- [ ] 不正な XML / 未サポートタグでクラッシュしない (graceful degradation)
- [ ] 全5ランタイムで同一 SSML -> 同一セグメント分割
- [ ] break の無音時間が +-50% 以内の精度
- [ ] prosody rate の制約がドキュメントに記載
- [ ] 既存の非 SSML テキスト入力に影響がない (回帰テスト)

## 6. 一から作り直すとしたら

**SSML アーキテクチャの再設計:**

現在のアプローチは「SSML パーサー -> セグメント分割 -> 各セグメントを個別に推論」だが、これにはセグメント境界での不連続性問題がある。ゼロから設計するなら:

1. **モデルレベルでの韻律制御**: prosody を phoneme_ids のメタデータとしてモデルに渡し、1回の推論で全テキストを処理。VITS の条件付けに rate/pitch をベクトルとして追加。最も自然だが、モデルの再学習が必要
2. **後処理方式**: 通常通り推論した音声を WSOLA (rate) / PSOLA (pitch) で後処理。モデル変更不要だが、品質劣化のリスク
3. **セグメント分割 + クロスフェード (現案の改良)**: セグメント境界で 50ms のクロスフェードを適用。実装コストは低いが、完全にはシームレスにならない

v2.0.0 では (3) を採用し、v2.1.0 以降で (1) に移行するのが現実的。ただし、(1) はモデルの学習データに prosody アノテーションが必要であり、データ準備のコストが大きい。

**pitch 制御について:** VITS の noise_scale による pitch 制御は本質的に限界がある。ゼロから設計するなら、F0 conditioning (Grad-TTS / NaturalSpeech 方式) を最初から導入し、pitch を明示的に制御可能にすべき。ただし F0 抽出の前処理 (CREPE / DIO) が全ランタイムで必要になるため、マルチランタイム対応のコストが大きい。
