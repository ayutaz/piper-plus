**ゴール:**
- 新モデル 4-5 件を voices.json に追加 (LJSpeech, あみたろ, HiFi-TTS, SIWIS, KO)
- 既存エントリの `md5_digest` を全て埋める
- 全実装の `--list-models` で新モデルが表示されることを確認
- `--model ljspeech` 等のエイリアスでモデルが解決されることを確認

### 2.2 md5_digest の計算

```bash
# 全モデルの md5_digest を計算
for onnx_file in /data/piper/output-*//*.onnx; do
    md5sum "$onnx_file"
done
```

既存エントリ (tsukuyomi-chan, css10-6lang) の `md5_digest` も同様に計算して更新。

### 2.3 スキーマ検証

voices.json の全エントリが以下のスキーマに準拠することを検証:

```
required fields:
  - key: string (unique)
  - name: string
  - language: { code, family, region, name_native, name_english, country_english }
  - quality: "medium" | "high" | "low"
  - num_speakers: integer >= 1
  - speaker_id_map: object
  - source: "piper-plus" | "community"
  - repo: string (HuggingFace repo)
  - files: { "<filename>": { size_bytes: integer, md5_digest: string } }
  - aliases: string[]
  - description: string
```

## 4. テスト計画

### 提供範囲（スコープ）
- IN: voices.json の更新、md5_digest の計算・入力、全実装での動作確認

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **M3-9〜13 の完了タイミング**: 全モデルが同時に完了するとは限らない。完了したモデルから順次 voices.json に追加する運用
2. **md5_digest の計算タイミング**: FP16 エクスポート後にファイルが変更されないことを確認してから md5 を計算
3. **voices.json の肥大化**: モデル数が増えると JSON ファイルが大きくなる。現時点では問題ないが、将来的にインデックス方式の検討が必要

### レビューチェックリスト

- [ ] 全エントリの `md5_digest` が空でないか
- [ ] 全エントリの `repo` が有効な HuggingFace リポジトリか
- [ ] `aliases` がグローバルでユニークか (既存エイリアスとの衝突がないか)
- [ ] `size_bytes` が実際のファイルサイズと一致するか
- [ ] JSON 構文が正しいか (trailing comma 等のエラーなし)
- [ ] 既存エントリ (tsukuyomi-chan, css10-6lang) が壊れていないか

## 6. 一から作り直すとしたら

voices.json を手動管理ではなく、HuggingFace のモデルカードから自動生成する仕組みを構築する。各モデルの HuggingFace リポジトリに標準化された `piper-plus-meta.json` を配置し、CI が定期的にクロールして voices.json を自動更新する。これにより md5_digest の計算漏れや size_bytes の不一致を防げる。ただし現段階ではモデル数が少ない (< 10) ため、手動管理で十分。
