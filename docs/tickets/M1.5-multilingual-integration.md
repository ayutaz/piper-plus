# M1.5 マルチリンガル統合 -- スウェーデン語の言語レジストリ統合

## マイルストーン紐づけ
- **マイルストーン**: M1.5
- **Phase**: Phase 1
- **対応要件**: FR-06
- **依存先チケット**: M1.4 (辞書ルックアップ統合 -- SwedishPhonemizer 完成)
- **ブロックするチケット**: M1.6 (テストスイート + CI)

## 達成したい内容

M1.4 で完成した `SwedishPhonemizer` を piper-plus の既存マルチリンガルフレームワークに統合する。4ファイルの変更により、`"sv"` を言語コードとして使用可能にし、既存6言語 (JA/EN/ZH/ES/FR/PT) との組み合わせ (例: `"ja-sv"`, `"en-sv"`, `"ja-en-zh-es-fr-pt-sv"`) をサポートする。

具体的には以下を実現する:

1. **registry.py**: `_auto_register()` に SwedishPhonemizer を登録し、`_detect_default_latin()` に `"sv"` を追加
2. **multilingual.py**: `UnicodeLanguageDetector` の `_latin_languages` に `"sv"` を追加
3. **multilingual_id_map.py**: `LANGUAGE_PHONEMES["sv"]` を登録し、`get_multilingual_id_map()` で SV 音素の ID 付与を有効化
4. **既存6言語への非影響保証**: JA/EN/ZH/ES/FR/PT の動作が一切変更されないことを回帰テストで保証

## 懸念点と影響範囲

| # | 懸念点 | 影響度 | 緩和策 |
|---|--------|--------|--------|
| 1 | `_latin_languages` 検出の制約 (C-05) | 中 | `en-sv` の組み合わせでは全ラテン文字が EN として処理され、SV テキストが英語として誤判定される。Phase 1 ではこの制約を受け入れる。`ja-sv` ではカナ/CJK が JA、ラテン文字が SV として正しく分離されるため、主要ユースケースは問題ない。Phase 2 で a/a/o ヒント検出の導入を検討する |
| 2 | 既存テストの回帰 | 高 | SV 追加により `registry.py`, `multilingual.py`, `multilingual_id_map.py` が変更されるため、既存6言語のテストが失敗するリスクがある。全テスト追加後に `python-tests.yml` CI で full regression を実行する。特に `test_multilingual_phonemizer.py` と `test_registry.py` (存在する場合) の PASS を必須とする |
| 3 | 音素ID衝突 (マルチリンガルモデル) | 高 | SV 固有音素16個に新規IDが付与される。既存6言語の `phoneme_id_map` (173シンボル) に SV を追加すると総シンボル数が ~189 に増加する。`get_multilingual_id_map()` の `seen` セットによる自動重複排除が正しく機能することを、テスト `test_en_sv_id_map_no_duplicates` と `test_7lang_id_map_includes_sv` で明示的に検証する |
| 4 | `_auto_register()` の ImportError 隔離 | 低 | SV 登録ブロックを `try/except ImportError` で囲むことで、`swedish.py` が存在しない環境 (Phase 2 の C++/Rust ランタイム等) でも他言語の登録に影響しない。既存の FR/PT/KO と同一パターンで実装する |
| 5 | config.json の language_id_map 順序 | 低 | 7言語モデルの `language_id_map` では `"sv": 6` が末尾に追加される。既存6言語の ID (ja=0, en=1, zh=2, es=3, fr=4, pt=5) は変更されない。新規学習時のみ SV が含まれるため、既存学習済みモデルへの影響はない |

## レビューする内容

1. **既存パターンとの一貫性**: `registry.py` の SV 登録ブロックが FR/PT 登録ブロックと同一構造 (`try/except ImportError` + ローカル import) であることを確認
2. **`_latin_languages` の正確性**: `multilingual.py` で SV がラテン文字言語として正しく分類されていること、`detect_char()` の `_RE_LATIN` パターンでスウェーデン語文字 (a/a/o) がマッチすることを確認
3. **音素重複排除の正確性**: `multilingual_id_map.py` で SV 音素が登録順序 (JA→EN→ZH→KO→ES→PT→FR→SV) に従って正しく重複排除されること。特に ZH 共有の ʂ/ɕ と FR 共有の ø/oe が同一IDを持つことを確認
4. **canonical sort の整合性**: `get_phonemizer("sv-en")` と `get_phonemizer("en-sv")` が同一インスタンスを返すことを確認。7言語コンボのソート順 (`"en-es-fr-ja-pt-sv-zh"`) が正しいことを確認
5. **回帰テストの網羅性**: JA/EN/ZH/ES/FR/PT の個別テストと6言語コンボテストが全て含まれていることを確認。`phonemize()` の出力が SV 追加前後で変化しないことを確認
6. **`_PUA_START` の整合性**: M1.2 で `0xE064` に更新された `_PUA_START` が、SV 追加後の動的割り当てと衝突しないことを確認
7. **config.json スキーマの後方互換性**: 既存6言語モデルの `config.json` が SV 追加後も有効であることを確認 (SV を含まない `language_id_map` でも動作すること)
