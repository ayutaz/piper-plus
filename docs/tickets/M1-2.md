## 1. 目的とゴール

`export_onnx.py` の `main()` 関数 L246-268 に、EMA 重み適用のインラインロジックが埋め込まれている。同時に `test_export_onnx.py` L142-154 では、この関数を呼び出す代わりにロジックを丸ごとコピー&ペーストして再実装している。

**問題点**:
1. **DRY 違反**: 本番コードとテストコードが同一ロジックを重複して持っている。本番側を修正してもテスト側が追従しない (またはその逆) リスクがある。
2. **テストが本番コードを呼んでいない**: テストは `export_onnx.py` の関数を一切 import せず、自前のループで EMA 適用を検証している。つまり本番コードのバグがテストに現れない。
3. **main() の肥大化**: `main()` に EMA 適用、emb_lang 統一、FP16 変換など複数の責務が混在しており、単体テストが困難。

**完了の定義**:
- `apply_ema_weights()` が `export_onnx.py` にトップレベル関数として抽出される
- `main()` が `apply_ema_weights()` を呼び出す形にリファクタリングされる
- `test_export_onnx.py` の `TestEMAWeightApplication` が `apply_ema_weights()` を直接 import して呼び出す形に書き換わる
- テストコード内の EMA ロジック再実装 (L142-154) が完全に除去される
- 既存の全テストが PASS する

**抽出後の関数**:

```python
def apply_ema_weights(
    decoder: torch.nn.Module,
    checkpoint_path: "str | Path",
) -> tuple[int, int]:
    """Apply EMA shadow weights to the decoder module.

    Loads the checkpoint, extracts ``ema_generator_state.shadow_params``,
    and copies matching parameters into *decoder* **in-place**.

    IMPORTANT: This must be called BEFORE ``remove_weight_norm()``, because
    EMA shadow params use ``weight_g``/``weight_v`` keys. ``remove_weight_norm()``
    fuses them into a single ``weight`` tensor, making EMA keys unmatchable.

    Parameters
    ----------
    decoder : torch.nn.Module
        The decoder sub-module (``model_g.dec``) whose parameters will be
        overwritten with EMA shadow values.
    checkpoint_path : str | Path
        Path to the ``.ckpt`` file containing ``ema_generator_state``.

    Returns
    -------
    (applied, skipped) : tuple[int, int]
        Number of parameters applied and skipped.
        ``(0, 0)`` if no EMA state was found in the checkpoint.
    """
    ckpt = torch.load(checkpoint_path, map_location="cpu")
    ema_state = ckpt.get("ema_generator_state")

    if not ema_state or "shadow_params" not in ema_state:
        _LOGGER.info("No EMA state found in checkpoint, skipping EMA")
        del ckpt
        return 0, 0

    applied = 0
    skipped = 0
    dec_params = dict(decoder.named_parameters())
    for name, shadow_param in ema_state["shadow_params"].items():
        if name in dec_params:
            dec_params[name].data.copy_(shadow_param)
            applied += 1
        else:
            skipped += 1

    if applied > 0:
        _LOGGER.info(
            "Applied EMA weights to decoder: %d parameters (skipped %d)",
            applied,
            skipped,
        )
    else:
        _LOGGER.warning("EMA state found but no matching decoder parameters")

    del ckpt
    return applied, skipped
```

### 2.2 `main()` の呼び出し側修正

L242-268 を以下に置換する:

```python
    # Apply EMA weights to decoder if available (always applied when present)
    # IMPORTANT: EMA must be applied BEFORE remove_weight_norm(), because EMA shadow
    # params use weight_g/weight_v keys. remove_weight_norm() fuses them into a single
    # "weight" tensor, making EMA keys unmatchable.

コメントは残すが、ロジックは1行の関数呼び出しに置き換わる。戻り値 `(applied, skipped)` は `main()` 側では使用しない (ログは関数内で出力される)。

from piper_train.export_onnx import apply_ema_weights

# --- TestEMAWeightApplication クラス内 ---

    def test_ema_weights_applied(self, tmp_path):
        """EMA state があればデコーダパラメータに適用される"""
        dec = torch.nn.Sequential(
            torch.nn.Linear(10, 10),
            torch.nn.Linear(10, 5),
        )

        # デコーダの元パラメータを記録
        original_params = {}
        for name, param in dec.named_parameters():
            original_params[name] = param.data.clone()

        # EMA shadow params を作成（元のパラメータ + 0.1）
        shadow_params = {}
        for name, param in dec.named_parameters():
            shadow_params[name] = param.data.clone() + 0.1

        ema_state = {"shadow_params": shadow_params}

        # モックチェックポイントを保存
        ckpt_path = tmp_path / "test_ema.ckpt"
        torch.save({"ema_generator_state": ema_state}, str(ckpt_path))

        # 抽出した関数を直接呼び出す
        applied, skipped = apply_ema_weights(dec, ckpt_path)

        assert applied > 0, "No EMA parameters were applied"
        assert skipped == 0, f"Unexpected skipped parameters: {skipped}"

        # パラメータが変更されたことを確認
        for name, param in dec.named_parameters():
            if name in original_params:
                assert not torch.equal(param.data, original_params[name]), (
                    f"Parameter {name} was not updated by EMA"
                )

    def test_no_ema_state_is_handled(self, tmp_path):
        """チェックポイントに EMA state がない場合はスキップされる"""
        dec = torch.nn.Sequential(torch.nn.Linear(10, 10))

        ckpt_path = tmp_path / "no_ema.ckpt"
        torch.save({"state_dict": {}}, str(ckpt_path))

        applied, skipped = apply_ema_weights(dec, ckpt_path)
        assert applied == 0
        assert skipped == 0
```

## 5. 懸念事項とレビュー項目

1. **`torch.load` の二重呼び出し**: 現在の `main()` では L211 で `VitsModel.load_from_checkpoint()` 、L246 で `torch.load()` と同一チェックポイントを 2 回ロードしている。`apply_ema_weights()` に抽出後もこの二重ロードは残る。将来的に `main()` の冒頭で 1 回だけロードし、ema_state を渡す形にリファクタリングすることも検討できるが、本チケットのスコープ外とする (M2 以降)。

2. **関数シグネチャの選択**: `apply_ema_weights(decoder, checkpoint_path)` vs `apply_ema_weights(decoder, ema_state)` の 2 案がある。
   - `checkpoint_path` 方式 (採用): ファイルI/Oを関数内に閉じ込め、呼び出し側が `torch.load` + `del ckpt` を意識しなくてよい。`main()` の呼び出しが 1 行で済む。
   - `ema_state` 方式: 純粋関数に近くテストが容易だが、`main()` 側に `torch.load` / `del ckpt` が残り、リファクタリング効果が薄い。
   - **結論**: `checkpoint_path` 方式を採用する。テスト側では `torch.save()` でモックチェックポイントを作成すれば十分テスト可能。

3. **`weights_only` パラメータ**: `torch.load()` は PyTorch 2.6+ で `weights_only=True` がデフォルトになる。現在のコードは L15-16 で `add_safe_globals([PosixPath, WindowsPath])` を設定してこれに対応済み。抽出した関数でも同じ `torch.load(path, map_location="cpu")` を使用するため、既存の safe globals 設定が適用される。

4. **ログレベルの一貫性**: `applied == 0` の場合は `_LOGGER.warning()`、EMA state なしの場合は `_LOGGER.info()` という現在の使い分けをそのまま維持する。

5. **型アノテーション**: `checkpoint_path` の型を `str | Path` として `from __future__ import annotations` なしで動作させるため、docstring 内の型記述と `"str | Path"` の文字列アノテーションを使用する。ファイル先頭に `from __future__ import annotations` を追加する方が望ましいが、既存コードとの一貫性を優先する。

## 6. 一から作り直すとしたら

もしこの EMA 適用機能を今から設計するなら:

1. **ema_state を直接受け取る純粋関数**: ファイルI/Oと重み適用を分離する。

   ```python
   def apply_ema_weights(
       decoder: torch.nn.Module,
       shadow_params: dict[str, torch.Tensor],
   ) -> tuple[int, int]:
       """Apply shadow params to decoder in-place."""
       ...

   def load_ema_state(checkpoint_path: Path) -> dict[str, torch.Tensor] | None:
       """Extract EMA shadow params from checkpoint."""
       ...
   ```

   テスト時に `torch.save` + ファイルI/Oが不要になり、`shadow_params` dict を直接渡せる。ただし `main()` 側の呼び出しが 2 行になる。

2. **`main()` のコマンドパターン化**: `main()` を「ロード → 前処理 (emb_lang 統一, EMA 適用, weight_norm 除去) → エクスポート → 後処理 (FP16, simplify)」のパイプラインとして明示的に構造化する。各ステップが独立した関数呼び出しになるため、テストやデバッグが容易。

3. **`ExportContext` データクラス**: チェックポイントパス、出力パス、各フラグをまとめた `@dataclass` を導入し、関数間で共有する。`args` 名前空間をそのまま渡す現在の設計よりも型安全。

## 7. 後続タスクへの連絡事項

- **公開 API の追加**: `apply_ema_weights()` は `export_onnx.py` のモジュールレベル関数として公開される。将来 `export_onnx` モジュールから関数を import するユースケース (スクリプトやノートブック) が増える場合、`__init__.py` での re-export を検討する。
- **二重ロード問題**: 本チケットでは `torch.load` の二重呼び出しを解消しない。M2 以降で `main()` 全体をリファクタリングする際に、チェックポイントを 1 回だけロードする設計に統合することを推奨する。
- **テスト import の前例**: 本チケットで `from piper_train.export_onnx import apply_ema_weights` のパターンを確立する。同モジュールの他のユーティリティ関数 (`should_unify_emb_lang`, `unify_emb_lang_weights`, `simplify_onnx_model`) も同様にテストから直接 import して検証するパターンを M2 以降で適用できる。
