# M3-6: CI 対応

> **マイルストーン**: M3
> **前提チケット**: M3-4, M3-5
> **後続チケット**: M4-1, M4-4
> **見積り**: 小
> **リスク**: 低

## タスク目的とゴール

M3-1 ~ M3-5 の変更 (G2P 移行 + deprecated コード削除) を受けて、CI ワークフローが引き続き正常に動作することを確認・更新する。パッケージサイズ制限の確認と `@piper-plus/g2p` 依存の解決が主な確認ポイント。

## 実装する内容の詳細

### 確認・更新対象

#### 1. `npm-publish.yml` の確認

- テストコマンド (`node --test`) の対象ファイルが変更されていないか確認
- M3-4 でテストファイル名は変更しない方針のため、変更不要の可能性が高い
- `npm pack` / `npm publish` のステップで削除ファイルが参照されていないか確認

#### 2. `g2p-wasm-ci.yml` の確認

- `@piper-plus/g2p` パッケージのテスト CI は M3 の変更に影響されないはず
- ただし、openjtalk-web のテストが `@piper-plus/g2p` を import するようになった場合、CI の依存インストールステップで `@piper-plus/g2p` が利用可能であることを確認

#### 3. `ci.yml` (メイン CI) の確認

- openjtalk-web のテスト実行ステップが正常に動作すること
- `npm install` / `npm ci` で `@piper-plus/g2p` (file:../g2p 依存) が正しく解決されること

#### 4. パッケージサイズチェック

- `npm pack` の出力サイズが制限内であることを確認:
  - 1MB 未満 (compressed)
  - 10MB 未満 (uncompressed)
- 6 ファイル (約 1200 行) の削除によりサイズが減少しているはず

#### 5. `package.json` の依存確認

- `@piper-plus/g2p` が `file:../g2p` として宣言されていること (既存)
- `peerDependencies` に影響がないこと

### 変更ファイル

| ファイル | 変更内容 |
|---------|---------|
| `.github/workflows/npm-publish.yml` | 必要に応じてテストコマンド更新 |
| `.github/workflows/ci.yml` | 必要に応じて依存インストール更新 |
| `src/wasm/openjtalk-web/package.json` | 必要に応じてスクリプト更新 |

**注意**: 変更が不要な場合は「変更なし、動作確認のみ」で完了とする。

## 提供範囲とテスト

### 提供範囲

- CI ワークフローの確認と必要に応じた更新
- パッケージサイズの検証結果レポート

### テスト項目

- 全 CI ワークフローが green であること
- `npm pack` が成功し、サイズ制限内であること
- `@piper-plus/g2p` 依存が CI 環境で正しく解決されること

### Unit テスト

- CI 環境でのテスト実行は M3-4 の scope。このチケットでは CI 設定の正当性を確認する

### E2E テスト

- GitHub Actions で全ワークフローを実行し、pass を確認
- PR 作成時の自動テストが pass すること

## 懸念事項とレビュー項目

### 懸念事項

1. **`file:../g2p` 依存の CI 解決**: `package.json` の `@piper-plus/g2p: "file:../g2p"` はローカルファイルシステム依存。CI のチェックアウト構成で相対パスが正しく解決されるか確認が必要
2. **npm publish 時の依存解決**: `file:` 依存は npm publish 時にバンドルされるか、npm レジストリへの公開前に書き換えが必要か確認

### レビュー項目

1. 全 CI ワークフロー (ci.yml, npm-publish.yml, g2p-wasm-ci.yml) の実行結果
2. パッケージサイズの before/after 比較
3. `file:../g2p` 依存が npm publish シナリオで問題ないこと
4. テストカバレッジレポートに異常がないこと

## 一から作り直すとしたら

CI パイプラインに「パッケージサイズ回帰テスト」をステップとして組み込む。サイズ閾値を CI で自動チェックし、意図しないサイズ増加を検知する仕組みがあればよかった。
