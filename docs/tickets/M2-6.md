## 1. 目的とゴール

`test_convert_fp16.py` L14-42 に定義されている `_onnx_inference()` は、ONNX 推論用ダミー入力の構築ロジックを独自に実装している。一方、本番コード `piper_train/tools/convert_fp16.py` L322-347 には同等の `_create_dummy_inputs()` が既に存在する。テスト側が本番関数を使わず独自実装していることは DRY 違反であり、本番コードの入力仕様が変更された際にテストとの乖離が発生するリスクがある。

**テスト側の現状コード** (`test_convert_fp16.py` L14-42):

```python
def _onnx_inference(onnx_path, phoneme_ids, prosody_features, noise_scale=0.667):
    """Run ONNX inference and return audio output."""
    import onnxruntime

    session = onnxruntime.InferenceSession(
        str(onnx_path), providers=["CPUExecutionProvider"]
    )
    text = np.expand_dims(np.array(phoneme_ids, dtype=np.int64), 0)
    text_lengths = np.array([text.shape[1]], dtype=np.int64)
    scales = np.array([noise_scale, 1.0, 0.8], dtype=np.float32)

    inputs = {
        "input": text,
        "input_lengths": text_lengths,
        "scales": scales,
    }

    input_names = [inp.name for inp in session.get_inputs()]
    if "prosody_features" in input_names:
        pf = []
        for feat in prosody_features:
            if feat is None:
                pf.append([0, 0, 0])
            else:
                pf.append([feat["a1"], feat["a2"], feat["a3"]])
        inputs["prosody_features"] = np.expand_dims(np.array(pf, dtype=np.int64), 0)

    outputs = session.run(None, inputs)
    return outputs[0].squeeze()
```

**本番側の対応関数** (`convert_fp16.py` L322-347):

```python
def _create_dummy_inputs(session) -> dict[str, np.ndarray] | None:
    """ONNXRTセッションからVITS用ダミー入力を生成する。"""
    inputs: dict[str, np.ndarray] = {}
    phoneme_length = 50

    for inp in session.get_inputs():
        name = inp.name
        if name == "input":
            inputs[name] = np.random.randint(
                0, 50, size=[1, phoneme_length], dtype=np.int64
            )
        elif name == "input_lengths":
            inputs[name] = np.array([phoneme_length], dtype=np.int64)
        elif name == "scales":
            inputs[name] = np.array([0.667, 1.0, 0.8], dtype=np.float32)
        elif name == "sid":
            inputs[name] = np.array([0], dtype=np.int64)
        elif name == "lid":
            inputs[name] = np.array([0], dtype=np.int64)
        elif name == "prosody_features":
            inputs[name] = np.zeros([1, phoneme_length, 3], dtype=np.int64)
        else:
            _LOGGER.warning("Unknown input: %s", name)
            return None

    return inputs
```

**問題点**:
1. **DRY 違反**: テスト側がダミー入力構築ロジックを独自実装しており、本番の `_create_dummy_inputs()` と重複
2. **カバレッジの空白**: 本番の `_create_dummy_inputs()` は `sid`, `lid` 対応など、テスト側より広い入力をカバーしている。テスト側がこれを使わないことで、multi-speaker/multilingual モデルへの対応がテストされない
3. **メンテナンスコスト**: 新しい入力 (例: 将来の conditional input) が追加された場合、本番とテストの両方を更新する必要がある

**完了の定義**:
- `test_convert_fp16.py` の `_onnx_inference()` が本番の `_create_dummy_inputs()` を import して使用する
- テストが `phoneme_ids` と `prosody_features` を引数として受け取る機能を維持する (特定のテストケースでは固定値ではなく任意の phoneme_ids を渡したい場合がある)
- 既存テストが全て PASS する

### 2.2 修正後のコード

```python
def _onnx_inference(onnx_path, phoneme_ids, prosody_features, noise_scale=0.667):
    """Run ONNX inference and return audio output.

    phoneme_ids/prosody_features を指定した場合はそれを使用し、
    それ以外の入力 (sid, lid 等) は本番 _create_dummy_inputs() に委譲する。
    """
    import onnxruntime

    from piper_train.tools.convert_fp16 import _create_dummy_inputs

    session = onnxruntime.InferenceSession(
        str(onnx_path), providers=["CPUExecutionProvider"]
    )

    # 本番関数でベースとなるダミー入力を生成
    inputs = _create_dummy_inputs(session)
    if inputs is None:
        raise RuntimeError("Failed to create dummy inputs for ONNX model")

    # テスト固有の phoneme_ids で上書き
    text = np.expand_dims(np.array(phoneme_ids, dtype=np.int64), 0)
    inputs["input"] = text
    inputs["input_lengths"] = np.array([text.shape[1]], dtype=np.int64)
    inputs["scales"] = np.array([noise_scale, 1.0, 0.8], dtype=np.float32)

    # prosody_features の上書き
    input_names = [inp.name for inp in session.get_inputs()]
    if "prosody_features" in input_names:
        pf = []
        for feat in prosody_features:
            if feat is None:
                pf.append([0, 0, 0])
            else:
                pf.append([feat["a1"], feat["a2"], feat["a3"]])
        inputs["prosody_features"] = np.expand_dims(
            np.array(pf, dtype=np.int64), 0
        )

    outputs = session.run(None, inputs)
    return outputs[0].squeeze()
```

### 2.3 設計判断

`_onnx_inference()` のシグネチャ (`phoneme_ids`, `prosody_features` 引数) は維持する。テストケースが特定の phoneme_ids で推論結果を比較する用途があるため、完全に `_create_dummy_inputs()` に置き換えることはできない。ベース入力を本番関数で生成し、テスト固有の値で上書きするハイブリッド方式を採用する。

## 5. 懸念事項とレビュー項目

1. **`_create_dummy_inputs` のアクセス性**: 本番側の `_create_dummy_inputs()` はアンダースコア付きのプライベート関数。テストからの import は慣習的に許容されるが、将来的に関数名が変更される可能性がある。公開 API に昇格させるか、テスト用のエイリアスを設けることを検討してもよい。

2. **phoneme_length の不一致**: 本番 `_create_dummy_inputs()` は固定長 50 でダミー入力を生成する。テスト側の `phoneme_ids` 引数で上書きする際に、`prosody_features` の長さが `phoneme_ids` と一致しない場合がある (`_create_dummy_inputs` は長さ 50 で生成、テストの `phoneme_ids` が別の長さの場合)。上書きロジックで整合性を保つ必要がある。

3. **`_create_dummy_inputs` が `None` を返すケース**: 未知の入力名がある場合に `None` を返す設計。テスト側では `RuntimeError` に変換しているが、テスト対象のモデルが標準的な VITS 入力のみを持つ場合は問題にならない。
