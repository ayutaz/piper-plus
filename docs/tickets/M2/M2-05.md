# [M2-5] CI path filter 導入 (dorny/paths-filter)

**マイルストーン**: M2a (v1.13.0)
**カテゴリ**: DX
**工数**: 2d
**依存**: なし
**ステータス**: 未着手

---

## 1. 目的とゴール

現在 38 個の CI ワークフローが存在し、PR ごとに不要なジョブが大量実行されている。`dorny/paths-filter` を導入し、変更されたファイルパスに応じて必要なジョブのみを実行することで CI 時間とコストを削減する。

**ゴール:**
- 変更パスに応じた条件付きジョブ実行
- Python のみの変更で Rust/C#/C++/Go テストが実行されない
- `.github/workflows/**` の変更は安全弁として全ジョブ実行
- 既存テストの信頼性を維持 (フィルタ漏れによるリグレッションを防止)

## 2. 実装内容の詳細

### 2.1 フィルタ定義

`ci.yml` (メインCI) に paths-filter ジョブを追加:

```yaml
jobs:
  changes:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: read
    outputs:
      python: ${{ steps.filter.outputs.python }}
      rust: ${{ steps.filter.outputs.rust }}
      csharp: ${{ steps.filter.outputs.csharp }}
      cpp: ${{ steps.filter.outputs.cpp }}
      wasm: ${{ steps.filter.outputs.wasm }}
      go: ${{ steps.filter.outputs.go }}
      ci-config: ${{ steps.filter.outputs.ci-config }}
    steps:
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            python:
              - 'src/python/**'
              - 'pyproject.toml'
              - 'uv.lock'
            rust:
              - 'src/rust/**'
              - 'Cargo.lock'
              - 'Cargo.toml'
            csharp:
              - 'src/csharp/**'
            cpp:
              - 'src/cpp/**'
              - 'CMakeLists.txt'
              - 'cmake/**'
            wasm:
              - 'src/wasm/**'
              - 'src/rust/piper-wasm/**'
            go:
              - 'src/go/**'
            ci-config:
              - '.github/workflows/**'
```

### 2.2 ジョブ条件の適用

各テストジョブに `needs` と `if` を追加:

```yaml
  python-tests:
    needs: changes
    if: ${{ needs.changes.outputs.python == 'true' || needs.changes.outputs.ci-config == 'true' }}
    ...

  rust-tests:
    needs: changes
    if: ${{ needs.changes.outputs.rust == 'true' || needs.changes.outputs.ci-config == 'true' }}
    ...
```

### 2.3 対象ワークフロー

| ワークフロー | フィルタ | 備考 |
|-------------|---------|------|
| `ci.yml` | 全フィルタ統合 | メイン CI |
| `cpp-tests.yml` | cpp, ci-config | C++ テスト |
| `csharp-ci.yml` | csharp, ci-config | C# テスト |
| `rust-tests.yml` | rust, ci-config | Rust テスト (個別WF) |
| `g2p-python-ci.yml` | python, ci-config | Python G2P テスト |
| `g2p-rust-ci.yml` | rust, ci-config | Rust G2P テスト |
| `g2p-wasm-ci.yml` | wasm, ci-config | WASM G2P テスト |
| `g2p-cross-platform-ci.yml` | python \|\| rust \|\| csharp \|\| go, ci-config | クロスプラットフォーム |
| `go-ci.yml` | go, ci-config | Go テスト (個別WF) |

### 2.4 push on dev ブランチへの対応

`push` イベント (dev ブランチ) では paths-filter が前コミットとの差分を使用する。マージコミットでは正しく動作するが、force-push 時はフォールバックとして全ジョブ実行。

### 2.5 非対象ワークフロー

以下は変更しない (手動トリガーやリリース専用のため):
- `deploy-*.yml` (手動 / タグトリガー)
- `release-*.yml` (タグトリガー)
- `npm-publish.yml` (タグトリガー)
- `docker-build.yml` (手動)

|---------|---------|---------|

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **フィルタ漏れ**: 共有ファイル (例: `pyproject.toml` が Rust にも影響) の見落とし
2. **dorny/paths-filter のバージョン固定**: v3 をピン。セキュリティアップデートへの追従
3. **merge queue との互換性**: GitHub merge queue 使用時に paths-filter が正しく動作するか
4. **個別ワークフロー vs 統合ワークフロー**: 38 個のワークフローすべてに paths-filter を入れるか、主要なものだけにするかの判断
5. **required checks**: GitHub branch protection の required checks と paths-filter スキップの整合性。スキップされたジョブは "skipped" ステータスになり、required check を通過しない問題がある

### レビューチェックリスト

- [ ] 全フィルタパスが実際のディレクトリ構造と一致しているか
- [ ] ci-config 安全弁が全ジョブに適用されているか
- [ ] `permissions: pull-requests: read` が設定されているか
- [ ] dev ブランチへの push イベントでもフィルタが動作するか
- [ ] required checks がスキップされたジョブで失敗しないか (paths-filter の `predicate-quantifier` 設定)
- [ ] 既存のテストが引き続きパスするか

## 6. 一から作り直すとしたら

38 個の個別ワークフローを統合し、1 つのメインワークフロー + reusable workflows 構成にする。paths-filter は 1 箇所で定義し、各 reusable workflow にフィルタ結果を渡す。現状の個別ワークフロー構成では paths-filter の適用箇所が分散し、メンテナンスコストが高い。
