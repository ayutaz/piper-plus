## 1. 目的とゴール

既存の PR テンプレートにコンポーネントチェックリストを追加し、PR 作成者が影響範囲を明示的に宣言できるようにする。レビュアーが変更の影響範囲を素早く把握でき、レビュー品質と速度が向上する。

**ゴール:**
- PR 作成時に影響コンポーネントをチェックリストで選択
- レビュアーが影響範囲を一目で把握
- 将来的に M2-5 の paths-filter と連動した自動ラベリングの基盤

## 2. 実装内容の詳細

### 2.1 変更対象ファイル

`.github/PULL_REQUEST_TEMPLATE.md`

### 2.2 追加内容

既存テンプレートの `## Checklist` セクションの前に以下を追加:

```markdown
## Affected components

<!-- Check all that apply. -->

- [ ] Python (src/python/)
- [ ] Rust (src/rust/)
- [ ] C# (src/csharp/)
- [ ] C++ (src/cpp/)
- [ ] Go (src/go/)
- [ ] WASM/npm (src/wasm/)
- [ ] Docker (docker/)
- [ ] CI/CD (.github/workflows/)
- [ ] Documentation only
```

### 2.3 完成形のテンプレート

> **既存テンプレートとの差分:** 既存の `.github/PULL_REQUEST_TEMPLATE.md` には既に「Test plan」セクションと「Breaking changes」セクションが存在する。本チケットでの変更は「Affected components」チェックリストセクションの追加のみであり、既存セクションは維持する。

```markdown
## Summary

<!-- Briefly describe what this PR does and why. -->

## Changes

<!-- List the main changes. -->

- 

## Affected components

<!-- Check all that apply. -->

- [ ] Python (src/python/)
- [ ] Rust (src/rust/)
- [ ] C# (src/csharp/)
- [ ] C++ (src/cpp/)
- [ ] Go (src/go/)
- [ ] WASM/npm (src/wasm/)
- [ ] Docker (docker/)
- [ ] CI/CD (.github/workflows/)
- [ ] Documentation only

## Checklist

- [ ] I have tested the changes locally
- [ ] I have added or updated tests as needed
- [ ] I have not added any GPL-licensed dependencies (e.g. espeak-ng)
- [ ] Breaking changes are documented below (if any)

## Test plan

<!-- How did you verify that this works? -->

## Breaking changes

<!-- If there are no breaking changes, write "None". -->

None
```

## 4. テスト計画

### 提供範囲（スコープ）
- IN: PR テンプレートの表示・動作
- OUT: 自動ラベリング (将来タスク)

### ユニットテスト

- テスト PR を作成し、テンプレートが正しく表示されることを確認
- チェックリストが GitHub UI で正しくレンダリングされることを確認
- 既存セクション (Summary, Changes, Checklist 等) が維持されていることを確認

### E2Eテスト

- 新規 PR 作成画面でテンプレートが自動適用されることを確認

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **チェックリストの粒度**: コンポーネントが増えた場合 (例: Android) の拡張性。現時点では 9 項目で十分
2. **記入の強制**: GitHub のテンプレートはソフトガイダンスであり、記入を強制できない。bot による自動チェックは将来検討

### レビューチェックリスト

- [ ] 既存テンプレートのセクションが欠損していないか
- [ ] Markdown のレンダリングが正しいか
- [ ] コンポーネント一覧がプロジェクトの実際のディレクトリ構造と一致しているか
- [ ] GPL チェック項目が維持されているか

## 6. 一から作り直すとしたら

GitHub の issue forms (YAML ベース) と同様に、PR テンプレートもフォームベースで必須入力を強制できる仕組みがあれば採用する。現時点では Markdown テンプレートが GitHub PR の標準。

## 7. 後続タスクへの連絡事項

- **M2-5**: paths-filter のフィルタ名とコンポーネント名を対応させておくことで、将来的に PR のチェック状態と CI ジョブの一致を検証できる。
- 新しいプラットフォーム (例: Android, Swift) を追加する際は、テンプレートにもコンポーネントを追加すること。
