## 2. 実装内容の詳細

### 2.1 テストファイル構成

```
src/python/piper_plus/tests/

```

### 2.2 テストフィクスチャ

```python
@pytest.fixture
def test_model_path():
    """テスト用 ONNX モデルパスを返す。モデルが存在しない場合はテストをスキップ。"""
    path = Path("test/models/multilingual-test-medium.onnx")
    if not path.exists():
        pytest.skip(f"Test model not found: {path} (git-lfs pull required)")
    return path

@pytest.fixture
def mock_session():
    """ORT セッションのモック"""
    ...

@pytest.fixture
def piper(test_model_path):
    """テストモデルで初期化済みの PiperPlus インスタンス"""
    return PiperPlus(str(test_model_path), download=False)
```

> **Note:** テストモデルは git-lfs で管理されている。CI ではモデルが存在しない場合があるため、`test_model_path` フィクスチャで `pytest.skip()` にフォールバックする。ユニットテスト (`@pytest.mark.unit`) はモックを使用するため影響なし。結合テスト (`@pytest.mark.integration`) のみがモデル依存となる。

### 2.3 テスト分類

| マーカー | 用途 | CI実行 |
|---------|------|--------|
| `@pytest.mark.unit` | ユニットテスト (モック使用) | 常時 |
| `@pytest.mark.integration` | テストモデル使用の結合テスト | 常時 |
| `@pytest.mark.slow` | HuggingFace DL を伴うテスト | 手動のみ |

### 2.5 CI 統合

- 新規 `python-api-ci.yml` ワークフローを M2-03 のスコープ内で作成する (M2-05 の path filter 適用は M2-05 で対応)
- `g2p-python-ci.yml` は G2P パッケージ専用として維持し、`piper-plus` API テストは分離する
- ワークフロー内容:
  ```yaml
  # .github/workflows/python-api-ci.yml
  on:
    push:
      branches: [dev]
    pull_request:
      branches: [dev]
  jobs:
    test:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4
        - uses: astral-sh/setup-uv@v4
        - run: uv run pytest src/python/piper_plus/tests/ -v -m "not slow"
  ```
- M2-05 で `python-api-ci.yml` に paths-filter を追加する

## 4. テスト計画

### 提供範囲（スコープ）
- IN: PiperPlus / AudioResult の全パブリック API、モデル解決、エラーケース、E2E 推論
- OUT: 音声品質の主観評価、GPU 推論テスト、PyPI パッケージングテスト (M2-4)

### ユニットテスト

**PiperPlus (test_piper_plus.py):**
- `test_init_with_path`: ファイルパス指定での初期化
- `test_init_with_alias`: エイリアス指定での初期化 (DL モック)
- `test_init_download_false`: download=False でローカルのみ検索
- `test_synthesize_returns_audio_result`: 戻り値の型チェック
- `test_synthesize_with_language`: language 指定の反映
- `test_synthesize_stream_yields`: ストリーミングのイテレータ検証
- `test_tts_to_file_creates_wav`: ファイル生成 + 内容検証
- `test_list_models_schema`: 戻り値のキー検証
- `test_invalid_model_raises`: 不正モデルで例外
- `test_empty_text_raises`: 空文字列で ValueError

**AudioResult (test_audio_result.py):**
- M2-2 で定義済みのテスト項目を実装

**モデル解決 (test_model_resolver.py):**
- `test_resolve_file_path`: 既存ファイルの解決
- `test_resolve_alias`: エイリアス -> HuggingFace URL
- `test_resolve_hf_repo`: リポジトリID 直接指定
- `test_cache_dir`: キャッシュディレクトリの正確性
- `test_download_failure`: ネットワークエラー時の例外

### E2Eテスト

**結合テスト (test_integration.py):**
- `test_full_pipeline_ja`: 日本語テキスト -> WAV ファイル生成 -> 音声長 > 0
- `test_full_pipeline_en`: 英語テキスト -> WAV ファイル生成
- `test_stream_pipeline`: ストリーミングで複数チャンクの受信
- `test_round_trip_save_load`: 保存 -> 読み込み -> バイナリ一致

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **テストモデルのサイズ**: `multilingual-test-medium.onnx` は git-lfs 管理。CI でのダウンロード時間に注意
2. **README のコード例の鮮度**: API 変更時に README が陳腐化するリスク。doctest 統合を検討
3. **ストリーミングテストの非決定性**: 文分割結果がモデルやテキストに依存

### レビューチェックリスト

- [ ] 全パブリックメソッドに対応するテストが存在するか
- [ ] README のコード例が実際に動作するか (doctest or 手動検証)
- [ ] テストモデルのパスがハードコードされていないか (conftest で一元管理)
- [ ] CI でテストが自動実行されるか
- [ ] エラーメッセージがユーザーフレンドリーか
