# [M2-2] `AudioResult` クラス実装 (save/to_bytes/play)

**マイルストーン**: M2a (v1.13.0)
**カテゴリ**: API設計
**工数**: 1d
**依存**: M2-1
**ステータス**: 完了 (M2-01に統合実装)

---

## 1. 目的とゴール

`PiperPlus.synthesize()` の戻り値となる `AudioResult` クラスを実装する。WAV エンコード、ファイル保存、バイト列取得、音声再生をワンライナーで行える使い勝手の良いデータクラスを提供する。npm パッケージの `AudioResult` と概念的に統一された API 設計とする。

**ゴール:**
- `result.save("output.wav")` でファイル保存
- `result.to_bytes()` で WAV バイト列取得
- `result.play()` で音声再生 (sounddevice オプション依存)
- `result.audio_data` で生の numpy 配列アクセス
- `result.sample_rate`, `result.duration` 等のメタデータアクセス

## 2. 実装内容の詳細

### 2.1 AudioResult クラス

```python
@dataclass
class AudioResult:
    audio_data: np.ndarray       # float32, shape=(samples,)
    sample_rate: int             # 22050 (モデル依存)
    
    @property
    def duration(self) -> float: ...           # 秒数
    
    def save(self, path: str | Path, format: str = "wav") -> Path: ...
    def to_bytes(self, format: str = "wav") -> bytes: ...
    def play(self, blocking: bool = True) -> None: ...
    def to_int16(self) -> np.ndarray: ...      # int16 変換
    
    def __len__(self) -> int: ...              # サンプル数
    def __add__(self, other: "AudioResult") -> "AudioResult": ...  # 結合
```

### 2.2 WAV エンコード

- 標準ライブラリ `wave` モジュールを使用 (外部依存なし)
- 出力フォーマット: 16bit PCM, モノラル
- `audio_data` (float32) -> int16 変換時にクリッピング処理
- `to_int16()` の実装は既存の `vits/utils.py` の `audio_float_to_int16()` を再利用する。同関数は `np.clip` + `int16` 変換を行っており、`AudioResult.to_int16()` はその薄いラッパーとする

### 2.2.1 `format` パラメータの扱い

v1 では WAV フォーマットのみをサポートする。`save()` および `to_bytes()` の `format` パラメータに `"wav"` 以外が指定された場合は `NotImplementedError("Only WAV format is supported in v1. MP3/OGG support is planned for a future release.")` を送出する。これにより将来の拡張時に API の後方互換性を維持できる。

### 2.3 音声再生

- `sounddevice` パッケージを optional dependency として使用
- 未インストール時は `ImportError` を `play()` 呼び出し時に発生させ、インストール手順を案内
- `blocking=True`: 再生完了まで待機、`blocking=False`: バックグラウンド再生

### 2.4 結合演算子

- `result1 + result2` で音声を時間方向に連結
- `sample_rate` が異なる場合は `ValueError`

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **sounddevice の可搬性**: macOS/Linux/Windows での動作差異。CoreAudio/PulseAudio/WASAPI バックエンド
2. **大規模音声のメモリ**: 長文テキストで生成される大きな音声データの扱い (ストリーミングとの整合性)
3. **フォーマット拡張**: 将来的に MP3/OGG 対応が求められる可能性。現時点では WAV のみで十分

### レビューチェックリスト

- [ ] `wave` モジュールのみで WAV 生成しており外部依存がないか
- [ ] `sounddevice` が optional dependency として正しく宣言されているか
- [ ] float32 -> int16 変換でオーバーフローが起きないか
- [ ] `__add__` で元の AudioResult が変更されないか (immutability)
- [ ] docstring に使用例が含まれているか
- [ ] numpy の型ヒントが正確か

## 6. 一から作り直すとしたら

`AudioResult` を immutable な frozen dataclass として設計する。`save()` や `play()` のような副作用メソッドをクラス本体から分離し、`piper_plus.audio.save(result, path)` のような関数型 API も併用する設計にする。ただし現状の使い勝手 (`result.save("out.wav")`) が直感的であり、v1 としてはメソッドベースで問題ない。
