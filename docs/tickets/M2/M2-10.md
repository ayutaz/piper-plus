# M2-10 Wyoming Protocol アダプタ実装

**マイルストーン**: M2b (v1.14.0)
**カテゴリ**: エコシステム
**工数**: 5d
**依存**: なし
**ステータス**: 完了

---

## 1. 目的とゴール

Home Assistant の音声パイプラインで piper-plus を TTS プロバイダーとして利用できるように、Wyoming Protocol のサーバーサイドアダプタを実装する。既存の `wyoming-piper` (eSpeak-ng 依存) の代替として、eSpeak-ng 不要の piper-plus エコシステムを HA ユーザーに提供する。

完了時の状態:
- `docker/wyoming-piper-plus/` にWyoming Protocolサーバーが実装されている
- `synthesize` イベントを受信し、`audio-start` / `audio-chunk` / `audio-stop` イベントで PCM 音声を返却できる
- PCM 16bit / 22050Hz mono フォーマットで音声を分割送信する
- 既存の `inference.py` の合成ロジックを共有モジュールとして活用する
- 6言語 (JA/EN/ZH/ES/FR/PT) の自動言語検出が動作する (SV/KO はG2Pコード実装済みだが学習済みモデル未含有のため実験的扱い)
- `--uri tcp://0.0.0.0:10200` でデフォルトポート待ち受けする

## 2. 実装内容の詳細

### ファイル構成

```
docker/wyoming-piper-plus/
  wyoming_piper_plus/
    __init__.py          # パッケージ初期化、バージョン定義
    __main__.py          # AsyncServer 起動エントリーポイント
    handler.py           # PiperPlusEventHandler (handle_event)
    synthesizer.py       # ONNX推論ラッパー (inference.py のロジック再利用)
  pyproject.toml         # 依存: wyoming>=1.5, piper-plus-g2p, onnxruntime
  README.md              # セットアップ手順
```

### 主要コンポーネント

#### `__main__.py`
- `argparse` で CLI 引数を定義: `--uri`, `--model`, `--config`, `--speaker-id`, `--language`, `--download-dir`
- `wyoming.server.AsyncServer` を起動し、`PiperPlusEventHandler` を登録
- モデル未指定時はデフォルトモデル (`tsukuyomi`) を自動ダウンロード

#### `handler.py`
- `wyoming.event.Event` の `type == "synthesize"` をハンドル
- `Synthesize` イベントからテキスト・言語・話者情報を抽出
- `synthesizer.py` で音声合成を実行
- PCM データを 4096 バイトチャンクに分割し、`AudioChunk` イベントとして送信
- エラー時は `Error` イベントを返却

#### `synthesizer.py`
- `piper_plus_g2p` の `MultilingualPhonemizer` で音素化
- ONNX Runtime で推論実行
- `docker/python-inference/inference.py` の合成パイプライン (phonemize -> encode -> infer -> decode) をモジュール化して共有
- セッション管理は `ort_utils.py` のパターンを踏襲

### Wyoming Protocol イベントフロー

```
Client (HA) -> Synthesize(text, voice, language)
Server -> AudioStart(rate=22050, width=2, channels=1)
Server -> AudioChunk(audio=<pcm_bytes>) x N
Server -> AudioStop()
```

### 依存パッケージ (pyproject.toml)

```toml
[project]
name = "wyoming-piper-plus"
version = "0.1.0"
dependencies = [
    "wyoming>=1.5",
    "piper-plus-g2p>=0.3.0",
    "onnxruntime>=1.17.0",
    "numpy>=1.24.0",
]
```

> **Note: `piper_train.ort_utils` への依存について**
> `inference.py` の合成ロジックは `piper_train.ort_utils` (`create_session_with_cache`, `warmup`) に依存している。Wyoming 側では以下のいずれかの方針を取る:
> 1. `pyproject.toml` の依存に `piper-tts-plus` を追加して `ort_utils` を直接 import する
> 2. `ort_utils.py` 相当のセッション管理・ウォームアップコードを `synthesizer.py` 内に自前実装する (推奨: 依存を最小化し、Wyoming パッケージを軽量に保つため)
>
> 実装時にどちらの方針を採用するか決定し、`pyproject.toml` を更新すること。

## 5. 懸念事項とレビュー項目

### 懸念事項
- `inference.py` との合成ロジック共有方法: 直接 import するか、共通モジュールに抽出するか。現時点では `synthesizer.py` に必要なロジックをコピーし、将来的にリファクタリングする方針
- Wyoming Protocol のバージョン互換性: `wyoming>=1.5` を要求するが、HA のバージョンによっては古い wyoming が入っている可能性がある
- モデルの初回ロード時間: ONNX モデルのロードに数秒かかるため、最初の synthesize リクエストがタイムアウトする可能性がある。サーバー起動時にウォームアップを実行する
- PCM チャンクサイズの最適化: 4096 バイトが最適かは HA 側の処理に依存。大きすぎると遅延が増え、小さすぎるとオーバーヘッドが増す

### レビューチェックリスト
- [ ] Wyoming Protocol v1.5 のイベント仕様に準拠している
- [ ] PCM フォーマット (16bit/22050Hz/mono) が HA の期待と一致する
- [ ] エラー時に適切な Error イベントが返却される
- [ ] モデル未指定時のデフォルト挙動が明確に定義されている
- [ ] asyncio のイベントループが正しく管理されている (リソースリーク防止)
- [ ] inference.py からのロジック共有に不整合がない

## 6. 一から作り直すとしたら

- `inference.py` の合成ロジックを最初から共通ライブラリとして設計すべきだった。現状は OpenAI 互換 API (`inference.py`) と Wyoming アダプタで同じロジックが二重に存在する。`docker/shared/synthesizer.py` のような共通モジュールを先に作成し、両方から import する設計が望ましい
- Wyoming Protocol はストリーミング向きなので、文単位の逐次合成 (C# の `TextSplitter.SplitSentences()` と同等) を最初から組み込むべき。長文テキストで最初の音声チャンクが返るまでの待ち時間を短縮できる
- Rust 推論エンジン (`piper-core`) を FFI で呼ぶ選択肢もあった。Python の ONNX Runtime よりも起動が速く、メモリ効率も良い
