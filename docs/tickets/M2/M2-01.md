## 1. 目的とゴール

ユーザーが `pip install piper-plus` 後に 3 行で音声合成できる高レベル Python API を提供する。既存の低レベルコンポーネント (`ort_utils.py`, `infer_onnx.py`, model manager) を内部で統合し、ONNX セッション管理・音素化・推論を薄いファサードとして公開する。

**ゴール:**
- `PiperPlus("tsukuyomi")` でモデル自動DL + セッション初期化
- `synthesize()` でテキストから `AudioResult` を返却
- `synthesize_stream()` で文単位ストリーミング
- `tts_to_file()` でワンライナーファイル出力
- `list_models()` で利用可能モデル一覧取得

## 2. 実装内容の詳細

### 2.1 ファイル構成

```
src/python/piper_plus/
  __init__.py          # PiperPlus, AudioResult を re-export
  api.py               # PiperPlus クラス本体
  audio.py             # AudioResult クラス (M2-2)
  _model_resolver.py   # モデル名解決 + HuggingFace DL
  py.typed             # PEP 561 マーカー
```

### 2.2 PiperPlus クラス

```python
class PiperPlus:
    def __init__(self, model: str = "tsukuyomi", *, config: str | None = None,
                 device: str = "auto", download: bool = True): ...
    def synthesize(self, text: str, *, speaker_id: int = 0,
                   language: str | None = None, noise_scale: float = 0.667,
                   length_scale: float = 1.0, noise_w: float = 0.8) -> AudioResult: ...
    def synthesize_stream(self, text: str, **kwargs) -> Iterator[AudioResult]: ...
    def tts_to_file(self, text: str, file_path: str = "output.wav", **kwargs) -> Path: ...
    @staticmethod
    def list_models(language: str | None = None) -> list[dict]: ...
```

### 2.3 内部統合

`piper-plus` PyPI パッケージは `piper-train` に依存するファサード方式を採用する。推論ロジック (`infer_onnx.py` の `text_to_phoneme_ids_and_prosody`, `synthesize` 等) を `piper_train` から import して利用し、`piper-plus` 自体に推論コードを重複させない。

**パッケージ依存ツリー:**

```
piper-plus  (PyPI: 高レベル API)
├── piper-train[inference]  (PyPI: 推論ロジック extras)
│   ├── onnxruntime / onnxruntime-gpu  (extras で分岐、後述)
│   ├── piper-plus-g2p  (PyPI: 多言語 G2P)
│   │   ├── pyopenjtalk (JA)
│   │   ├── g2p-en (EN)
│   │   └── pypinyin (ZH)  ...etc
│   └── numpy
├── huggingface-hub  (モデル自動DL)
└── sounddevice  (optional: [play] extras)
```

**G2P 二重依存の回避:** `piper-plus` -> `piper-train[inference]` -> `piper-plus-g2p` の推移的依存により G2P が供給される。`piper-plus` が直接 `piper-plus-g2p` に依存する必要はなく、pip の依存解決で単一バージョンに統一されるため G2P パッケージが重複インストールされることはない。

| 既存コンポーネント | 役割 | 統合方法 |
|-------------------|------|---------|
| `ort_utils.py` | ORT セッション作成・warmup・キャッシュ | `from piper_train.ort_utils import ...` で import。`__init__` でセッション生成、warmup 自動実行 |
| `infer_onnx.py` | phonemize + 推論パイプライン | `from piper_train.infer_onnx import text_to_phoneme_ids_and_prosody, synthesize` で import。`synthesize()` 内部で呼び出し |
| `piper_plus_g2p` | 多言語音素化 | `piper-train[inference]` の推移的依存として供給。`MultilingualPhonemizer` を内部インスタンス化 |
| model manager | HuggingFace 自動DL | `_model_resolver.py` でエイリアス解決 + DL |

### 2.4 モデル解決ロジック

1. `model` が既存ファイルパス -> そのまま使用
2. `model` がエイリアス ("tsukuyomi", "base" 等) -> HuggingFace リポジトリから DL
3. `model` が HuggingFace リポジトリID -> 直接 DL
4. DL先: `~/.cache/piper-plus/models/`

### 2.5 device パラメータ

| 値 | 動作 |
|----|------|
| `"auto"` | CUDA 利用可能なら GPU、なければ CPU |
| `"cpu"` | CPU 強制。ORT providers: `["CPUExecutionProvider"]` |
| `"cuda"` | GPU 0 を使用。ORT providers: `[("CUDAExecutionProvider", {"device_id": "0"})]` |
| `"cuda:N"` | GPU N を使用。`"cuda:0"` -> `{"device_id": "0"}`, `"cuda:1"` -> `{"device_id": "1"}` 等。ORT `provider_options` の `device_id` にマッピング |

### 2.6 エラーハンドリング

- `ModelNotFoundError`: モデルが見つからない / DL 失敗
- `PiperError`: 推論時の一般エラー
- `ValueError`: 不正なパラメータ

## 4. テスト計画

### 提供範囲（スコープ）
- IN: `PiperPlus` クラスの全パブリックメソッド、モデル解決ロジック、エラーハンドリング
- OUT: ORT 推論の内部動作 (既存テストでカバー)、AudioResult の詳細 (M2-2)、PyPI パッケージング (M2-4)

### ユニットテスト

- `test_piper_plus_init`: モデルパス / エイリアス / HuggingFace ID それぞれの初期化
- `test_synthesize_basic`: テキスト入力 -> AudioResult 返却の型・属性チェック
- `test_synthesize_params`: noise_scale, length_scale, noise_w の反映
- `test_synthesize_stream`: イテレータが AudioResult を yield すること
- `test_tts_to_file`: ファイル生成と WAV ヘッダー検証
- `test_list_models`: 戻り値のスキーマ検証
- `test_model_not_found`: 不正なモデル名で ModelNotFoundError
- `test_device_selection`: auto/cpu/cuda の ORT provider 選択

### E2Eテスト

- テストモデル (`test/models/multilingual-test-medium.onnx`) を使用した実推論テスト
- 日本語 / 英語テキストで音声生成 -> WAV バイト長 > 0 の検証

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **依存関係の肥大化**: `onnxruntime` は重い依存。`piper-plus[cpu]` / `piper-plus[gpu]` の extras で分離を検討
2. **モデルDLのネットワーク依存**: CI で HuggingFace DL をモックする必要あり
3. **既存 `infer_onnx.py` のリファクタ範囲**: 推論ロジックを共有モジュールに切り出す必要があるが、`infer_onnx.py` の CLI 互換性を壊さないこと
4. **スレッドセーフティ**: 同一 `PiperPlus` インスタンスの並行呼び出しに対する ORT セッションの安全性

### レビューチェックリスト

- [ ] パブリック API の型ヒントが完全か
- [ ] docstring が Google style で記述されているか
- [ ] `__init__` でのネットワークアクセスが `download=False` で回避できるか
- [ ] ORT セッションオプションが `ort-session-contract.toml` に準拠しているか
- [ ] GPL 依存が混入していないか
- [ ] `py.typed` マーカーが含まれているか

## 6. 一から作り直すとしたら

推論ロジックを `infer_onnx.py` から完全に分離した共有モジュール (`_inference.py`) として設計し、CLI (`infer_onnx.py`) と高レベル API (`api.py`) の両方がそれを利用する構造にする。現状は `infer_onnx.py` が CLI エントリポイントとロジックを兼ねているため、統合時に薄い重複が生じうる。

## 7. 後続タスクへの連絡事項

- **M2-2**: `AudioResult` クラスのインターフェースは `synthesize()` の戻り値型として本チケットで定義する。M2-2 はその実装を担当。
- **M2-3**: テストは本チケットで基本構造を作成するが、網羅的テスト + ドキュメントは M2-3 で対応。
- **M2-4**: `pyproject.toml` のパッケージメタデータ (name, version, dependencies) は本チケットで定義する。M2-4 で PyPI 公開。
- モデルエイリアス一覧は Rust CLI / C# CLI と統一するため、将来的に共通 JSON 定義を検討。
- **M3-1 (Speaker Encoder) への橋渡し**: M3-1 は `PiperPlus` クラスの推論パイプライン (`synthesize()` 内部の phonemize -> inference フロー) を Speaker Encoder 統合の基盤として利用する。`PiperPlus` の内部構造を M3-1 が拡張しやすい設計にしておくこと (例: 推論パイプラインのフック可能性、speaker embedding の注入ポイント)。
