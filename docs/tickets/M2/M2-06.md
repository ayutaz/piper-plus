# [M2-6] concurrency group 追加 (全PRトリガーWF)

**マイルストーン**: M2a (v1.13.0)
**カテゴリ**: DX
**工数**: 0.5d
**依存**: M2-5
**ステータス**: 未着手

---

## 1. 目的とゴール

同一 PR に対して複数回 push した際に、古い CI ランが自動キャンセルされず並行実行される問題を解消する。全 PR トリガーワークフローに `concurrency` グループを追加し、最新の push のみが CI を実行するようにする。

**ゴール:**
- 同一 PR の古い CI ランが自動キャンセルされる
- dev ブランチへの push は並行実行を許可 (キャンセルしない)
- CI コストとキュー待ち時間の削減

## 2. 実装内容の詳細

### 2.1 concurrency 設定

全 PR トリガーワークフローに以下を追加:

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.head_ref || github.ref }}
  cancel-in-progress: true
```

- `github.head_ref`: PR の場合、ソースブランチ名 (同一 PR で一意)
- `github.ref`: push の場合、ブランチ ref (dev への push はブランチ名で一意)
- `cancel-in-progress: true`: 同一グループの古いランをキャンセル

### 2.2 対象ワークフロー

PR トリガー (`pull_request`) を持つ全ワークフロー:

| ワークフロー | 現在のトリガー | concurrency 追加 |
|-------------|--------------|----------------|
| `ci.yml` | push + PR | 追加 |
| `cpp-tests.yml` | push + PR | 追加 |
| `csharp-ci.yml` | push + PR | 追加 |
| `rust-tests.yml` | push + PR | 追加 |
| `g2p-python-ci.yml` | push + PR | 追加 |
| `g2p-rust-ci.yml` | push + PR | 追加 |
| `g2p-wasm-ci.yml` | push + PR | 追加 |
| `g2p-cross-platform-ci.yml` | push + PR | 追加 |
| `go-ci.yml` | push + PR | 追加 |
| `cpp-lint.yml` | push + PR | 追加 |
| `docker-test.yml` | push + PR | 追加 |
| `webui-test.yml` | push + PR | 追加 |

### 2.3 非対象ワークフロー

- `workflow_dispatch` のみ: concurrency 不要
- タグトリガーのみ (`release-*.yml`, `npm-publish.yml`): concurrency 不要
- `schedule` のみ: concurrency 不要

### 2.4 dev ブランチの扱い

dev ブランチへの push では `github.head_ref` が空になるため `github.ref` (= `refs/heads/dev`) がグループキーとなる。これにより dev への連続 push でも最新のみが実行される。ただし、dev ブランチのマージは通常 PR 経由のため、直接 push で古いランがキャンセルされるのは問題ない。

## 4. テスト計画

### 提供範囲（スコープ）
- IN: PR トリガーの全ワークフローの concurrency 動作
- OUT: タグトリガー / 手動トリガーのワークフロー

### ユニットテスト

以下の手順で検証:

1. テスト PR を作成し push
2. CI が開始されたことを確認
3. 同一 PR に追加 push
4. 前のランが "Cancelled" ステータスになることを確認
5. 最新のランのみが実行完了することを確認

### E2Eテスト

- 複数の PR が同時に存在する場合、それぞれの concurrency group が独立していることを確認
- dev ブランチへのマージ後、push トリガーのランが正常に実行されることを確認

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **required checks とのキャンセル競合**: キャンセルされたランが required check の失敗としてカウントされないか確認が必要。GitHub はキャンセルされたランを "neutral" 扱いにするため通常は問題ないが、branch protection の設定による
2. **matrix ジョブのキャンセル**: matrix strategy の一部ジョブのみキャンセルされるケースがないか
3. **reusable workflow との互換性**: `_build-test-cpp.yml` 等の reusable workflow は呼び出し元で concurrency を設定すれば問題ない

### レビューチェックリスト

- [ ] 全 PR トリガーワークフローに concurrency が追加されているか
- [ ] `cancel-in-progress: true` が設定されているか
- [ ] グループキーが `${{ github.workflow }}-${{ github.head_ref || github.ref }}` の形式か
- [ ] タグトリガー / 手動トリガーのワークフローが変更されていないか
- [ ] YAML 構文エラーがないか (`actionlint` で検証)

## 6. 一から作り直すとしたら

全ワークフローに個別に concurrency を追加するのではなく、reusable workflow のテンプレートに concurrency を組み込み、全ワークフローがそのテンプレートを利用する構成にする。ただし GitHub Actions の制約上、concurrency はワークフローレベルでの設定が最もシンプルで確実。

## 7. 後続タスクへの連絡事項

- M2-5 の paths-filter と組み合わせることで、「変更に関係ないジョブはスキップ」+「同一 PR の古いランはキャンセル」の二重最適化が実現する。
- 新規ワークフロー作成時は必ず concurrency グループを追加すること。テンプレートとして本チケットの設定をコピーする。
