## 1. 目的とゴール

新規コントリビューターが最初の PR を出しやすくするため、`good first issue` ラベルを整備し、明確なスコープと手順を持つ初期 Issue を 5 件作成する。piper-plus のエコシステム認知度向上とコントリビューション促進を目的とする。

**ゴール:**
- `good first issue` ラベルが GitHub リポジトリに存在する
- 各言語/プラットフォームから 1 件以上の Issue が存在
- 各 Issue に実装手順と期待結果が明記されている
- 新規コントリビューターが CONTRIBUTING.md を参照して自力で取り組める

## 4. テスト計画

### 提供範囲（スコープ）
- IN: ラベルの作成、Issue の起票、Issue 内容の正確性
- OUT: Issue の実装自体 (コントリビューターが行う)

### ユニットテスト

- 各 Issue に記載されたファイルパスが実際に存在するか確認
- 各 Issue に記載されたテストコマンドが動作するか確認
- Issue テンプレートの Markdown レンダリングが正しいか確認

### E2Eテスト

- GitHub 上で Issue が正しく表示されるか確認
- ラベルが正しく適用されているか確認
- `good first issue` フィルタで 5 件が表示されるか確認

## 5. 懸念事項とレビュー項目

### 懸念事項

1. **Issue の難易度**: "good first issue" が実際に初心者にとって取り組みやすいか。過度に簡単だとやりがいがなく、難しすぎると離脱する
2. **Issue の鮮度**: 時間が経つと Issue の内容が陳腐化する (ファイルパスの変更等)。定期的な棚卸しが必要
3. **重複作業の防止**: 複数人が同時に同じ Issue に取り組むリスク。Issue のアサイン運用ルールを明記

### レビューチェックリスト

- [ ] 全 5 件の Issue に `good first issue` ラベルが付いているか
- [ ] 各 Issue に修正対象ファイルが明記されているか
- [ ] 各 Issue にテストコマンドが記載されているか
- [ ] CONTRIBUTING.md へのリンクが含まれているか
- [ ] Issue の難易度が適切か (初回 PR で完了可能な範囲)
- [ ] 複数プラットフォームに分散しているか

## 6. 一から作り直すとしたら

GitHub Issue テンプレート (`.github/ISSUE_TEMPLATE/good-first-issue.yml`) を YAML フォームとして定義し、`good first issue` 作成時に必須フィールド (修正対象ファイル、テストコマンド、難易度) を入力させる。これにより将来の good first issue 作成時も一定品質が担保される。

## 7. 後続タスクへの連絡事項

- Issue が消化された後は、新たな good first issue を補充する運用を確立する。
- コントリビューターが PR を出した際は、迅速なレビュー (48時間以内) を心がける。レビュー遅延はコントリビューター離脱の主因。
- 各 Issue のスコープは意図的に小さく保っている。大きなタスクは通常の Issue として起票する。
