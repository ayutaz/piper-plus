# M2-11 Wyoming Dockerfile + docker-compose

**マイルストーン**: M2b (v1.14.0)
**カテゴリ**: エコシステム
**工数**: 1d
**依存**: M2-10
**ステータス**: 未着手

---

## 1. 目的とゴール

M2-10 で実装した Wyoming Protocol アダプタを Docker イメージとしてパッケージングし、`docker-compose up` 一発で起動できる環境を提供する。HA ユーザーがモデルダウンロードから Wyoming サーバー起動までをシームレスに実行できるようにする。

完了時の状態:
- `docker/wyoming-piper-plus/Dockerfile` が存在し、マルチステージビルドでイメージサイズが最適化されている
- `docker/wyoming-piper-plus/docker-compose.yml` が存在し、`docker-compose up` で Wyoming サーバーが起動する
- モデルファイルがボリュームマウントで永続化される
- ヘルスチェックが設定されている
- amd64 / arm64 のマルチアーキテクチャに対応している

## 2. 実装内容の詳細

### Dockerfile

```dockerfile
# Stage 1: ビルド
FROM python:3.11-slim AS builder
WORKDIR /build
COPY pyproject.toml .
RUN pip install --no-cache-dir --prefix=/install .

# Stage 2: ランタイム
FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /install /usr/local
COPY wyoming_piper_plus/ ./wyoming_piper_plus/

# モデルダウンロードディレクトリ
RUN mkdir -p /data/models
VOLUME /data/models

EXPOSE 10200
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD python -c "import socket; s=socket.create_connection(('localhost',10200)); s.close()"

ENTRYPOINT ["python", "-m", "wyoming_piper_plus"]
CMD ["--uri", "tcp://0.0.0.0:10200", "--download-dir", "/data/models"]
```

### docker-compose.yml

```yaml
services:
  wyoming-piper-plus:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "10200:10200"
    volumes:
      - piper-models:/data/models
    environment:
      - PIPER_DEFAULT_MODEL=${PIPER_DEFAULT_MODEL:-tsukuyomi}
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "python", "-c", "import socket; s=socket.create_connection(('localhost',10200)); s.close()"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 120s

volumes:
  piper-models:
```

### 設計方針
- ベースイメージは `python:3.11-slim` を使用 (eSpeak-ng 不要のため Debian ライブラリが不要)
- マルチステージビルドで pip install のキャッシュを最終イメージに含めない
- モデルファイルは named volume (`piper-models`) で永続化。コンテナ再起動でもモデル再ダウンロード不要
- `PIPER_DEFAULT_MODEL` 環境変数でデフォルトモデルを設定可能
- ヘルスチェックは TCP ソケット接続で Wyoming サーバーの起動を確認

### マルチアーキテクチャ対応
- `onnxruntime` は amd64/arm64 の wheel を提供しているため、特別な対応なしでマルチアーキテクチャビルドが可能
- `docker buildx build --platform linux/amd64,linux/arm64` でクロスビルド

## 4. テスト計画

### 提供範囲 (スコープ)
- IN: Dockerfile、docker-compose.yml、ヘルスチェック、ボリュームマウント
- OUT: HA 側の設定 (M2-12)、CI/CD パイプライン構築、Docker Hub への公開

### ユニットテスト
- なし (インフラ設定ファイルのため)

### E2Eテスト
- `docker-compose up -d` で正常に起動することを確認
- ヘルスチェックが healthy になることを確認
- Wyoming クライアントから TCP 接続して synthesize イベントが処理されることを確認
- コンテナ停止・再起動後もモデルファイルが永続化されていることを確認
- `docker-compose down -v` でボリュームを含めたクリーンアップが動作することを確認

## 5. 懸念事項とレビュー項目

### 懸念事項
- `onnxruntime` の arm64 wheel サイズが大きい (~200MB)。イメージ全体で 500MB 以下に収まるか確認が必要
- Raspberry Pi 4 (arm64) での動作速度。ONNX Runtime の CPU 推論が実用的な速度で動作するか要検証
- 初回起動時のモデルダウンロード中はヘルスチェックが失敗する。HA がタイムアウトで Wyoming サーバーを無効化しないように、`start_period` の設定が必要かもしれない

### レビューチェックリスト
- [ ] マルチステージビルドで不要なビルドキャッシュが最終イメージに含まれていない
- [ ] root ユーザーで実行していない (可能であれば非root ユーザーを使用)
- [ ] ボリュームのパーミッションが適切に設定されている
- [ ] 環境変数のデフォルト値が適切
- [ ] ヘルスチェックの `start_period` が初回モデルダウンロード時間を考慮している
- [ ] `.dockerignore` で不要ファイルがコンテキストから除外されている

## 6. 一から作り直すとしたら

- Alpine ベースイメージを検討すべきだった。ただし `onnxruntime` が glibc に依存しているため、musl 環境では動作しない可能性がある。結局 `python:3.11-slim` が最適解
- Rust 推論エンジンをスタティックリンクしたバイナリをベースにすれば、Python ランタイム不要でイメージサイズを大幅に削減できた (50MB 以下)。ただし M2 スコープ外
- Docker Hub への自動パブリッシュ CI を同時に整備すべき。HA ユーザーは `docker pull` で使いたい
