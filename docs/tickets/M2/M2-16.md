## 2. 実装内容の詳細

### 記事構成

```
# eSpeak-ngを捨てて8言語G2Pを自前実装した話

## モチベーション
- eSpeak-ng の問題 (GPL, 外部バイナリ依存, WASM 移植困難)
- 代替手段の調査結果 (phonemizer, espeak-phonemizer, gruut)

## 設計方針
- Phonemizer ABC (抽象基底クラス) による言語プラグイン設計
- 言語レジストリパターン
- 4プラットフォーム統一 (Python/Rust/C#/Go)

## 言語別の実装戦略
### 日本語 (pyopenjtalk / jpreprocess)
  - OpenJTalk 辞書、PUA マッピング、疑問詞マーカー
### 英語 (g2p-en / cmudict)
  - CMU Pronouncing Dictionary ベース、ストレスマーカー
### 中国語 (pypinyin)
  - ピンイン変換、声調番号のIPA変換
### 韓国語 (g2pk2)
  - ハングル分解、連音化・口蓋音化ルール
### ロマンス語3言語 (ES/FR/PT)
  - 規則ベース (外部依存なし) の設計判断
### スウェーデン語 (SV)
  - コミュニティ貢献 (PR #294) による実装

## クロスプラットフォーム一貫性の保証
- 音素 ID マップの統一 (id_maps.py / token_map)
- クロスプラットフォーム CI (g2p-cross-platform-ci.yml)
- 約4000テストケースの言語横断テスト

## WASM への移植
- Rust G2P を wasm-bindgen でブラウザ対応
- 日本語辞書 (naist-jdic) のバンドル戦略
- peerDependency としての onnxruntime-web

## パフォーマンス
- LRU キャッシュによる日本語音素化高速化
- RTF 比較

## まとめと今後
- 残課題 (SV/KO のモデル学習)
- コントリビューション募集
```

## 5. 懸念事項とレビュー項目

### 懸念事項
- 「eSpeak-ngを捨てた」という表現が挑発的に受け取られる可能性がある。技術的な判断理由 (GPL、WASM 移植困難、外部バイナリ依存) を丁寧に説明し、eSpeak-ng 自体の価値は認める姿勢を示す
- G2P の音素化精度について、eSpeak-ng との比較を求められる可能性がある。定量的な比較データがない場合は「規則ベースのため特定のケースで差異がある」と正直に記載する
- 4プラットフォームの G2P 一貫性についての技術的な深掘りが求められた場合に備え、クロスプラットフォーム CI のテスト数やカバレッジを記載する
