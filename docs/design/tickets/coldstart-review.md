## 1. マイルストーン戦略レビュー

### 【高】M2 の成功基準「初回発話 ≤ 200ms」が過楽観的

- **問題:** warmup はセッション作成後の ORT 最適化キャッシュのみが対象。ONNX 推論自体（50–80ms）＋テンソル構築（20ms）で既に 100ms 超。M3・M4 との組み合わせなしに 200ms は達成困難。
- **修正案:** M2 の成功基準を「初回発話 ≤ 400ms」に引き下げ、「200ms 達成は M3 + M4 の組み合わせ後」と明記。

### 【高】M1 の削減量見積もり（300–500ms）に根拠がない

- **問題:** `ORT_DISABLE_ALL` → `ORT_ENABLE_ALL` で 200–400ms 削減と確定的に記述されているが、Rust 側（`engine.rs`）はデフォルトで最適化有効の可能性があり、C# のみが対象。
- **修正案:** M1 実施前にベースライン実測値を取得し、C#/Rust 別に根拠を明確化。見積もりを「200–300ms（過大見積り回避）」に修正。

### 【中】「M1–M3 完了後 ~300ms」の前提条件が非明示

- **問題:** モデルサイズ・テキスト長・ハードウェア・言語の条件が一切記載されていない。
- **修正案:** 目標達成シナリオに前提条件を追加（例: 「日本語モデル / テキスト「こんにちは」/ Intel i7-12700K / CPU 推論」）。

### 【中】M3 の jpreprocess スレッドセーフ性確認が前提条件に入っていない

- **問題:** `jpreprocess::JPreprocess` が `!Send` だった場合、`std::thread::spawn()` がコンパイル失敗する。M3 着手前の確認が必須だが、マイルストーン記述では「注意点」扱い。
- **修正案:** M3 の前提条件に「jpreprocess の `Send + Sync` 実装確認済み」を明示。

### 【中】M4 の JSON フォールバック設計の記述が混乱を招く

- **問題:** `include_bytes!()` ビルド時埋め込みと「実行時 JSON フォールバック」が混在した記述になっており、実運用ではフォールバックが発生しない。
- **修正案:** 「ビルド時埋め込み（デフォルト）」と「外部ファイル参照（オプション）」を明確に分離して記述。

### 【中】M5 のデバイス別モデル管理設計が未完成

- **問題:** DirectML/CUDA 最適化済みモデルは CPU では動作しないと警告があるが、C#/Rust で命名規則が統一されていない。C# 側に `device_type_label()` 相当の実装が記載されていない。
- **修正案:** M5 着手前に「デバイス別モデル選択ロジックの詳細設計」を作成（M5a として別出し）。

### 【低】M6 の期待削減量の内訳が矛盾

- **問題:** 「~100–200ms 短縮」と記載されているが、GPU 検出キャッシュのヒット時効果は 5ms 程度とチケット内に記載されており、合計が 100ms に届かない。
- **修正案:** 「~50–100ms（ReadyToRun JIT 短縮が主）」に修正。

## 2. M1–M3 技術レビュー

### 【高】ort v2.0.0-rc.12 で `with_intra_threads()` が存在するか未確認（M1・M2共通）

- **問題:** `SessionBuilder::with_intra_threads()` / `with_inter_threads()` が `ort 2.0.0-rc.12` に存在するか未検証。実装開始前に `cargo doc` で確認必須。
- **影響チケット:** COLD-M1（対策B）

### 【高】VITS モデルの DirectML 動的次元名が未確認（M1）

- **問題:** M1 対策C で `AddFreeDimensionOverrideByName("batch_size", 1)` を使用するが、実際のモデルの動的次元名が `"batch_size"` である保証がない。Netron でモデルを確認するまで実装不可。
- **影響チケット:** COLD-M1（対策C）

### 【高】M2 の Rust warmup で所有権モデルが未決定

- **問題:** `std::thread::spawn()` でバックグラウンド warmup を実行する場合、現状の `PiperVoice` 構造は `Clone` 不可能な可能性がある。`Arc<Mutex<>>` ラップの選択肢はチケットに記載されているが、実装方針が未確定。
- **影響チケット:** COLD-M2（Rust 実装）

### 【高】M2 のダミー入力の有効性が未検証

- **問題:** warmup に使用するダミー phoneme ID（`[1, 8, 8, 8, 2]` など）が実際のモデルで受け付けられるか未確認。空配列はエラーになることは確認済みだが、有効な ID セットの定義が必要。
- **影響チケット:** COLD-M2（全プラットフォーム）

### 【高】M3 の `jpreprocess` スレッドセーフ性が未確認（最重要）

- **問題:** `jpreprocess::JPreprocess` が `!Send` の場合、M3 のコアロジック（`std::thread::spawn` での並列初期化）がコンパイル不可。着手前に確認必須。
- **確認方法:** `cargo check` で `JPreprocess: Send + Sync` を確認するか、jpreprocess のドキュメント/ソースを参照。
- **影響チケット:** COLD-M3（Rust 実装）

### 【高】M3 の `VoiceConfig::Clone` 実装が未確認

- **問題:** M3 の Rust 並列化では `config.clone()` が必要だが、`VoiceConfig` に `#[derive(Clone)]` が実装されているか未確認。
- **影響チケット:** COLD-M3（Rust 実装）

### 【中】M3：M2 の warmup task との競合検証が不足

- **問題:** M3 で phonemizer と engine を並列初期化する際、M2 の warmup がバックグラウンドで実行中の場合の競合が考慮されていない。
- **修正案:** M3 のテスト項目に「warmup 実行中に synthesize() が呼ばれた場合の整合性確認」を追加。

## 3. M4–M6 技術レビュー

### 【高】M5 で `with_optimized_model_filepath()` の ort v2.0.0-rc.12 対応が未確認

- **問題:** M5 の核心となる API `with_optimized_model_filepath()` が `ort 2.0.0-rc.12` で提供されているか未確認。RC バージョンでは API が不安定な可能性がある。
- **修正案:** `ort` crate のドキュメント・Changelog を確認してから M5 を着手。存在しない場合は代替手段（手動で ORT Python ツールで事前変換）を検討。

### 【高】M5 の C#/Rust 間でデバイスラベル命名規則が不一致

- **問題:** Rust 側では `gpu.rs` の `DeviceType` enum を元に `device_type_label()` を実装する案があるが、C# 側で同一命名規則を適用する実装が記載されていない。両者で `dml` vs `directml` のように名前がずれると、一方で生成した最適化済みモデルが他方で認識されない。
- **修正案:** M5 着手前にデバイスラベルの仕様を一文書にまとめ、C#/Rust 両方に適用する。

### 【中】M4 の `build.rs` 実装に Windows ファイル時刻精度問題

- **問題:** `bm.modified() > jm.modified()` による差分ビルド判定は、Windows の FAT32/NTFS で秒単位以下の時刻精度が失われるため誤動作する可能性がある。
- **修正案:** `cargo:rerun-if-changed` のみに依存し、ファイル時刻比較は廃止。

### 【中】M5 の ORT バージョン比較に文字列比較を使用

- **問題:** バージョンを文字列で比較する場合、`2.0.0` と `2.0.0-rc.12` が別物と判定され、不必要な再最適化が頻発する。
- **修正案:** セマンティクバージョン比較ライブラリ（Rust: `semver` crate、C#: `NuGet.Versioning`）の使用を検討。

### 【中】M6 の CI に `stat -c%s` が Windows で動作しない

- **問題:** バイナリサイズチェックに `stat -c%s` を使用しているが、Windows CI（PowerShell）では動作しない。
- **修正案:** `$(Get-Item path).length`（PowerShell）または `wc -c` へのクロスプラットフォーム対応。

### 【中】M6 の GPU 検出キャッシュでドライババージョン取得実装が未記載

- **問題:** `device_cache.json` の `driver_version` フィールドをどう取得するか（Windows: `nvidia-smi.exe`、Linux: nvml ライブラリ）が `gpu.rs` 実装案に記載されていない。
- **修正案:** ドライババージョン取得の実装方針（外部コマンド実行 vs ライブラリ）を M6 チケットに追記。

### 【低】M4 の WASM 適用スコープが曖昧

- **問題:** bincode の WASM 適用について「計測結果次第」と記述されているが、計測の責務がどのチケットにあるか不明確。
- **修正案:** M4 の「提供範囲」に「WASM 向けはサイズ実測後に M4b チケットで別途対応」と明記。

## 4. チケット横断的レビュー

### 【高】CI による性能計測の自動化が計画されていない

- **問題:** 全チケットで「計測値を PR に記載」と記述されているが手動実行に頼っており、将来的なパフォーマンス回帰を自動検知できない。
- **修正案:** `Criterion.rs`（Rust）/ `BenchmarkDotNet`（C#）で計測テストを CI 統合する共通タスクを **M1 着手前** に実施。

### 【中】音声品質回帰テストの基準が全チケット間でバラバラ

- **問題:** M1「RMSE ≤ 0.01」、M4「RMS 誤差 1% 未満」など基準が統一されていない。
- **修正案:** 全チケット共通の基準として「参照音声 vs 出力音声の RMSE ≤ 0.005」を定義し、定数としてコードに共有。

### 【中】並列実施可能なマイルストーンが直列前提になっている

- 以下のペアは互いに非依存であり、並列実施によってスケジュールを短縮できる：
  - **M1 + M4**: ORT 設定修正と辞書ロードは完全に独立
  - **M2 + M3**: warmup（フェーズ2の消化）と並列化（フェーズ1の短縮）は独立
  - **M5 + M6**: ORT キャッシュと .NET R2R は独立
- **修正案:** マイルストーン一覧に「並列実施可能」フラグを追加。

### 【低】M2 の「tokio 非同期ファクトリ」提案は実現不可能

- **問題:** M2「一から作り直すとしたら」セクションで `tokio::join!` による非同期初期化を提案しているが、`piper-core` は WASM 対応のため `tokio` 非依存が前提。この提案は将来の設計と矛盾する。
- **修正案:** 提案を「WASM 非対応の `piper-cli` 専用ではあり得る選択肢」と限定して記述を修正。
