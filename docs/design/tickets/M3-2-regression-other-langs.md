# M3-2: 他言語回帰テスト

**マイルストーン**: M3
**依存チケット**: M3-1
**後続チケット**: なし (最終チケット)

## タスク目的とゴール

日本語 WASM G2P 統合 (M1) によって、既存の他言語 (EN, ZH, ES, FR, PT) の音声合成が壊れていないことを確認する。JS G2P パス (`@piper-plus/g2p`) が M1 の変更後も正常に動作し、音声品質や合成速度に回帰がないことを保証する。

#### 検証項目 (各テストセンテンスに対して)

1. **音声生成**: 「音声合成」ボタンクリック後、音声が再生されること
2. **音声品質**: 以前の動作と同等の品質であること (主観評価)
3. **合成速度**: 初期化済み状態でボタンクリックから音声再生までの体感時間が以前と大きく変わらないこと
4. **コンソールエラー**: DevTools コンソールにエラーが出力されないこと
5. **言語検出**: 言語を自動検出に設定した場合、正しい言語が選択されること

## 懸念事項

- **JS G2P パスへの間接的影響**: M1-1 で `G2P.create()` に渡す言語リストから `'ja'` を除外する変更がある。この変更により JS G2P の初期化パラメータが変わるため、他言語に予期しない影響が出る可能性がある (例: 言語リストの順序が変わる)
- **言語検出の変更**: M1-3 で `_wasmPhonemizer.detectLanguage()` を優先使用する変更がある。Rust WASM と JS G2P の言語検出結果が異なる場合、非 JA テキストの検出精度が変わる可能性がある
- **主観的品質評価の限界**: 「以前と同等の品質」の判断は主観的であり、微妙な劣化を見逃す可能性がある
- **M1 統合前の比較データ**: パフォーマンス比較には M1 統合前のデータが必要。M3-2 実施前に事前計測しておくこと

## 一から作り直すとしたら

音声回帰テストの自動化を検討すべきである。手動テストでは音声品質の微妙な劣化を検知できない。

改善案:
- **ゴールデンオーディオ比較**: 各言語のテストセンテンスに対して「ゴールデン」音声ファイルを保存し、合成結果との類似度 (PESQ, MCD 等) を自動計算する。閾値を下回った場合に CI を fail させる
- **波形レベルのスモークテスト**: 音声の長さ、RMS エネルギー、サンプルレートの妥当性を自動チェック。波形一致は不要だが、無音出力や極端な長さ変動を検知可能
- **Playwright + Web Audio API**: CI で Playwright を使い、合成結果の `AudioBuffer` を取得して自動検証する。`audioContext.decodeAudioData()` で波形を取得し、長さと RMS を検証
- **コスト vs 効果**: ゴールデンオーディオ方式は初期セットアップに ~2日、メンテナンスコストは中程度 (モデル更新時にゴールデン再生成)。スモークテスト方式は ~0.5日でセットアップ可能だが検知力は低い

## 後続タスクへの連絡事項

- M3-2 は最終チケットであり、テスト完了をもって日本語 WASM G2P 統合 (M1-M3) のクローズ条件とする
- テスト結果レポートは `docs/design/` 配下または Issue に記録し、将来の回帰テスト時の比較基準として使う
- パフォーマンス計測データは、将来的な WASM バイナリサイズ最適化やキャッシュ戦略の改善検討材料として保存すること
