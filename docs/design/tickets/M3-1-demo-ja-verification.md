# M3-1: デモページ日本語動作確認

**マイルストーン**: M3
**依存チケット**: M1-1, M1-2, M1-3, M2-1, M2-2, M2-3
**後続チケット**: M3-2

## タスク目的とゴール

GitHub Pages のデモページ (https://ayutaz.github.io/piper-plus/) で、日本語テキストから音声合成が正常に動作することを確認する。M1 で実装した Rust WASM ローダーが実環境で期待通りに動作し、ユーザーが日本語 TTS を利用できることを保証する。

## 実装する内容の詳細

### 手動テスト手順

#### 前提条件
- M1 の全変更が `dev` ブランチにマージ済み
- GitHub Pages のデプロイが完了 (WASM バイナリ含む)
- Chrome / Firefox / Safari の最新版を使用

#### 確認ステップ

1. **初期化確認**
   - https://ayutaz.github.io/piper-plus/ にアクセス
   - コンソール (DevTools) を開いた状態で初期化を待つ
   - 「準備完了！」(または同等のメッセージ) が表示されること
   - コンソールに `openjtalkModule is required` エラーがないこと
   - コンソールに `is not a function` エラーがないこと
   - WASM ロード関連のログ (`[piper-plus] Rust WASM G2P loaded` 等) を確認

2. **日本語テキスト合成**

   以下のテストセンテンスを順に入力し、音声が生成・再生されることを確認:

   | # | テキスト | 期待される動作 |
   |---|---------|--------------|
   | 1 | こんにちは、つくよみちゃんです。 | 自然な日本語音声 (約3秒) |
   | 2 | 今日は天気がいいですね。 | 自然な日本語音声 |
   | 3 | 東京タワーに行きたいです。 | カタカナ混在テキストの正常処理 |
   | 4 | 1234円です。 | 数字混在テキストの正常処理 |
   | 5 | Hello、こんにちは。 | 英日混在テキスト (言語検出が JA になること) |

3. **エラー確認**
   - 各合成後にコンソールにエラーが出力されていないこと
   - 「音声合成」ボタンが再度クリック可能になること (UI が固まらないこと)

4. **WASM ロード失敗時のフォールバック確認** (任意)
   - DevTools の Network タブで `piper_plus_wasm_bg.wasm` をブロック
   - ページリロード
   - 英語テキスト "Hello, how are you?" で音声合成が動作すること
   - 日本語テキストで適切なエラーメッセージが表示されること (または無音で失敗)

## 懸念事項

- **WASM バイナリサイズ**: `piper_plus_wasm_bg.wasm` (~60MB, ~19MB gzip) の初回ダウンロードに時間がかかる。ネットワーク速度によっては初期化に 10-30 秒かかる可能性がある
- **ブラウザ互換性**: Safari での WASM Memory64 や SharedArrayBuffer の対応状況。Rust WASM (`wasm-pack`) のブラウザサポート範囲を事前確認すること
- **モデルのダウンロード**: デモページのモデル (HuggingFace) のダウンロードと WASM ロードが並行するため、初期化順序によってはタイムアウトの可能性がある
- **GitHub Pages のキャッシュ**: デプロイ後、CDN キャッシュが伝搬するまでに数分かかる場合がある。テスト前にハードリロード (Ctrl+Shift+R) を実施すること

## 一から作り直すとしたら

手動ブラウザテストは再現性が低く、リリースごとに人手が必要になる。CI で自動化することを検討すべきである。

改善案:
- **Playwright E2E テスト**: Playwright を使い、CI でヘッドレスブラウザを起動してデモページをテストする。`page.goto()` → テキスト入力 → ボタンクリック → `<audio>` 要素の再生確認まで自動化可能
- **GitHub Actions での構成**: `actions/setup-node` + `npx playwright install` で CI 環境を構築。ローカルサーバー (`npx serve`) でデモページを serve し、Playwright テストを実行
- **音声出力のスモークテスト**: 合成された音声の長さ (> 0.5 秒)、サンプルレート (22050 Hz)、無音でないこと (RMS > threshold) を自動検証。波形の完全一致は不要だが、明らかな異常を検知可能
- **コスト**: Playwright テストの追加は ~1日の作業量。メンテナンスコストは低い (デモページの UI が大きく変わらない限り)
