## 1. タスク目的とゴール

### 1.1 目的

感情ラベル付き音声データセット (CREMA-D 等) から `facebook/pe-av-small` モデルを用いて audio embedding を抽出し、fork `yusuke-ai/piper-plus` の PE-A emotion loss と byte-for-byte 互換の style bank `.npz` ファイルを生成するツールを実装する。yusuke-ai fork の `.npz` スキーマ (`emotion_names`, `emotion_centroids`, `global_centroid`) は解析済みで、本家 `ayutaz/piper-plus` で独自生成する工程を本チケットで確立する。

style bank は Phase 4 (PE-A emotion loss) の `_init_pea_emotion_loss()` で `register_buffer` として読み込まれ、Phase 5 (fine-tune) で 3 項 loss (direction + centroid + margin) の教師信号として利用される。

## 2. 実装内容の詳細

### 2.1 対象ファイル

- `src/python/piper_train/tools/build_pea_style_bank.py` (新規、~400 行)

### 2.2 実装手順

1. `phase-3-4.md §3.2` のコード例を起点にしてモジュールを作成
2. `EmotionAudioDataset` クラスを実装:
   - `_load_from_crema_d()`: ファイル名パースで `{speaker}_{sentence}_{emotion_code}_{intensity}.wav` を検出
   - `_load_from_manifest()`: CSV/JSONL からロード、`audio_path,emotion` 形式
   - `__getitem__`: waveform ロード → stereo→mono → resample to 16 kHz → normalize
3. `load_pea_model()`: P0-T03 で確定した loader コード (`AutoModel.from_pretrained("facebook/pe-av-small", trust_remote_code=True)` or fallback) を使用
4. `extract_audio_embedding()`: `get_audio_embeds()` API で audio → embedding 変換、L2-normalize 適用
5. `build_style_bank()`: 全サンプルを逐次処理 (batch_size=1、PE-A は可変長のため)、感情別にまとめて centroid 計算
6. `save_style_bank()`: `np.savez` で `.npz` に保存。`emotion_names` は object dtype で str 配列として保存
7. `generate_report()`: cosine similarity matrix + 感情統計を JSON に書き出し
8. CLI パースと main 関数を追加

### 2.3 コード例 (phase-3-4.md §3.2 からの抜粋)

```python
import argparse
import json
import logging
from pathlib import Path
from typing import Optional

import numpy as np
import torch
import torchaudio
from tqdm import tqdm

_LOGGER = logging.getLogger("build_pea_style_bank")

PE_A_SAMPLE_RATE = 16000

EMOTION_CODE_MAP = {
    "ANG": "angry", "DIS": "disgusted", "FEA": "fearful",
    "HAP": "happy", "NEU": "neutral", "SAD": "sad",
}


class EmotionAudioDataset(torch.utils.data.Dataset):
    def __init__(self, data_dir, manifest_path=None, sample_rate=PE_A_SAMPLE_RATE):
        self.data_dir = Path(data_dir)
        self.sample_rate = sample_rate
        self.samples = []
        if manifest_path:
            self._load_from_manifest(manifest_path)
        else:
            self._load_from_crema_d()

    def _load_from_crema_d(self):
        audio_dir = self.data_dir / "AudioWAV"
        if not audio_dir.exists():
            raise FileNotFoundError(f"AudioWAV directory not found: {audio_dir}")
        for wav_file in sorted(audio_dir.glob("*.wav")):
            parts = wav_file.stem.split("_")
            if len(parts) < 4:
                continue
            emotion = EMOTION_CODE_MAP.get(parts[2])
            if not emotion:
                continue
            self.samples.append({"audio_path": wav_file, "emotion": emotion})

    def __getitem__(self, idx):
        item = self.samples[idx]
        waveform, sr = torchaudio.load(str(item["audio_path"]))
        if waveform.shape[0] > 1:
            waveform = waveform.mean(dim=0, keepdim=True)
        if sr != self.sample_rate:
            waveform = torchaudio.functional.resample(waveform, sr, self.sample_rate)
        max_val = waveform.abs().max()
        if max_val > 0:
            waveform = waveform / max_val
        return {
            "audio": waveform.squeeze(0),
            "emotion": item["emotion"],
            "audio_path": str(item["audio_path"]),
        }


def build_style_bank(dataset, model, device, per_utterance_dir=None):
    emotion_embeddings = {}
    all_embeddings = []
    if per_utterance_dir:
        per_utterance_dir = Path(per_utterance_dir)
        per_utterance_dir.mkdir(parents=True, exist_ok=True)

    for i in tqdm(range(len(dataset)), desc="Processing audio"):
        sample = dataset[i]
        embedding = extract_audio_embedding(model, sample["audio"], device)
        all_embeddings.append(embedding)
        emotion_embeddings.setdefault(sample["emotion"], []).append(embedding)
        if per_utterance_dir:
            utt_id = Path(sample["audio_path"]).stem
            np.save(str(per_utterance_dir / f"{utt_id}.npy"), embedding.numpy())

    emotion_names = sorted(emotion_embeddings.keys())
    emotion_centroids = torch.stack([
        torch.stack(emotion_embeddings[emotion]).mean(dim=0)
        for emotion in emotion_names
    ])
    all_embeddings_tensor = torch.stack(all_embeddings)
    global_centroid = all_embeddings_tensor.mean(dim=0)
    return emotion_names, emotion_centroids, global_centroid, all_embeddings_tensor


def save_style_bank(output_path, emotion_names, emotion_centroids, global_centroid):
    output_path = Path(output_path)
    output_path.parent.mkdir(parents=True, exist_ok=True)
    np.savez(
        str(output_path),
        emotion_names=np.array(emotion_names, dtype=object),
        emotion_centroids=emotion_centroids.cpu().numpy().astype(np.float32),
        global_centroid=global_centroid.cpu().numpy().astype(np.float32),
    )
```

完全なコードは `phase-3-4.md §3.2` (~400 行) を参照。

### 2.4 .npz スキーマ (fork との互換性)

| キー | 型 | 形状 | 備考 |
|------|-----|------|------|
| `emotion_names` | object (str) | `[N]` | ソート済み (e.g., `["angry", "disgusted", ...]`) |
| `emotion_centroids` | float32 | `[N, D]` | L2-normalize 後の centroid。D は PE-A embedding 次元 (想定 256 or 512) |
| `global_centroid` | float32 | `[D]` | 全 embedding の mean (L2-normalize 前の raw mean) |

**重要**: `emotion_centroids` の各行は L2 norm ≈ 1.0 になるように処理する (P3-T04 で検証)。fork の `_init_pea_emotion_loss()` で `F.normalize()` が再度呼ばれるため、保存時に normalize 済みでも二重 normalize では変化しない (idempotent)。

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **PE-A モデルのバッチ推論で GPU OOM**: 可変長入力のため batch_size=1 で処理。7,442 サンプルを逐次処理すると 1〜2h かかる。大規模 dataset (ESD 35,000 サンプル等) では無視できない時間コスト
- **HuggingFace datasets API 変更**: `AutoModel.from_pretrained()` の API が将来変更される可能性。P0-T03 で確定した loader コードをそのまま流用することで影響を最小化
- **GPU メモリ不足 (大規模 dataset)**: `--per-utterance-dir` で毎サンプルの embedding を disk に書き出す場合、ファイル数と容量 (7,442 × ~2 KB = 15 MB) は小さいが、ファイルシステムのオーバーヘッド (inode 消費) に注意
- **数値精度**: float32 で保存するが、mean 計算時に float64 で中間計算した方が精度が高い。大規模 dataset で累積誤差が無視できない場合は float64 中間バッファを検討
- **感情ラベルの欠損**: CREMA-D では `XX` が neutral だが、CSV manifest の場合 `emotion=""` のサンプルが混在する可能性。bad sample はスキップしてログ警告する
- **L2-normalize のタイミング**: `extract_audio_embedding()` で embedding を L2-normalize → centroid 計算 → 再 L2-normalize という 2 段階正規化。fork の実装と一致するか P0-T03 で確認

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A**: K-means クラスタリングで感情を自動抽出 (ラベルなし学習)
    - 利点: ラベル付きデータセットに依存しない。言語/文化を超えた感情クラスタリングが可能
    - 欠点: クラスタ数 (6?) の決定が難しい、意味ラベル (angry/happy) が付かないので infer 時に人間が対応付けが必要
    - 適用: 将来的な「教師なし感情 bank 生成」機能として追加

- **代替案 B**: t-SNE/UMAP で 2D 可視化ツール追加
    - 利点: 感情分離度を視覚的に検証できる、レポートの品質向上
    - 欠点: 依存パッケージ増加 (scikit-learn, umap-learn)、可視化が opinionated になりがち
    - 適用: P3-T04 に含める or 別チケットとして分離

- **代替案 C**: Hugging Face datasets 統合
    - 利点: 複数データセット (CREMA-D, ESD, EmoV-DB) を統一 API で扱える、ストリーミング DL で大規模対応
    - 欠点: datasets パッケージ依存、`.npz` スキーマを HF 側で保持する設計になる
    - 適用: 次期バージョンで `datasets.load_dataset("crema_d", split="audio_wav")` ベースに移行

- **代替案 D**: ESD と CREMA-D をデフォルト同梱
    - 利点: 多言語 (ZH) 対応、感情数増加 (ESD は 5 感情)
    - 欠点: ESD は研究目的ライセンス → 商用利用で制約あり。`peav-style-conditioning.md §12` より CREMA-D 単独が商用に安全
    - 適用: `--dataset {crema_d|esd|emov_db}` の CLI フラグで選択可能にする

- **代替案 E**: Style bank を Hugging Face Hub に置いて自動 DL
    - 利点: build_pea_style_bank.py を実行せずに即座に PE-A loss を試せる
    - 欠点: style bank の品質が HF 配布元に依存、ユーザー独自感情の追加ができない
    - 適用: Phase 5 完了後に公式 style bank を `ayousanz/piper-plus-pea-style-bank` として HF Hub 公開

### 7.2 現在の実装を選んだ理由

- fork `yusuke-ai/piper-plus` の `.npz` スキーマと byte-for-byte 互換にすることで、Phase 4 の PE-A loss を `.npz` 読み込みから変更なしで動作させる
- `phase-3-4.md §3.2` でコード例が完全提示されており、実装リスクが最小化されている
- CREMA-D 単独は商用可 (ODbL)、7,442 サンプルは品質 PoC に十分、英語のみだが Phase 0 段階では言語横断より品質優先
- K-means / UMAP は魅力的だが、Phase 3 のスコープを超える (4〜8h 工数に収まらない)

### 7.3 リファクタ機会 (将来)

- `build_pea_style_bank.py` を `piper_train.perception.style_bank` モジュール配下に移動し、programmatic API を提供 (`from piper_train.perception.style_bank import build_style_bank`)
- 複数データセットをマージして style bank を生成する機能 (CREMA-D + EmoV-DB → 共通感情 6 つを平均) を追加
- Style bank の visualizer CLI (`--visualize out.html`) で UMAP 2D プロットを生成
- 差分更新: 既存 `.npz` に新しい感情だけ追加する `--append-emotion <name>` オプション
