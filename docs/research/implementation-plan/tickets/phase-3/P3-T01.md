## 1. タスク目的とゴール

### 1.1 目的

PE-A emotion style bank の第一候補データセットである CREMA-D (Crowd-sourced Emotional Multimodal Actors Dataset) を、Phase 3 以降で利用可能な状態に準備する。CREMA-D は 7,442 発話、91 話者 (46 女性 + 45 男性)、6 感情 (angry / disgusted / fearful / happy / neutral / sad)、16 kHz サンプルレート (PE-A と一致) で、ODbL (Open Database License) 1.0 により商用利用・学習利用ともに可能な公開データセット。

本チケットでは、(1) CREMA-D リポジトリの取得、(2) ディスク容量と完全性の検証、(3) LJSpeech 互換の metadata 整形、(4) 感情タグ (`ANG`, `DIS`, `FEA`, `HAP`, `NEU`, `SAD`) の utterance metadata 注入 を行う。Phase 3 の build_pea_style_bank.py (P3-T02) および inject_style_labels.py (P3-T03) が本チケット成果物をそのまま参照する。

### 1.2 ゴール (Definition of Done)

- [ ] `src/python/piper_train/tools/download_crema_d.py` (または `scripts/prepare_crema_d.sh`) を新規作成
- [ ] GitHub リポジトリ (https://github.com/CheyneyComputerScience/CREMA-D) から AudioWAV/ ディレクトリをダウンロードし、`--data-dir` で指定された場所に配置
- [ ] ダウンロード後、WAV ファイル数が 7,442 であることを検証 (±1 は許容)
- [ ] 各 WAV のサンプルレートが 16 kHz、ビット深度 16-bit であることをサンプル検証 (先頭 100 ファイル)
- [ ] LJSpeech 互換 `metadata.csv` を生成 (`utt_id|text|emotion` 形式、wav ファイル名由来の utt_id)
- [ ] 感情タグを `emotions.csv` (`utt_id,emotion` 形式) に書き出し、P3-T03 (inject_style_labels.py) が直接読める形にする
- [ ] ODbL 1.0 + Community License の LICENSE ファイルを `--data-dir/LICENSE_CREMA_D.txt` にコピー配置
- [ ] CLI オプション: `--data-dir`, `--skip-if-exists`, `--verify-only`
- [ ] ダウンロード時間とディスク使用量をログに記録 (目安: 27 GB 圧縮 / 48 GB 解凍、2〜3h @ 100 Mbps)

## 2. 実装内容の詳細

### 2.1 対象ファイル

- `src/python/piper_train/tools/download_crema_d.py` (新規、~200 行)
- オプションで `scripts/prepare_crema_d.sh` (bash wrapper, ~50 行)

### 2.2 実装手順

1. `download_crema_d.py` 冒頭で `argparse` CLI をセットアップ (`--data-dir`, `--skip-if-exists`, `--verify-only`)
2. `download()`: GitHub リポジトリを `git clone --depth=1 https://github.com/CheyneyComputerScience/CREMA-D.git <data-dir>` で取得。`--skip-if-exists` 時は既存ディレクトリをスキップ
3. `verify()`: `AudioWAV/*.wav` の数をカウントし、7,442 に近いことを確認。`soundfile.info()` で先頭 100 ファイルのサンプルレート・ビット深度を検証
4. `generate_metadata()`: `AudioWAV/1001_IWW_ANG_HI.wav` のようなファイル名から `speaker`, `sentence`, `emotion_code`, `intensity` をパースし、LJSpeech 互換 `metadata.csv` を生成
5. `generate_emotion_csv()`: `{utt_id},{emotion}` 形式の CSV を `--data-dir/emotions.csv` に書き出し
6. `copy_license()`: リポジトリ内の `LICENSE.txt` または `README.md` から ODbL 1.0 表記を抽出し、`LICENSE_CREMA_D.txt` として保存
7. ログ出力: 開始時刻、終了時刻、WAV 数、総容量、失敗ファイル一覧

### 2.3 コード例 (抜粋)

```python
#!/usr/bin/env python3
"""Download CREMA-D dataset and prepare LJSpeech-style metadata."""

import argparse
import logging
import subprocess
from pathlib import Path
from typing import Optional

import soundfile as sf
from tqdm import tqdm

_LOGGER = logging.getLogger("download_crema_d")

CREMA_D_REPO = "https://github.com/CheyneyComputerScience/CREMA-D.git"
EXPECTED_WAV_COUNT = 7442

EMOTION_CODE_MAP = {
    "ANG": "angry",
    "DIS": "disgusted",
    "FEA": "fearful",
    "HAP": "happy",
    "NEU": "neutral",
    "SAD": "sad",
}

# CREMA-D の 12 sentences (IWW = "It's eleven o'clock", etc.)
SENTENCE_MAP = {
    "IEO": "It's eleven o'clock.",
    "TIE": "That is exactly what happened.",
    "IOM": "I'm on my way to the meeting.",
    "IWW": "I wonder what this is about.",
    "TAI": "The airplane is almost full.",
    "MTI": "Maybe tomorrow it will be cold.",
    "IWL": "I would like a new alarm clock.",
    "ITH": "I think I have a doctor's appointment.",
    "DFA": "Don't forget a jacket.",
    "ITS": "I think I've seen this before.",
    "TSI": "The surface is slick.",
    "WSI": "We'll stop in a couple of minutes.",
}


def download(data_dir: Path, skip_if_exists: bool = False) -> None:
    if data_dir.exists() and skip_if_exists:
        _LOGGER.info("Skipping download (already exists): %s", data_dir)
        return
    data_dir.parent.mkdir(parents=True, exist_ok=True)
    _LOGGER.info("Cloning CREMA-D from %s into %s", CREMA_D_REPO, data_dir)
    subprocess.run(
        ["git", "clone", "--depth=1", CREMA_D_REPO, str(data_dir)],
        check=True,
    )


def verify(data_dir: Path, sample_n: int = 100) -> dict:
    """Verify WAV count and audio format sampling."""
    audio_dir = data_dir / "AudioWAV"
    if not audio_dir.exists():
        raise FileNotFoundError(f"AudioWAV directory not found: {audio_dir}")

    wav_files = sorted(audio_dir.glob("*.wav"))
    wav_count = len(wav_files)
    _LOGGER.info("WAV count: %d (expected ~%d)", wav_count, EXPECTED_WAV_COUNT)
    if abs(wav_count - EXPECTED_WAV_COUNT) > 1:
        _LOGGER.warning(
            "WAV count deviates from expected: %d vs %d",
            wav_count, EXPECTED_WAV_COUNT,
        )

    bad_files = []
    for wav_file in tqdm(wav_files[:sample_n], desc="Verifying sample"):
        try:
            info = sf.info(str(wav_file))
            if info.samplerate != 16000:
                bad_files.append((wav_file.name, f"sr={info.samplerate}"))
            if info.subtype != "PCM_16":
                bad_files.append((wav_file.name, f"subtype={info.subtype}"))
        except Exception as e:
            bad_files.append((wav_file.name, str(e)))

    if bad_files:
        _LOGGER.warning("Found %d problematic files (sample of %d)", len(bad_files), sample_n)
        for fn, err in bad_files[:10]:
            _LOGGER.warning("  %s: %s", fn, err)

    return {"wav_count": wav_count, "bad_files": bad_files}


def generate_metadata(data_dir: Path) -> None:
    audio_dir = data_dir / "AudioWAV"
    metadata_path = data_dir / "metadata.csv"
    emotions_path = data_dir / "emotions.csv"

    with open(metadata_path, "w") as meta_f, open(emotions_path, "w") as emo_f:
        emo_f.write("# utt_id,emotion\n")
        for wav_file in sorted(audio_dir.glob("*.wav")):
            parts = wav_file.stem.split("_")
            if len(parts) < 4:
                _LOGGER.warning("Skipping malformed filename: %s", wav_file.name)
                continue
            speaker, sentence_code, emotion_code, intensity = parts[:4]
            emotion = EMOTION_CODE_MAP.get(emotion_code)
            if not emotion:
                _LOGGER.warning("Unknown emotion code '%s'", emotion_code)
                continue
            text = SENTENCE_MAP.get(sentence_code, sentence_code)
            utt_id = wav_file.stem
            meta_f.write(f"{utt_id}|{text}|{emotion}\n")
            emo_f.write(f"{utt_id},{emotion}\n")
    _LOGGER.info("metadata.csv and emotions.csv generated in %s", data_dir)


def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--data-dir", type=Path, required=True)
    parser.add_argument("--skip-if-exists", action="store_true")
    parser.add_argument("--verify-only", action="store_true")
    args = parser.parse_args()

    logging.basicConfig(level=logging.INFO, format="%(asctime)s %(levelname)s %(message)s")

    if not args.verify_only:
        download(args.data_dir, skip_if_exists=args.skip_if_exists)

    stats = verify(args.data_dir)
    _LOGGER.info("Verify stats: %s", stats)

    generate_metadata(args.data_dir)
    _LOGGER.info("CREMA-D preparation complete: %s", args.data_dir)


if __name__ == "__main__":
    main()
```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **CREMA-D の 27 GB DL が不安定**: 大学機関のホスティングのため、帯域制限や一時的な接続失敗が起こりうる。`git clone --depth=1` が失敗した場合は `git lfs` 経由 or HTTPS 直接 DL のフォールバックを検討
- **ディスク容量不足**: 解凍後 48 GB 必要。`/data` パーティションの空き容量を事前確認
- **ネットワーク切断**: 途中で切断された場合、`git clone` は中途ファイルを残す可能性あり。`--skip-if-exists` は単純な存在チェックなので、破損状態の検知には `verify()` を呼ぶ必要がある
- **ライセンス表記の不備**: ODbL は attribution 要求があるため、生成物には必ず `LICENSE_CREMA_D.txt` を同梱する必要がある。README に引用追記も必要
- **GitHub 経由ではなく Zenodo や他のミラー**: 公式は GitHub だが、GitHub の容量制限により Git LFS or 外部ホスティングに移行する可能性あり。README を事前確認

### 6.2 レビュー項目

- [ ] ODbL 1.0 + Community License の attribution 表記が `LICENSE_CREMA_D.txt` と `metadata.csv` コメント行に含まれていること
- [ ] `peav-style-conditioning.md §12` 記載のライセンス情報と整合していること
- [ ] `--skip-if-exists` が冪等 (複数回実行しても結果が変わらない) であること
- [ ] `verify()` が先頭 100 ファイルのみ検証 (全数検証は時間がかかりすぎる) し、サンプリングでエラー率を推定する方式になっていること
- [ ] ログ出力が INFO レベルで、タイムスタンプ付きになっていること
- [ ] 一時失敗時の再試行戦略 (git clone 失敗時の DL 再開) がドキュメント化されていること

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A**: Hugging Face Datasets 経由で `datasets.load_dataset("crema_d")` を使う
    - 利点: キャッシュ機構が洗練されており、部分 DL・ストリーミングが可能。複数データセットを統一 API で扱える
    - 欠点: HF datasets に CREMA-D がホスト済みか要確認。ホストされていない場合は自前登録が必要。audio 特有の Arrow フォーマット変換で容量が増加する可能性
- **代替案 B**: Zenodo or academic archive の ZIP 直接 DL
    - 利点: git clone より安定。ネットワーク帯域を全量使える
    - 欠点: CREMA-D 公式の Zenodo ホスト先が未確認。非公式ミラーだと改変の可能性
- **代替案 C**: torchaudio.datasets.CREMAD のような PyTorch 公式データセットローダーを使う (仮)
    - 利点: PyTorch 生態系との統合、DL 進捗表示が標準装備
    - 欠点: torchaudio に CREMA-D loader が存在するか要確認。存在しない場合は自前実装が必要

### 7.2 現在の実装を選んだ理由

- CREMA-D 公式リポジトリ (GitHub) が最もシンプルで信頼性が高く、`git clone --depth=1` で必要最小限の DL で済む
- `phase-3-4.md §3.1` で公式リポジトリを指定しており、研究計画との整合性が取れる
- 2〜3h の DL 時間は許容範囲内で、Phase 3 全体の所要時間 (4〜8h) の半分以下
- ODbL ライセンス表記が確実に含まれる (LICENSE ファイルがリポジトリルートに存在)

### 7.3 リファクタ機会 (将来)

- Hugging Face Hub にミラー公開 (ayousanz/crema-d-mirror のような形で、piper-plus 用に整形済み) → DL 速度と再現性が向上
- `prepare_crema_d.sh` として shell wrapper を用意し、`uv run` に依存しない環境でも使えるようにする
- 複数データセット (ESD、EmoV-DB、JTES) に対応する統一 downloader (`download_emotion_dataset.py --dataset crema_d|esd|emov_db|jtes`) にリファクタ
- HuggingFace datasets の `datasets.load_dataset()` + cache_dir を使った実装への移行 (ESD の研究目的ライセンスの取扱いも含めて統一設計)
