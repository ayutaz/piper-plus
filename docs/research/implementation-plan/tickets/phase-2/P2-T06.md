```go
// "style_vector": float32 [1, styleDim] + "style_vector_mask": int64 [1, 1]
if e.capabilities.HasStyleVector {
    styleDim := e.capabilities.StyleVectorDim
    var styleData []float32
    var maskVal int64

    if len(req.StyleVector) > 0 {
        if int32(len(req.StyleVector)) != styleDim {
            return nil, &InferenceError{Msg: fmt.Sprintf(
                "StyleVector length %d != expected %d", len(req.StyleVector), styleDim)}
        }
        styleData = req.StyleVector
        maskVal = 1
    } else {
        styleData = make([]float32, styleDim)
        maskVal = 0
    }

    styleTensor, err := ort.NewTensor(ort.NewShape(1, int64(styleDim)), styleData)
    if err != nil {
        return nil, &InferenceError{Msg: "failed to create style_vector tensor", Err: err}
    }
    inputNames = append(inputNames, "style_vector")
    inputs = append(inputs, styleTensor)

    maskTensor, err := ort.NewTensor(ort.NewShape(1, 1), []int64{maskVal})
    if err != nil {
        return nil, &InferenceError{Msg: "failed to create style_vector_mask tensor", Err: err}
    }
    inputNames = append(inputNames, "style_vector_mask")
    inputs = append(inputs, maskTensor)
}
```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX metadata 読み込み標準化されていない問題**: Go の `ort` binding (おそらく `yalue/onnxruntime_go` or `github.com/ort-rs/onnxruntime-go`) の metadata API がバージョンによって異なる。既存コードでどのように `SpeakerEmbeddingDim` を取得しているかを Explorer が確認すること
- **mask パターンで input が NULL 扱いされる ONNX Runtime の制約**: Go の `ort.NewTensor` は nil data を受けないため、`has_style_vector=true` のときは必ず `make([]float32, dim)` でゼロ初期化
- **`.npy` parser の複雑さ**: Python dict literal をパースする必要 (`{'descr': '<f4', 'fortran_order': False, 'shape': (256,), }`)。正規表現ではなく字句解析が必要だが、シンプル版 (shape と descr のみ抽出) で実用上十分
- **slice の shared backing array 問題**: `req.StyleVector` を `ort.NewTensor` に渡す際、ORT 側が参照を保持する場合、呼び出し側が slice を変更すると inference が崩れる。Go の慣習として ORT 側が内部コピーするはずだが、要確認
- **`flag` package vs `cobra`**: 既存 `cmd/piper-plus/main.go` が `flag` package を使っていれば踏襲、`cobra` なら合わせる

### 6.2 レビュー項目

- [ ] `SynthesisRequest.StyleVector` が `[]float32` (not `*[]float32`) で `nil` デフォルト動作
- [ ] `InferenceError` の wrap が一貫 (`&InferenceError{Msg: ..., Err: err}` pattern)
- [ ] `LoadFloat32Npy` が `<f4` (little-endian float32) のみ許可
- [ ] 既存 793 テストに回帰なし
- [ ] CLI の `--style-vector` help message に format 要件記載

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: `SynthesisRequest` を builder pattern (`NewSynthesisRequest().WithStyleVector(...).Build()`) に変更**
    - 利点: optional fields が増えても公開 struct field が不要
    - 欠点: Go idiomatic な struct literal の簡潔さが失われる
- **代替案 B: `.npy` の代わりに `encoding/gob` で serialize した `.gob` ファイル**
    - 利点: Go native format、parser 簡単
    - 欠点: Python / Rust との互換性ゼロ (cross-runtime test 不可)
- **代替案 C: `StyleVector []float32` を struct field ではなく functional option (`WithStyleVector(v)`) で受ける**
    - 利点: 必須 vs optional の区別が明確
    - 欠点: 既存 `SpeakerEmbedding` との対称性が崩れる

### 7.2 現在の実装を選んだ理由

- 既存 `SpeakerEmbedding []float32` と同じ構造で対称性、Go コミュニティの conventional Go style
- `.npy` format は Python / Rust / C# と共通で cross-runtime 互換性テスト (T08) に有利
- struct field の直接 expose は Go の慣習 (interface-less data types)

### 7.3 リファクタ機会 (将来)

- `SynthesisRequest` に optional fields が増えるにつれ、`NewSynthesisRequest(opts ...Option)` pattern に移行候補
- `LoadFloat32Npy` / `LoadInt64Npy` / etc. を `numpy` サブパッケージに集約
