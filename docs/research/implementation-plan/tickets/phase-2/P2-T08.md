## 1. タスク目的とゴール

### 1.1 目的

6 ランタイム (Python / C++ / Rust / C# / Go / WASM) が同じ `style_vector` 入力・同じ ONNX モデル・同じ phoneme_ids から生成する wav の byte 一致 (またはε以内の浮動小数誤差) を検証する。既存の Phoneme Timing cross-runtime test パターン (Rust/Go/C++/C# が `(hop_length / sample_rate) * 1000` で byte-for-byte 互換) と同じ品質を、style_vector 機能でも担保する。

### 1.2 ゴール (Definition of Done)

- [ ] `tests/cross_runtime/test_style_vector_compat.py` 新規作成 (pytest ベース)
- [ ] 以下 5 パターンの cross-runtime 比較テスト:
    1. **Python ↔ Rust**: `uv run piper_train.infer_onnx` vs `cargo run -p piper-plus-cli` で wav byte 一致
    2. **Python ↔ C++**: `infer_onnx.py` vs `./build/piper` で wav byte 一致
    3. **Python ↔ C#**: `infer_onnx.py` vs `dotnet run --project PiperPlus.Cli` で wav byte 一致
    4. **Python ↔ Go**: `infer_onnx.py` vs `go run ./cmd/piper-plus` で wav byte 一致
    5. **Python ↔ WASM**: `infer_onnx.py` vs Playwright で browser 経由の wav byte 比較 (ε 許容)

### 2.2 実装手順

1. 小型 fixture ONNX を生成 (既存 `test/models/multilingual-test-medium.onnx` を元にし、style_vector_dim=256 でエクスポート)
2. Python ランタイム (T02 で実装済み) で reference wav を生成 → `expected_python.wav` に保存
3. `test_style_vector_compat.py` で subprocess ベースの比較テスト実装:
    - Python CLI (`uv run python -m piper_train.infer_onnx ...`)
    - Rust CLI (`cargo run --release -p piper-plus-cli ...`)
    - C++ CLI (`./build/piper ...`)
    - C# CLI (`dotnet run --project src/csharp/PiperPlus.Cli ...`)
    - Go CLI (`go run ./src/go/cmd/piper-plus ...`)
    - WASM: Playwright で `npm run dev` → page.evaluate で synthesize 実行 → audio result を base64 で回収
4. byte 比較: `filecmp.cmp(ref_wav, target_wav, shallow=False)` で完全一致を確認
5. 誤差許容: FP16 GPU / WASM は `numpy.allclose(ref_audio, target_audio, atol=1e-3)` (float32 wav sample LSB 数 bit 許容)
6. `style-vector-contract.toml` に以下を記載:
    - ONNX 入力名: `style_vector`, `style_vector_mask`
    - shape: `[1, style_vector_dim]`, `[1, 1]`
    - dtype: `float32`, `int64`
    - metadata キー: `style_vector_dim`, `style_condition_mode`
    - config.json フィールド: `style_vector_dim`, `style_condition_mode`
    - CLI 引数: `--style-vector <path>` (6 ランタイム共通)
    - `.npy` フォーマット: v1.0/v2.0、dtype=`<f4`、shape 1D or 2D
7. CI ワークフロー: Linux runner で Python + Rust + C++ + C# + Go を並列実行 (WASM は別 job)

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX Runtime provider 差異**: CPU provider (MLAS) は決定論的だが、CUDA / CoreML / DirectML は非決定論的要素あり。本テストは CPU provider 固定で実行
- **onnxruntime-web の FP32 kernel**: WASM 版 onnxruntime-web は SIMD 最適化により CPU provider とわずかに異なる結果を出す場合あり。ε=1e-3 で許容
- **fixture ONNX の Git LFS サイズ**: 小型でも 10〜50MB 程度になる可能性。代替案として `tests/fixtures/generate_style256.py` で動的生成 (固定 seed の VitsModel 初期化 → export) する手も検討
- **CI 実行時間**: 6 ランタイムの build + test で CI が長時間化 (30〜45 分)。PR では Linux のみ、nightly で全 OS の matrix を走らせる運用
- **Playwright 依存追加**: WASM テストに Playwright が必要。既に wasm-build.yml で利用していれば流用、なければ依存追加のコスト
- **フリーズ DP や EMA などの non-determinism**: Phase 1 の学習側で `torch.manual_seed` / `torch.use_deterministic_algorithms(True)` を保証しておかないと fixture 再現できない

### 6.2 レビュー項目

- [ ] `style-vector-contract.toml` の内容が 6 ランタイムの実装と完全一致
- [ ] fixture ONNX の style_vector_dim が 256 で、metadata に反映されている

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: `cross_runtime_test.sh` シェルスクリプトで全ランタイム順次実行、exit code で pass/fail**
    - 利点: CI YAML 不要、開発者がローカルで実行しやすい
    - 欠点: GitHub Actions の job parallelism を活用できず実行時間が長い
- **代替案 B: 各ランタイムの native test framework に byte 比較テストを追加 (Python pytest, Rust cargo test, C++ GoogleTest, etc.) し、共通 fixture を参照**
    - 利点: 各言語の開発者が自分の慣れた framework で実行できる
    - 欠点: テスト結果の集約が難しい、重複コード
- **代替案 C: Docker Compose で全ランタイム + fixtures をコンテナ化、`docker compose run cross-runtime-test`**
    - 利点: 環境再現性最高、CI でも同じ image を使える
    - 欠点: Docker build コスト、開発時のイテレーション低下

### 7.2 現在の実装を選んだ理由

- pytest + subprocess の構成は既存 `tests/` と統一され、開発者が追加テストを書きやすい
- `filecmp.cmp` は stdlib で外部依存なし、byte-for-byte の厳密比較に適する
- `cross-runtime-test.yml` で matrix を活用し、並列実行で CI 時間短縮

### 7.3 リファクタ機会 (将来)

- fixture 生成を `make test-fixtures` 等の Makefile target に集約
- `CrossRuntimeTestHarness` クラスで各ランタイムの呼び出しを共通化、新しいランタイム (例: Swift, Kotlin) 追加時の boilerplate 削減
- `style-vector-contract.toml` の内容を各ランタイムのコードジェン入力に使い、手書き差分を削減 (代替案 C の codegen アプローチ)
