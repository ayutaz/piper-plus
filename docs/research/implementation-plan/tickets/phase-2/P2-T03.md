## 1. タスク目的とゴール

### 1.1 目的

`libpiper_plus` (C ABI 互換共有ライブラリ) と `piper` C++ CLI に style_vector 入力サポートを追加する。既存 `speaker_embedding` パターンに合わせる形で `PiperPlusSynthOptions` 構造体を拡張し、`_reserved` フィールドを使い切ることで ABI 互換性を維持する (Dart FFI / Godot GDExtension の既存バインディング破壊を避ける)。

```cpp
// style_vector_mask: int64 [1, 1]
if (model.has_style_vector) {
    std::array<int64_t, 2> styleShape = {1, model.style_vector_dim};
    std::vector<float> styleData;

    if (!config.styleVector.empty()) {
        assert(static_cast<int32_t>(config.styleVector.size()) == model.style_vector_dim);
        styleData = config.styleVector;
    } else {
        styleData.assign(model.style_vector_dim, 0.0f);
    }

    auto styleTensor = Ort::Value::CreateTensor<float>(
        memInfo, styleData.data(), styleData.size(), styleShape.data(), styleShape.size());
    inputTensors.push_back(std::move(styleTensor));
    inputNames.push_back("style_vector");

    std::array<int64_t, 2> maskShape = {1, 1};
    std::array<int64_t, 1> maskData = {config.styleVector.empty() ? 0 : 1};
    auto maskTensor = Ort::Value::CreateTensor<int64_t>(
        memInfo, maskData.data(), maskData.size(), maskShape.data(), maskShape.size());
    inputTensors.push_back(std::move(maskTensor));
    inputNames.push_back("style_vector_mask");
}
```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX metadata 読み込み標準化されていない問題**: Ort::ModelMetadata の C++ API は `GetCustomMetadataMapKeys(allocator)` と `LookupCustomMetadataMap(key, allocator)` の 2 ステップが必要。allocator の管理を間違えるとメモリリーク
- **mask パターンで input が NULL 扱いされる ONNX Runtime の制約**: Ort::Value は nullptr を許容しないため、`has_style_vector=true` なら必ずゼロ初期化した tensor を渡す必要
- **ABI 互換性破壊リスク**: `_reserved[5]` → `_reserved[3]` + `style_vector` + `style_vector_dim` の変更で `sizeof` が維持されるか、compiler padding で変わる可能性。`static_assert` で防御
- **Windows MSVC での `int32_t` alignment 問題**: `const float *` が 8 byte (x64) で、`int32_t` が 4 byte のため、padding が挿入されて sizeof が変わる恐れ。`#pragma pack` を現状でも使用していなければ、`_reserved` の個数計算を再確認
- **Dart FFI と GDExtension の既存バインディング**: Dart の `ffi.Struct` に手書きでフィールド列挙されている場合、struct 拡張で既存コードが動かなくなる。`examples/dart/` を更新する必要あり

### 6.2 レビュー項目

- [ ] `piper_plus_default_options()` が `{0}` 初期化と等価 (ゼロ埋めで style_vector=NULL になる)

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: `style_vector` を `speaker_embedding` に concatenate して 1 入力にまとめる**
    - 利点: C API の struct 拡張不要、ABI 完全互換
    - 欠点: 学習側で concatenation の順序・次元を別管理が必要、dim_style と dim_speaker_emb の組み合わせ爆発
- **代替案 B: 新規 API `piper_plus_synthesize_v2(opts_v2, ...)` を追加、既存 `piper_plus_synthesize` は変えない**
    - 利点: ABI 100% 保証、既存バインディング無改修
    - 欠点: API 重複で保守コスト増、呼び出し側が v1/v2 を選ぶ必要
- **代替案 C: `PiperPlusSynthOptions` を opaque handle + setter/getter API に書き換え**
    - 利点: 将来の拡張がすべて API 追加のみで済む
    - 欠点: 既存 API の全面書き換え、Dart FFI / Godot GDExtension 全サンプルの更新が必要

### 7.2 現在の実装を選んだ理由

- `_reserved` フィールドを先行投資として確保した当初設計の意図通り、拡張余地を活用できる
- ABI サイズ維持により Dart FFI / Godot GDExtension の手書きバインディングが動作継続
- `speaker_embedding` との構造的対称性により、他ランタイム (T04〜T07) と同じメンタルモデルで実装レビューできる

### 7.3 リファクタ機会 (将来)

- `PiperPlusSynthOptions` に optional input を追加する度に `_reserved` を食いつぶすパターンは数年で枯渇する。Phase 6 以降で opaque handle ベースの v2 API を検討 (代替案 C)
- ONNX metadata 読み込みユーティリティ `loadOnnxCustomMetadata(session, key, default)` を `src/cpp/onnx_utils.h` に切り出して、`speaker_embedding_dim`, `style_vector_dim`, 将来の `emotion_dim` で共通化
