### 2.2 実装手順

1. `voice.py` の `PiperVoice.synthesize` シグネチャに `style_vector` を追加 (既存 `speaker_embedding=None` の後ろ)
2. `_synthesize_ids_core` 内で、ONNX session の `get_inputs()` に `style_vector` が含まれるか検出 (`has_style_vector = "style_vector" in input_names`)
3. `has_style_vector` が True の場合:
    - `style_vector` 引数が渡されていれば `.reshape(1, -1)` で `[1, dim]` shape に整形、`inputs["style_vector"] = ...`、`inputs["style_vector_mask"] = np.array([[1]], dtype=np.int64)`
    - 渡されていなければ `np.zeros((1, style_vector_dim), dtype=np.float32)` + `np.array([[0]], dtype=np.int64)` を入力
4. `style_vector_dim` は次の順で解決: (1) ONNX metadata の `custom_metadata_map["style_vector_dim"]`、(2) `config.style_vector_dim`、(3) デフォルト 0
5. `infer_onnx.py` CLI:
    - `parser.add_argument("--style-vector", default=None, metavar="PATH", help="...")` 追加
    - 引数が `.npy` ファイルパスなら `np.load(path).astype(np.float32)` で読込
    - `PiperVoice.synthesize(..., style_vector=loaded_array)` に渡す
6. `http_server.py`:
    - FastAPI Pydantic model (`TTSRequest`) に `style_vector: Optional[str]` (base64 エンコードされた float32 array) フィールド追加
    - `base64.b64decode` + `np.frombuffer(..., dtype=np.float32)` でデコード

### 2.4 style_vector 版への書き換え

```python
# voice.py 内 (synthesize メソッド)
def synthesize(
    self,
    text,
    ...,
    speaker_embedding=None,
    speaker_embedding_mask=None,
    style_vector: Optional[np.ndarray] = None,  # 新規
):
    ...

# _synthesize_ids_core 内
input_names = {inp.name for inp in self.session.get_inputs()}
has_style_vector = "style_vector" in input_names

if has_style_vector:
    style_vector_dim = self._resolve_style_vector_dim()  # metadata → config → 0

    if style_vector is not None and style_vector_dim > 0:
        style_vec = np.asarray(style_vector, dtype=np.float32)
        if style_vec.ndim == 1:
            style_vec = style_vec.reshape(1, -1)
        assert style_vec.shape == (1, style_vector_dim), \
            f"style_vector shape mismatch: expected (1, {style_vector_dim}), got {style_vec.shape}"
        inputs["style_vector"] = style_vec
        inputs["style_vector_mask"] = np.array([[1]], dtype=np.int64)
    else:
        inputs["style_vector"] = np.zeros((1, style_vector_dim), dtype=np.float32)
        inputs["style_vector_mask"] = np.array([[0]], dtype=np.int64)

def _resolve_style_vector_dim(self) -> int:
    meta = self.session.get_modelmeta().custom_metadata_map
    if "style_vector_dim" in meta:
        return int(meta["style_vector_dim"])
    return getattr(self.config, "style_vector_dim", 0)
```

CLI:

```python
parser.add_argument(
    "--style-vector",
    default=None,
    metavar="PATH",
    help="Path to a .npy file containing style_vector (float32 array)",
)

    style_vec = np.load(args.style_vector).astype(np.float32)

```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX metadata 読み込み標準化されていない問題**: onnxruntime の `custom_metadata_map` は python バージョンによって返す型が `Dict[str, str]` でない場合あり (古いバージョンでは Mapping)。`dict(meta)` で明示変換すべき
- **mask パターンで input が NULL 扱いされる ONNX Runtime の制約**: `np.zeros((1, style_vector_dim), dtype=np.float32)` を常に入力する必要。dim=0 のときは入力をスキップ (既存モデル互換性)
- **PyTorch / numpy の dtype 取り違え**: `style_vector.dtype` が `float64` のままだと onnxruntime が拒否。`astype(np.float32)` の明示変換が必須
- **`.npy` ファイル形式の仕様合意**: Phase 3 の `build_pea_style_bank.py` が出力する npy 形式 (shape `[num_styles, dim]` か `[dim]` か) を T02 と合わせる必要。ここでは **1D (dim,)** または **2D (1, dim)** の両方を受理する実装にする
- **FastAPI / Pydantic での float32 array 受け渡し**: Pydantic v2 は ndarray を直接サポートしないため、base64 文字列で受ける実装とし、サーバ側で decode

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: `style_vector` を `PiperVoice.__init__` の cache に渡し、毎回 synthesize に渡さない**
    - 利点: 同じ voice で複数テキストを合成するとき、毎回 np.ndarray を引数に渡す手間が減る
    - 欠点: stateless な synthesize API の原則に反する。声質とスタイルを動的に切り替える用途 (例: 対話 UI での感情切替) で使いづらい
- **代替案 B: config.json ではなく HF Hub model card の README front matter に metadata を書く**
    - 利点: 単一ソースの真実性 (config.json と metadata が乖離しない)
    - 欠点: オフライン運用でモデルカード取得できない場合に詰む、HF 依存が増える
- **代替案 C: Pydantic v2 の `Field(default_factory=...)` で ndarray を直接受けるカスタム型を作る**
    - 利点: base64 encode/decode が不要、API が直感的
    - 欠点: ndarray の JSON serialization は Pydantic 標準にないため、カスタムバリデータが複雑化

### 7.2 現在の実装を選んだ理由

- `speaker_embedding` と同じパターンで T03〜T07 の他ランタイム実装との対応関係を明確にできる
- metadata → config → default の 3 段階フォールバックは ONNX Runtime 各言語 binding の仕様差を吸収できる
- base64 encode での API 受け渡しは Rust / C# / Go / JS からの HTTP アクセスで他言語間互換性が最も高い

### 7.3 リファクタ機会 (将来)

- `PiperVoice` の conditioning パラメータ (speaker_id, speaker_embedding, style_vector, language_id, prosody_features) を `ConditioningInputs` データクラスに集約。シグネチャの膨張を抑える
- `_resolve_style_vector_dim()` / `_resolve_speaker_embedding_dim()` の共通化: `_resolve_optional_input_dim(name, default)` ヘルパーに抽出
