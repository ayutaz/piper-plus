## 1. タスク目的とゴール

### 2.4 style_vector 版への書き換え

```python
# Style vector inputs (新規追加: speaker_embedding ブロックの直後)
style_vector_dim = getattr(model_g, "style_vector_dim", 0)
if style_vector_dim > 0:
    dummy_style_vector = torch.zeros(1, style_vector_dim, dtype=torch.float32)
    dummy_style_vector_mask = torch.ones(1, 1, dtype=torch.int64)

    dummy_input_list.append(dummy_style_vector)
    input_names.append("style_vector")
    dynamic_axes["style_vector"] = {0: "batch_size"}

    dummy_input_list.append(dummy_style_vector_mask)
    input_names.append("style_vector_mask")
    dynamic_axes["style_vector_mask"] = {0: "batch_size"}

    _LOGGER.info(
        "Exporting model with style_vector support (dim=%d, mode=%s)",
        style_vector_dim,
        getattr(model_g, "style_condition_mode", "global"),
    )
```

`torch.onnx.export()` 後のメタデータ書き込み:

```python
if style_vector_dim > 0:
    import onnx
    model_onnx = onnx.load(str(args.output))
    model_onnx.metadata_props.extend([
        onnx.helper.make_attribute_proto("style_vector_dim", str(style_vector_dim)),
        onnx.helper.make_attribute_proto(
            "style_condition_mode",
            getattr(model_g, "style_condition_mode", "global"),
        ),
    ])
    onnx.save(model_onnx, str(args.output))
```

**注意**: `onnx.helper.make_attribute_proto` は AttributeProto を返すため、`metadata_props` (`StringStringEntryProto` のコレクション) と不整合。実装時は以下のいずれかに置き換え:

```python
from onnx import StringStringEntryProto
entry = StringStringEntryProto()
entry.key = "style_vector_dim"
entry.value = str(style_vector_dim)
model_onnx.metadata_props.append(entry)
```

phase-2.md のサンプルコードは誤記のため、正しい API を採用すること。

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX metadata 読み込み標準化されていない問題**: `metadata_props` は各ランタイムで読み取り方が異なる (onnxruntime-python は `session.get_modelmeta().custom_metadata_map`、Rust は `ort::Session::metadata`、C# は `InferenceSession.ModelMetadata.CustomMetadataMap`)。ランタイム側 (T02〜T07) で共通キー名 `style_vector_dim` を合意する必要があり、フォールバックとして `config.json` に同じ値を書き込む運用を採用
- **mask パターンで input が NULL 扱いされる ONNX Runtime の制約**: ONNX Runtime は `Optional` 型 input を正式サポートしていないため、本プロジェクトでは「常に入力を渡し、mask=0 で無効化」方式を採用 (speaker_embedding と同じ)。これはすべてのランタイムで `style_vector_dim` と同じサイズの zeros tensor を生成するコストが必要になるが、通常 dim=256〜512 で負担は軽微
- **`make_attribute_proto` API の誤用**: phase-2.md §2.2.3 に `onnx.helper.make_attribute_proto` の使用例があるが、これは `AttributeProto` 用 API で、`metadata_props` (型: `StringStringEntryProto`) と互換性なし。実装では `StringStringEntryProto()` を直接構築して `.append` すること
- **`model_g.style_vector_dim` 属性の存在保証**: Phase 1 で generator クラスに `style_vector_dim` 属性が必ず追加される保証がない。`getattr(..., 0)` デフォルトで防御するが、Phase 1 統合時に attribute 命名ルールを合意する必要あり

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: style_vector を metadata (constant initializer) として ONNX に埋め込む**
    - 利点: ランタイムが入力を渡す必要がなく、モデルファイル単体で style を確定できる
    - 欠点: モデルごとに 1 style に固定される。複数の style_vector を切り替える用途には不向き (fine-tune のターゲットに使えない)
- **代替案 B: `style_vector` を `speaker_embedding` に concatenate して 1 入力にまとめる**
    - 利点: 入力数が増えず、既存ランタイムの実装変更を最小化できる (shape だけ変わる)
    - 欠点: speaker_embedding と style_vector の独立性が失われる (speaker_id との排他性設計を共用する羽目になる)、後から style_condition_mode を per-frame に拡張するときに shape mismatch になる
- **代替案 C: ONNX Optional 型を正式に使う (opset 15+)**
    - 利点: mask パターンが不要になり、入力の有無を API レベルで表現できる
    - 欠点: ONNX Runtime の Optional サポートが各ランタイムで未成熟 (特に C# / Go / WASM binding)。本プロジェクトは多言語対応を重視するため現時点では非採択

### 7.2 現在の実装を選んだ理由

- `speaker_embedding` の mask パターンが 5 ランタイム全てで既に実装・テスト済みで、同じ構造を踏襲することで Agent 並列実装のテンプレ化が容易
- Phase 4 (PE-A emotion loss) では学習側が style_vector を自由に切り替える必要があり、metadata 固定型 (案A) は不適合
- opset 依存による互換性リスク (代替案C) を避けることで、既存エクスポートパイプラインへの影響を最小化

### 7.3 リファクタ機会 (将来)

- ONNX 入力追加パターンが speaker_embedding / style_vector / 将来の emotion_vector 等で反復しているため、`export_onnx.py` に `add_optional_input(name, dim, dtype, mask=True)` ヘルパー関数を切り出し、DRY 化する
- `metadata_props` 書き込み処理を `save_with_metadata(path, metadata_dict)` ユーティリティに抽出、onnx-simplifier 適用後の metadata 復元も一手で済ませる
