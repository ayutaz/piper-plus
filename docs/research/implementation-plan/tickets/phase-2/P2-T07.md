`index.js::_infer()`:

```javascript
async _infer(phonemeIds, prosodyFeatures, { noiseScale, lengthScale, noiseW, language, speakerEmbedding, styleVector }) {
    ...
    // speaker_embedding 処理 (既存)
    if (speakerEmbedding && speakerEmbedding.length > 0) {
        feeds["speaker_embedding"] = new ort.Tensor("float32", speakerEmbedding, [1, speakerEmbedding.length]);
        feeds["speaker_embedding_mask"] = new ort.Tensor("int64", [1n], [1, 1]);
    } else if (this.hasSpeakerEmbedding) {
        // has_speaker_embedding=true なのに値 null → zeros + mask=0
        const zeros = new Float32Array(this.speakerEmbeddingDim);
        feeds["speaker_embedding"] = new ort.Tensor("float32", zeros, [1, this.speakerEmbeddingDim]);
        feeds["speaker_embedding_mask"] = new ort.Tensor("int64", [0n], [1, 1]);
    }

    // style_vector 処理 (新規)
    if (this.hasStyleVector) {
        if (styleVector && styleVector instanceof Float32Array && styleVector.length === this.styleVectorDim) {
            feeds["style_vector"] = new ort.Tensor("float32", styleVector, [1, styleVector.length]);
            feeds["style_vector_mask"] = new ort.Tensor("int64", [1n], [1, 1]);
        } else {
            const zeros = new Float32Array(this.styleVectorDim);
            feeds["style_vector"] = new ort.Tensor("float32", zeros, [1, this.styleVectorDim]);
            feeds["style_vector_mask"] = new ort.Tensor("int64", [0n], [1, 1]);
        }
    }
    ...
}
```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX metadata 読み込み標準化されていない問題**: onnxruntime-web の `session.metadata()` API は v1.17+ でのみ利用可能。古いバージョンでは `session.inputNames` のみ確認できる。peer dependency の最低バージョンを `"onnxruntime-web": ">=1.17.0"` に設定
- **mask パターンで input が NULL 扱いされる ONNX Runtime の制約**: onnxruntime-web は Tensor に `null` を渡せないため、`hasStyleVector=true` のとき必ず zeros Tensor を生成
- **WASM は postMessage で大容量 Float32Array を渡すオーバーヘッド**: Web Worker を使う場合、Float32Array は構造化クローンで渡されるが、大規模 (数万要素) では遅い。将来は `Transferable` や `SharedArrayBuffer` を検討 (CORS / COOP headers が必要) — 本チケットでは 256〜512 次元の小サイズを想定し、通常の引数渡しで十分
- **`Array.from(float32Array)` のコピーコスト**: wasm-bindgen が `Vec<f32>` を受ける際、JS 側から変換が発生する。既存 speaker_embedding でも同様の処理があるため、パフォーマンス問題化していなければそのままで OK
- **ブラウザ専用制約**: 本 npm パッケージは Node.js 非対応 (eSpeak-ng 不使用で browser 前提)。テストは Node.js test runner でモックを使い、実ブラウザは別途手動確認

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: `SharedArrayBuffer` で Float32Array を共有**
    - 利点: postMessage オーバーヘッドなし、Web Worker 間で style_vector を共有できる
    - 欠点: COOP/COEP headers (`Cross-Origin-Opener-Policy: same-origin`, `Cross-Origin-Embedder-Policy: require-corp`) が必須で、多くのホスティング環境で設定が手間
- **代替案 B: `styleVector` を `base64` string で受け、ブラウザ側で Float32Array に decode**
    - 利点: HTTP API / fetch で JSON request body に直接含められる
    - 欠点: decode オーバーヘッド、256 次元 float32 (1KB) は base64 で 1.3KB と数割増
- **代替案 C: wasm-bindgen の `js_sys::Float32Array` を直接受ける (ゼロコピー)**
    - 利点: `Array.from()` 不要、メモリ効率最高
    - 欠点: wasm-bindgen 側で `&js_sys::Float32Array` として受けると、callee が to_vec() で結局コピーするケースが多い

### 7.2 現在の実装を選んだ理由

- 既存 `speakerEmbedding: Float32Array` と同じ JS 型で対称性維持
- 256〜512 次元 float32 (1〜2KB) は web では誤差レベルで最適化優先度低い
- TypeScript 型で `Float32Array` を明示することで、利用者が誤った型を渡すエラーを DTS レベルで検出

### 7.3 リファクタ機会 (将来)

- Float32Array を受ける optional input が増えた場合、`ConditioningOptions` interface を切り出して `{ speakerEmbedding, styleVector, emotionVector }` を一括管理
- `index.js` と `piper-plus.js` の役割分担明確化 (前者は低レベル API、後者は高レベル wrapper)
- WebWorker 対応: `Transferable` を使って ArrayBuffer を zero-copy で Worker に渡す
