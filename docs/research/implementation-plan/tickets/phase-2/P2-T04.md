### 2.4 style_vector 版への書き換え

```rust
// engine.rs
pub struct SynthesisRequestData {
    // ... 既存フィールド
    pub speaker_embedding: Option<Vec<f32>>,
    pub style_vector: Option<Vec<f32>>,  // 新規
}

impl Default for SynthesisRequestData {
    fn default() -> Self {
        Self {
            // ... 既存
            speaker_embedding: None,
            style_vector: None,  // 新規
        }
    }
}

pub struct ModelCapabilities {
    pub has_speaker_embedding: bool,
    pub speaker_embedding_dim: u32,
    pub has_style_vector: bool,      // 新規
    pub style_vector_dim: u32,       // 新規
}

// synthesize() 内
let style_vector_tensor = if self.capabilities.has_style_vector {
    if let Some(ref vec) = request.style_vector {
        assert_eq!(vec.len() as u32, self.capabilities.style_vector_dim,
                   "style_vector dim mismatch");
        Some(Value::from_array(([1, vec.len()], vec.as_slice()))
            .map_err(|e| PiperError::Inference(format!("style_vector tensor: {e}")))?)
    } else {
        let zeros = vec![0.0f32; self.capabilities.style_vector_dim as usize];
        Some(Value::from_array(([1, zeros.len()], zeros.as_slice()))
            .map_err(|e| PiperError::Inference(format!("style_vector tensor: {e}")))?)
    }
} else {
    None
};

let style_vector_mask_tensor = if self.capabilities.has_style_vector {
    let mask_val: i64 = if request.style_vector.is_some() { 1 } else { 0 };
    Some(Value::from_array(([1, 1], vec![mask_val].as_slice()))
        .map_err(|e| PiperError::Inference(format!("style_vector_mask tensor: {e}")))?)
} else {
    None
};

// inputs 構築 (speaker_embedding の直下)
if let Some(t) = style_vector_tensor {
    inputs.push(("style_vector".into(), t.into()));
}
if let Some(t) = style_vector_mask_tensor {
    inputs.push(("style_vector_mask".into(), t.into()));
}
```

```rust
// main()
let style_vec = if let Some(path) = &args.style_vector {
    Some(load_f32_npy(path)?)
} else {
    None
};
request.style_vector = style_vec;

fn load_f32_npy(path: &Path) -> Result<Vec<f32>> {
    use ndarray_npy::read_npy;
    let arr: ndarray::Array1<f32> = read_npy(path)
        .with_context(|| format!("Failed to read .npy: {}", path.display()))?;
    Ok(arr.to_vec())
}
```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX metadata 読み込み標準化されていない問題**: `ort` crate (Rust ORT binding) の `Session::metadata()` API はバージョン差あり。`ort = "2.0"` で `session.metadata()?.custom_metadata_iter()` を使うが、`2.0-rc` 系と `2.0.0` で微妙に異なる。crate バージョンを固定
- **mask パターンで input が NULL 扱いされる ONNX Runtime の制約**: `Option<Value>` で None の場合 `inputs.push()` をスキップする必要あり。`has_style_vector=true` のときのみ push
- **`ndarray-npy` の big-endian 問題**: Python から書いた `.npy` が little-endian なら問題ないが、big-endian system から書かれたファイルは dtype 不一致エラー。`f32le` の magic number チェック必須
- **PyO3 の `Vec<f32>` 受け渡しオーバーヘッド**: Python から 256 要素程度なら無視可能だが、per-frame モード (style_condition_mode="per_frame") で数千要素になったとき GIL release でコピーが発生する可能性。`numpy::PyArray1<f32>` を直接受ける実装に将来変更
- **`piper-wasm` への影響**: T07 で扱うため本チケットでは対象外だが、`SynthesisRequestData` の変更は wasm crate でも共通。wasm-bindgen の `impl` に `Option<Vec<f32>>` をそのまま渡せないため T07 側で Wrapper 実装が必要

### 6.2 レビュー項目

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: `SynthesisRequestData` ではなく Builder pattern で `SynthesisRequest::builder().style_vector(vec).build()` に切替**
    - 利点: optional fields が増えても API が膨張しない、公開 struct のフィールド露出なし
    - 欠点: 既存 `SynthesisRequestData::default()` パターンとの後方互換性が失われる
- **代替案 B: `style_vector` を `numpy::PyArray1<f32>` で PyO3 に直接渡す (ゼロコピー)**
    - 利点: Python から大容量 Float32Array を渡す際のオーバーヘッド削減
    - 欠点: `numpy` crate 依存が追加、wasm binding では使えない (cfg 分岐が増える)
- **代替案 C: style_vector を rust 側 struct ではなく、呼び出し時の builder に置く**
    - 利点: struct layout が固定され、ABI stability が向上
    - 欠点: 呼び出し箇所 (CLI, PyO3, WASM) すべてで builder 呼び出しが必要で boilerplate 増

### 7.2 現在の実装を選んだ理由

- 既存 `speaker_embedding: Option<Vec<f32>>` と同じ構造で対称性維持、レビュー負荷最小
- `Option<Vec<f32>>` は PyO3 / wasm-bindgen 両対応 (ゼロコピーではないが 256 次元程度なら問題なし)
- `SynthesisRequestData` の public フィールドは本プロジェクトでは意図的設計 (test fixture から直接構築する場面多し)

### 7.3 リファクタ機会 (将来)

- `Option<Vec<f32>>` が `Option<&[f32]>` や `Option<Cow<[f32]>>` に置き換え可能か検討 (アロケーション削減)
- `ModelCapabilities::from_session()` が speaker_embedding, style_vector, 将来の emotion_vector で反復する構造。`detect_optional_input(session, name) -> Option<(bool, u32)>` ヘルパーに抽出
