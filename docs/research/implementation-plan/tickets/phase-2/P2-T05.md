### 2.2 実装手順

1. `PiperModel` constructor で `_session.InputMetadata.ContainsKey("style_vector")` により検出、`HasStyleVector` プロパティに格納
2. `_session.ModelMetadata.CustomMetadataMap` (`IReadOnlyDictionary<string, string>`) から `"style_vector_dim"` を取得、なければ `config.json` (PiperModelConfig) の `StyleVectorDim` を参照
3. `SynthesisInput` record に `StyleVector` プロパティ追加 (既存 `SpeakerEmbedding` と同じ位置)
4. `PiperSession.RunInference()` 内で、speaker_embedding の OrtValue 構築処理の直下に style_vector 版を追加
5. `NumpyLoader.LoadFloat32Array(path)` を実装 (`.npy` magic header `\x93NUMPY` + shape + data 読込)
6. CLI `Program.cs` に `--style-vector` オプション追加、パース時に `NumpyLoader.LoadFloat32Array()` 呼び出し、`SynthesisInput.StyleVector` に代入
7. xUnit テスト新規追加

### 2.4 style_vector 版への書き換え

`PiperModel.cs`:

```csharp
public bool HasStyleVector { get; }
public int StyleVectorDim { get; }

public PiperModel(...)
{
    // ... 既存
    HasStyleVector = _session.InputMetadata.ContainsKey("style_vector");

    // metadata → config → 0 の順で解決
    var metadata = _session.ModelMetadata.CustomMetadataMap;
    if (metadata.TryGetValue("style_vector_dim", out var dimStr)
        && int.TryParse(dimStr, out var dimFromMeta))
    {
        StyleVectorDim = dimFromMeta;
    }
    else
    {
        StyleVectorDim = config.StyleVectorDim ?? 0;
    }
}
```

`PiperSession.cs`:

```csharp
public sealed record SynthesisInput(
    ...,
    float[]? SpeakerEmbedding = null,
    float[]? StyleVector = null  // 新規
);

// RunInference() 内 (speaker_embedding の直下)
if (_model.HasStyleVector)
{
    if (input.StyleVector is not null && input.StyleVector.Length > 0)
    {
        if (input.StyleVector.Length != _model.StyleVectorDim)
        {
            throw new ArgumentException(
                $"StyleVector length {input.StyleVector.Length} != expected {_model.StyleVectorDim}");
        }
        var styleTensor = OrtValue.CreateTensorValueFromMemory(
            input.StyleVector, new long[] { 1, input.StyleVector.Length });
        inputs["style_vector"] = styleTensor;
        inputs["style_vector_mask"] = OrtValue.CreateTensorValueFromMemory(
            new long[] { 1 }, new long[] { 1, 1 });
    }
    else
    {
        var zeros = new float[_model.StyleVectorDim];
        inputs["style_vector"] = OrtValue.CreateTensorValueFromMemory(
            zeros, new long[] { 1, zeros.Length });
        inputs["style_vector_mask"] = OrtValue.CreateTensorValueFromMemory(
            new long[] { 0 }, new long[] { 1, 1 });
    }
}
```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- **ONNX metadata 読み込み標準化されていない問題**: `ModelMetadata.CustomMetadataMap` は `IReadOnlyDictionary<string, string>` だが、古い onnxruntime (< 1.16) では `Dictionary<string, string>` を返す。`TryGetValue` 経由で正常動作するが、型アサーションは避ける
- **mask パターンで input が NULL 扱いされる ONNX Runtime の制約**: `OrtValue` は `null` を inputs dictionary に追加できない。`has_style_vector=true` のとき必ず zeros tensor を生成
- **`OrtValue.CreateTensorValueFromMemory` の lifetime**: `float[]` が GC で回収される前に `_session.Run()` が完了する必要。`using var` pattern で明示管理 (既存 speaker_embedding コードでの慣習を踏襲)
- **.npy フォーマット parser の complete/incomplete**: NuGet の `NumSharp` / `NumpyDotNet` 等外部依存が大きい。自前実装する場合は magic header `\x93NUMPY` + version 1.0/2.0 + dtype `<f4` (little-endian float32) のみサポートで十分
- **net8.0 / net9.0 TFM 両方でのビルド**: `PiperPlus.Core` は net8.0 (LTS)、`PiperPlus.Cli` は net9.0。`Span<T>` 等新 API の使用は Core 側で控える

### 6.2 レビュー項目

- [ ] `SynthesisInput` record の `StyleVector` プロパティが `init` アクセス (`float[]? StyleVector = null`)
- [ ] `OrtValue` の dispose が RunInference の after で確実に行われる (既存 speaker_embedding の paterrn を踏襲)
- [ ] `NumpyLoader` が `<f4` (little-endian float32) のみサポートし、それ以外の dtype で例外を出す
- [ ] `--style-vector` オプションに `FileInfo?` 型を使用、存在チェックは handler 側
- [ ] ~1005 → ~1010 テスト数に増加、既存テスト回帰なし

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A: `SynthesisInput` ではなく builder pattern (`SynthesisInputBuilder`)**
    - 利点: optional fields が増えても ctor 爆発しない
    - 欠点: record の positional syntax の簡潔さが失われる、既存コード全面書き換え
- **代替案 B: `float[]` ではなく `ReadOnlyMemory<float>` / `Span<float>` を受ける**
    - 利点: ゼロコピー、大規模 vector でメモリ効率向上
    - 欠点: record の property として `Span<T>` は使えない (ref struct 制約)、API 複雑化
- **代替案 C: `.npy` ではなく `.bin` (raw float32 LE) 独自フォーマット**
    - 利点: parser 実装コスト削減
    - 欠点: Python / Rust との相互運用性失われる、標準フォーマットの恩恵なし

### 7.2 現在の実装を選んだ理由

- 既存 `SpeakerEmbedding` プロパティと同じ pattern で対称性維持、レビュー容易
- `float[]` は .NET 標準型でシリアライザ (JSON, Protobuf 等) 対応が容易
- `.npy` は Python/Rust と共通で cross-runtime byte-for-byte 互換性テスト (T08) に有利

### 7.3 リファクタ機会 (将来)

- `OrtValue.CreateTensorValueFromMemory` の呼び出しパターンを `TensorFactory.CreateFloat32([shape], data)` ヘルパーに抽出
- `PiperModel` の capability 検出ロジック (HasSpeakerEmbedding, HasStyleVector, 将来の HasEmotionVector) を `OptionalInputDetector` クラスに抽出
