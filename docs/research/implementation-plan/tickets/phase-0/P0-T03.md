## 2. 実装内容の詳細

### 2.1 対象ファイル

- `src/python/piper_train/tools/test_pe_av_small.py` (`benchmark()` + `main()` 整備)
- `docs/research/peav-style-conditioning.md` (実測値追記、または本チケットの §10)

### 2.2 実装手順

1. `benchmark(model, n_runs=5)` を定義
    - warmup 1 回: dummy audio で 1 回推論を行い JIT/kernel 初期化を済ませる
    - `torch.cuda.reset_peak_memory_stats(device)` を呼び、メモリ計測を初期化 (CUDA のみ)
    - `n_runs` 回、`time.perf_counter()` で推論を timing しリストに記録
    - `avg = sum(times) / len(times)` を計算
    - CUDA の場合は `torch.cuda.max_memory_allocated(device) / 1024**2` を MB 単位で取得
2. `main()` を整備
    - `logging.basicConfig(...)` で INFO 出力
    - `[Test 1] Loading model` → `test_loading()`
    - `[Test 2] Running inference` → `test_inference(model)` (T02 由来)
    - `[Test 3] Benchmarking` → `benchmark(model, n_runs=5)`
    - 最終行: `Summary: method=<...>, embedding_dim=<...>, avg_latency=<...>ms, peak_gpu_mem=<...>MB`
3. 計測結果を `docs/research/peav-style-conditioning.md §12` (ライセンス・可用性章の末尾) もしくは本チケットの §10 に以下の表形式で追記:

    | 項目 | 仮説 (phase-0-1.md §0.6) | 実測 | 乖離有無 |
    |------|-------|-----|---------|
    | embedding dim | 256 or 512 | - | - |
    | 推論時間 (16kHz, 3s) | 40-80 ms | - | - |
    | GPU メモリ (推論) | 200-500 MB | - | - |

### 2.3 コード例 (抜粋)

```python
import time
import torch

_LOGGER = logging.getLogger(__name__)


def benchmark(model: torch.nn.Module, n_runs: int = 5) -> dict:
    device = next(model.parameters()).device
    sample_rate = 16000
    duration_s = 3

    # Warmup 1 回
    audio = torch.randn(1, sample_rate * duration_s, device=device)
    with torch.no_grad():
        if hasattr(model, "get_audio_embeds"):
            _ = model.get_audio_embeds(audio)
        else:
            _ = model(audio)

    if device.type == "cuda":
        torch.cuda.reset_peak_memory_stats(device)
        torch.cuda.synchronize(device)

    times = []
    for i in range(n_runs):
        audio = torch.randn(1, sample_rate * duration_s, device=device)
        if device.type == "cuda":
            torch.cuda.synchronize(device)
        start = time.perf_counter()
        with torch.no_grad():
            if hasattr(model, "get_audio_embeds"):
                _ = model.get_audio_embeds(audio)
            else:
                _ = model(audio)
        if device.type == "cuda":
            torch.cuda.synchronize(device)
        end = time.perf_counter()
        elapsed_ms = (end - start) * 1000
        times.append(elapsed_ms)
        _LOGGER.info("Run %d: %.2f ms", i + 1, elapsed_ms)

    avg = sum(times) / len(times)
    _LOGGER.info("Average latency: %.2f ms (n=%d)", avg, n_runs)

    peak_gpu_mem_mb = None
    if device.type == "cuda":
        peak_gpu_mem_mb = torch.cuda.max_memory_allocated(device) / 1024**2
        _LOGGER.info("Peak GPU memory: %.1f MB", peak_gpu_mem_mb)

    return {"avg_ms": avg, "peak_gpu_mem_mb": peak_gpu_mem_mb, "times_ms": times}


def main():
    logging.basicConfig(
        level=logging.INFO,
        format="%(asctime)s - %(name)s - %(levelname)s - %(message)s",
    )
    _LOGGER.info("=== Phase 0 PoC: facebook/pe-av-small ===")
    _LOGGER.info("[Test 1] Loading model...")
    model = test_loading()
    _LOGGER.info("[Test 2] Running inference...")
    embeddings, method = test_inference(model)
    _LOGGER.info("[Test 3] Benchmarking...")
    stats = benchmark(model, n_runs=5)
    _LOGGER.info(
        "Summary: method=%s, embedding_dim=%d, avg_latency=%.2fms, peak_gpu_mem=%s",
        method,
        embeddings.size(-1),
        stats["avg_ms"],
        f"{stats['peak_gpu_mem_mb']:.1f}MB" if stats["peak_gpu_mem_mb"] is not None else "n/a (cpu)",
    )


if __name__ == "__main__":
    main()
```

## 6. 懸念事項とレビュー項目

### 6.1 懸念事項

- CPU 実行時は GPU メモリが測れない (`peak_gpu_mem=n/a (cpu)`)。Phase 0 完了条件としては CPU 実行でも可とするが、Phase 5 の fine-tune 判断には GPU 実測が必須
- warmup 1 回では JIT が安定しないケース (特に transformers の trust_remote_code 版) があり得る。測定ばらつき (stdev) が avg の 30% を超える場合は warmup を 3 回に増やすことを検討
- `time.perf_counter()` のみの測定では、CUDA 非同期実行の見落としが発生する。`torch.cuda.synchronize()` を必ず前後に挟むことを Reviewer が確認
- 仮説 (`phase-0-1.md §0.6`) と実測値に大きな乖離 (例: latency が 40-80ms 想定に対して実測 500ms) が出た場合、Phase 4 の `--pea-emotion-every-n-steps` デフォルト値見直しが必要

## 7. 一から作り直すとしたら

### 7.1 代替案の検討

- **代替案 A**: PoC を Python ではなく shell script (`huggingface-cli download` + `python -c "..."`) で最小化
    - 利点: 依存が薄い、CI で再現しやすい
    - 欠点: ベンチマークや L2 normalize 検証が書きにくく、Phase 4 の loader 流用ができない
- **代替案 B**: ベンチマークをスキップし、Phase 4 の実学習ログで初めて速度・メモリを測る
    - 利点: 15-30分の工数削減
    - 欠点: Phase 4 で初めてボトルネックが判明するとロールバック工数が大きい。Phase 0 で事実を固めるほうが総工数は小さい
- **代替案 C**: `torch.profiler` を使った詳細プロファイリング
    - 利点: レイヤ単位のボトルネック把握
    - 欠点: Phase 0 のスコープを超える。Phase 4 で学習速度に問題が出た時点で実施すればよい

### 7.2 現在の実装を選んだ理由

- 15-30分という工数目安に収めつつ、Phase 1 / 3 / 4 / 5 すべての設計決定に必要な最小情報 (embedding dim、latency、GPU memory) を取得する必要がある
- `time.perf_counter` + CUDA `synchronize` + warmup 1 回の組合せは、transformers モデルの PoC ベンチマークで業界標準的に使われている最小構成
- 結果を `peav-style-conditioning.md` に追記することで、後続 Phase のチケット作成時に再調査不要となる

### 7.3 手抜き可能ポイント (工数圧縮時)

- CPU 実行のみに限定すれば GPU 環境セットアップが不要で 5 分短縮できる (ただし Phase 5 判断材料が欠ける)
- `n_runs=5` を `n_runs=3` に縮小すれば 1-2 分短縮 (統計的信頼性は落ちる)
- レポート追記を省略し、stdout ログを PR 本文に貼るだけで済ませる (推奨しない: Phase 3/4 で参照しづらい)

### 7.4 リファクタ機会 (将来)

- Phase 4 完了後、`benchmark()` を `src/python/piper_train/tools/bench_pe_av.py` として独立 CLI 化し、CI で継続回帰 (latency 悪化検知) を可能にする
- CPU / CUDA / MPS の 3 環境でのベンチを標準化する Makefile ターゲットを追加
