# Issue #109 実装計画の批判的レビュー結果

## 🔴 重大な問題点

### 1. VITSアーキテクチャの根本的な誤解
- **誤解**: アテンション行列から音素タイミングを抽出できる
- **実際**: VITSの"attention"は**Duration Predictor**による決定論的なアライメントパス
- `generate_path()`は音素の継続時間（duration）から生成される二値行列

### 2. 既存ONNXモデルとの互換性
- **致命的問題**: 現在のONNXモデルは音声データのみ出力
- 全ての既存モデル（30言語以上）の再エクスポートが必要
- コミュニティが作成したカスタムモデルが動作しなくなる

### 3. パフォーマンスへの影響
- アテンション行列のサイズ: `[batch, 1, 音声フレーム数, 音素数]`
- 5秒音声の例: 約430フレーム × 50音素 = 21,500個のfloat値
- メモリ使用量: 10-20%増加
- ストリーミングモードでの影響が特に大きい

## 🟡 技術的な課題

### 1. 精度の限界
- フレームレベルの粒度（256サンプル = 約11.6ms @22050Hz）
- サブ音素情報なし
- 音素境界の曖昧さ

### 2. 量子化モデルへの対応
- 多くのPiperモデルはINT8量子化済み
- アテンション行列はFP32を維持する必要
- モデルサイズと推論速度への影響

## 🟢 代替アプローチの提案

### アプローチ1: Duration Predictorの直接利用（推奨）
```python
# VITSモデル内部
logw = self.dp(x, x_mask, g=g)  # Duration Predictor出力
w = torch.exp(logw) * x_mask * length_scale  # 各音素の継続時間
```

**利点**:
- 既にモデル内で計算されている
- 追加のメモリ不要
- 音素の継続時間を直接取得

**実装例**:
```cpp
// 音素の継続時間から開始・終了時刻を計算
float current_time = 0.0;
for (int i = 0; i < phoneme_count; ++i) {
    float duration = durations[i] * frame_length / sample_rate;
    phoneme_timings[i] = {
        .phoneme = phonemes[i],
        .start = current_time,
        .end = current_time + duration
    };
    current_time += duration;
}
```

### アプローチ2: 後処理による強制アライメント
- Montreal Forced Aligner等の既存ツール使用
- 生成音声とテキストから高精度なアライメント
- 追加処理時間が必要だが、最も正確

### アプローチ3: ハイブリッドアプローチ
1. Duration Predictorで大まかなタイミング
2. エネルギーベースのセグメンテーションで境界調整
3. 言語別の後処理
