# C++ / Python 実装差分調査レポート

> **調査日**: 2026-03-08
> **対象ブランチ**: `dev`
> **調査方法**: 15エージェント並行調査 + Docker推論実測比較
> **検証モデル**: `ayousanz/piper-plus-tsukuyomi-chan` (tsukuyomi-wavlm-300epoch.onnx)

---

## エグゼクティブサマリー

Python（学習・前処理）とC++（推論）の実装差分を包括的に調査した結果、**C++推論パイプラインがPython側の最新機能に追従できていない箇所が複数発見された**。特に Issue #204（疑問詞マーカー）と Issue #207（文脈依存Nバリアント）に関する音素化処理がC++側で未実装であり、これらの新トークンで学習されたモデルをC++で推論した場合、**音質劣化や意図しない発音**が発生する可能性がある。

Docker推論による実測比較（8テストケース）では、**全テストケースでphoneme_ids列が不一致**であり、C++は平均3.1トークン少ないことが確認された。詳細は [セクション11](#11-docker-推論比較結果実測データ) を参照。

### 重要度別差分一覧

| 重要度 | 差分 | 影響 |
|--------|------|------|
| **Critical** | C++ openjtalk_phonemize に疑問詞マーカー未実装 | 疑問文の抑揚が正しく生成されない |
| **Critical** | C++ openjtalk_phonemize にNバリアント未実装 | 「ん」の発音が文脈非依存になる |
| **Critical** | C++ openjtalk_phonemize にプロソディマーク（`[`, `]`, `#`）の挿入ロジックが**未実装**（実測確認済み） | アクセント情報が欠落し韻律が劣化 |
| **Medium** | C++ JSON入力で prosody_features を外部指定不可 | 事前計算済みprosodyを渡せない |
| **Medium** | C++ カスタム辞書の日本語単語境界処理にバグの可能性 | 日本語辞書エントリのマッチ失敗 |
| **Low** | C++ puaToPhoneme（ログ表示用）に新PUA未登録 | ログ表示のみ影響、推論には無影響 |
| **Info** | prosody_features テンソル型の整合性 | ONNX Cast ノードで吸収済み |

## 1. 音素化パイプラインの差分（最重要）

### 1.1 疑問詞マーカー（Issue #204）— C++ 未対応

**Python側の実装** (`japanese.py:40-78`):

テキスト末尾のパターンから5種類の疑問詞を区別:

| マーカー | ID | PUA | テキストパターン |
|---------|-----|-----|----------------|
| `$` (平叙文) | 2 | なし | 疑問符なし |
| `?` (汎用疑問) | 3 | なし | `?`, `？` |
| `?!` (強調疑問) | 4 | U+E016 | `?!`, `！？`, `？！` |
| `?.` (平叙疑問) | 5 | U+E017 | `?.`, `。？`, `？。` |
| `?~` (確認疑問) | 6 | U+E018 | `?~`, `～？`, `？～` |

**C++側の状況**:

- `openjtalk_phonemize.cpp` の `phonemeToPua` マップに `?!`, `?.`, `?~` のエントリが**存在しない**
- OpenJTalkは末尾の `sil` ラベルのみ返すため、テキスト末尾の句読点パターン解析ロジック自体がC++側に**実装されていない**
- C++は疑問文の終端を常に一律で処理（`$` または `?` のいずれか）

**影響**: 新しい疑問詞マーカー付きで学習されたモデルをC++で推論すると、ID 4-6 のトークンが生成されず、疑問文の種類に応じた抑揚の違いが失われる。

**phoneme_parser テストでの矛盾**: `test_phoneme_parser.cpp` には `?!` → 0xE016, `?.` → 0xE017, `?~` → 0xE018 のテストが存在するが、これは `parsePhonemeString()` （`--raw-phonemes` 入力時のパーサー）のテストであり、テキスト→音素の自動変換パイプラインには適用されない。つまり、**パーサーは対応済みだが音素化ロジックが未実装**。

### 1.2 文脈依存「ん」バリアント（Issue #207）— C++ 未対応

**Python側の実装** (`japanese.py:85-130`):

`_apply_n_phoneme_rules()` で後続音素に基づき `N` を4種に分類:

| バリアント | ID | PUA | 条件（後続音素） |
|-----------|-----|-----|----------------|
| `N_m` | 26 | U+E019 | m, my, b, by, p, py |
| `N_n` | 27 | U+E01A | n, ny, t, ty, d, dy, ts, ch |
| `N_ng` | 28 | U+E01B | k, ky, kw, g, gy, gw |
| `N_uvular` | 29 | U+E01C | 語末 / 母音 / その他 |

**C++側の状況**:

- `openjtalk_phonemize.cpp` の `phonemeToPua` マップに `N_m`, `N_n`, `N_ng`, `N_uvular` のエントリが**存在しない**
- OpenJTalkは「ん」を常に `N` として出力するため、後続音素に基づく分類ロジック自体がC++側に**実装されていない**
- `phoneme_parser` のテストには `N_m` → 0xE019 等のテストが存在するが、上記と同様 `--raw-phonemes` 入力時のみ有効

**影響**: v2データセット（Issue #207対応済み）で学習されたモデルをC++で推論すると、「ん」が常に汎用 `N`（ID 25）として処理され、文脈に応じた「ん」の発音の区別が失われる。

### 1.3 プロソディマーク挿入（栗原方式）— C++ 未実装（実測確認済み）

**Python側の実装** (`japanese.py:222-240`):

OpenJTalkラベルのA2値を先読みして3種のプロソディマークを挿入:

| 条件 | マーク | ID | 意味 |
|------|--------|-----|------|
| `a1 == 0 and a2_next == a2 + 1` | `]` | 9 | アクセント核（ピッチ下降点） |
| `a2 == a3 and a2_next == 1` | `#` | 7 | アクセント句境界 |
| `a2 == 1 and a2_next == 2` | `[` | 8 | ピッチ上昇点（句頭） |

**C++側の状況（Docker実測で確認）**:

- `openjtalk_wrapper.c` の `read_and_parse_output_with_prosody()` はA1/A2/A3を抽出しているが、**プロソディマーク（`[`, `]`, `#`）を音素列に挿入するロジックは未実装**
- 実測例: 「こんにちは、今日は良い天気ですね。」で Python は 38 トークン（`[` × 4, `]` × 3 を含む）だが C++ は 31 トークン（プロソディマークなし）
- 全8テストケースで C++ にプロソディマークが一切含まれないことを確認

**影響**: プロソディマークが欠落すると、アクセント句の境界やピッチの上昇・下降ポイントの情報が失われ、自然な韻律が生成されない。全テストケースで影響が確認されている。

## 4. BOS/EOS/パディング処理の差分

### 4.1 日本語

| 処理 | C++ | Python | 一致 |
|------|-----|--------|:---:|
| BOS挿入 | `sil` → 音素列には含めず、`phonemes_to_ids` でBOS(1)追加 | `sil` → `^`(BOS) としてトークン列に含む | ⚠️ 要確認 |

## 7. 修正推奨事項

### Priority 1: Critical（モデル品質に直接影響）

#### 7.1 疑問詞マーカーの実装

**対象ファイル**: `src/cpp/openjtalk_phonemize.cpp`

必要な変更:
1. `phonemeToPua` マップに3エントリ追加:
   ```cpp
   {"?!", 0xE016}, {"?.", 0xE017}, {"?~", 0xE018}
   ```
2. テキスト末尾の句読点パターン解析ロジックを実装（Python `_get_question_type()` と同等）
3. 末尾 `sil` の変換時に疑問詞タイプに応じたトークンを生成

#### 7.2 文脈依存Nバリアントの実装

**対象ファイル**: `src/cpp/openjtalk_phonemize.cpp`

必要な変更:
1. `phonemeToPua` マップに4エントリ追加:
   ```cpp
   {"N_m", 0xE019}, {"N_n", 0xE01A}, {"N_ng", 0xE01B}, {"N_uvular", 0xE01C}
   ```
2. `_apply_n_phoneme_rules()` と同等の後続音素に基づくN分類ロジックを実装
3. 分類ルール:
   - `N_m`: 後続が m, my, b, by, p, py
   - `N_n`: 後続が n, ny, t, ty, d, dy, ts, ch
   - `N_ng`: 後続が k, ky, kw, g, gy, gw
   - `N_uvular`: 語末 / 母音 / その他

#### 7.3 プロソディマーク挿入の実装

**対象ファイル**: `src/cpp/openjtalk_phonemize.cpp`

**実測で未実装が確認済み。** Python `japanese.py:222-240` の栗原方式ロジックを移植する必要がある:
1. A2値の先読み（次の音素のA2値を参照）
2. 条件に応じて `[`(ID=8), `]`(ID=9), `#`(ID=7) を音素列に挿入

```
  │     ├── sil → ^(BOS) / $/?/?!/?./?(EOS)
  │     ├── pau → _
  │     └── プロソディマーク挿入 ([, ], #)  ← 栗原方式
  │
  ├── _apply_n_phoneme_rules()  ← N → N_m/N_n/N_ng/N_uvular
  │
  ├── map_sequence()  ← 多文字トークン → PUA文字
  │
  ├── phoneme_id_map で ID変換
  │
  ├── post_process_ids()  ← 英語のみBOS/EOS/PAD挿入
  │
  └── ONNX Runtime 推論
        input, input_lengths, scales, [sid], [prosody_features]
```

## 10. ONNXモデルの入出力仕様（リファレンス）

### 入力

| 名前 | 型 | 形状 | dynamic_axes | 条件 |
|------|-----|------|-------------|------|
| `input` | int64 | `[B, T]` | B, T | 常に |
| `input_lengths` | int64 | `[B]` | B | 常に |
| `scales` | float32 | `[3]` | なし | 常に |
| `sid` | int64 | `[B]` | B | num_speakers > 1 |
| `prosody_features` | int64 | `[B, T, 3]` | B, T | prosody_dim > 0 |

### 出力

| 名前 | 型 | 形状 | 説明 |
|------|-----|------|------|
| `output` | float32 | `[B, 1, time]` | 音声波形 |
| `durations` | float32 | `[B, T]` | 各音素のデュレーション（フレーム数） |

### モデル内部処理

```
input (phoneme_ids) → TextEncoder → x, m_p, logs_p
prosody_features → Linear(3→16) → prosody_proj
x + prosody_proj → Duration Predictor → logw → w
w × length_scale → alignment → attn
attn × (m_p, logs_p) → z_p
z_p → Flow(reverse) → z
z → Generator → audio
```
