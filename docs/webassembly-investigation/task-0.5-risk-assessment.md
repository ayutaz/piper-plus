# Task 0.5 リスク評価書

作成日: 2025-07-21

## 概要

Piper-plus WebAssembly実装プロジェクトの技術的・運用的リスクを評価し、緩和策を提示します。

## リスク評価マトリクス

### リスクレベル定義
- **影響度**: 低(1) / 中(2) / 高(3) / 致命的(4)
- **発生確率**: 低(1) / 中(2) / 高(3) / ほぼ確実(4)
- **リスクスコア**: 影響度 × 発生確率

## 技術リスク

### 1. 辞書圧縮の技術的限界

| 項目 | 内容 |
|------|------|
| **説明** | 103MB → 2-3MBの圧縮目標が達成できない可能性 |
| **影響度** | 高(3) |
| **発生確率** | 中(2) |
| **リスクスコア** | 6 |
| **影響** | - 初期ロード時間の増大<br>- メモリ使用量の増加<br>- Unity WebGL統合の困難化 |
| **緩和策** | 1. 段階的圧縮アプローチ採用<br>2. 最小限辞書（頻出5000語）での初期リリース<br>3. 差分ロードシステムの実装<br>4. CDN配信による高速化 |
| **代替案** | サーバーサイドAPI併用によるハイブリッド方式 |

### 2. Unity WebGLメモリ制限

| 項目 | 内容 |
|------|------|
| **説明** | Unity WebGLの256MBメモリ制限を超過するリスク |
| **影響度** | 高(3) |
| **発生確率** | 低(1) |
| **リスクスコア** | 3 |
| **影響** | - Unity統合不可<br>- ユーザー体験の低下 |
| **緩和策** | 1. 現在43MB使用（余裕213MB）<br>2. メモリプロファイリング継続<br>3. 不要データの積極的解放<br>4. Unity Memory Profiler活用 |
| **監視項目** | - ヒープ使用量<br>- ピークメモリ<br>- GC頻度 |

### 3. ブラウザ互換性の制限

| 項目 | 内容 |
|------|------|
| **説明** | Chrome限定によるユーザーベース制限 |
| **影響度** | 中(2) |
| **発生確率** | ほぼ確実(4) |
| **リスクスコア** | 8 |
| **影響** | - 35%のユーザーが利用不可<br>- Safari/Firefox未対応 |
| **緩和策** | 1. 初期はChrome限定で品質確保<br>2. 段階的な他ブラウザ対応<br>3. フォールバック実装（サーバーAPI）<br>4. ユーザーへの明確な案内 |
| **将来計画** | 6ヶ月後に他ブラウザ対応評価 |

### 4. ONNX Runtime Web初期サイズ

| 項目 | 内容 |
|------|------|
| **説明** | ONNX Runtime Webの初期ロードサイズが大きい（約10MB） |
| **影響度** | 中(2) |
| **発生確率** | 高(3) |
| **リスクスコア** | 6 |
| **影響** | - 初期ロード時間増加<br>- ユーザー離脱率上昇 |

## リスク対応戦略

### 即時対応必要（スコア6以上）

1. **ブラウザ互換性**
   - Chrome限定を明確に文書化
   - フォールバックAPIの設計開始

2. **辞書圧縮**
   - Phase 1実装を優先
   - 圧縮アルゴリズムのプロトタイプ作成

3. **ONNX Runtime**
   - 遅延ロード実装を初期から組み込み
   - CDN戦略の確定

### 継続監視項目

- メモリ使用量（週次レポート）
- パフォーマンスメトリクス（自動計測）
- エラー率（Sentryなどで監視）
- ユーザーフィードバック

## コンティンジェンシープラン

### シナリオ1: 辞書圧縮失敗

```
条件: 3ヶ月後に10MB以下達成不可
対応:
1. サーバーサイドAPIハイブリッド実装
2. 頻出語限定版での初期リリース
3. Progressive Enhancement戦略
```

### シナリオ2: Unity統合不可

```
条件: メモリ制限超過
対応:
1. スタンドアロンWebアプリとして提供
2. Unity → Web通信方式に切り替え
3. 軽量版モデルの開発
```

## 結論

主要なリスクは特定され、すべて管理可能なレベルです。最大のリスクであるブラウザ互換性は、プロジェクト方針として受容済みです。他の技術的リスクは、段階的アプローチと既存実装の活用により緩和可能です。

定期的なリスクレビュー（月次）を実施し、必要に応じて対応策を更新することを推奨します。
