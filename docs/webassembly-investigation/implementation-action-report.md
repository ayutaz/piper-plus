# Piper WebAssembly実装 - 対応計画レポート

作成日: 2025-07-21
作成者: Claude (Anthropic)

## エグゼクティブサマリー

Piper-plusのWebAssembly対応（Issue #106）について、包括的な技術調査と計画策定を完了しました。日本語TTSを最優先とし、Unity WebGLでの動作を前提とした10-12週間の実装計画を策定しました。技術的実現可能性は高く、段階的なアプローチにより確実な実装が可能です。

## 1. プロジェクト概要

### 目的
- ブラウザ内での日本語音声合成（TTS）実現
- Unity WebGLゲームでの音声合成機能提供
- オフライン動作によるプライバシー保護とレスポンス向上

### 主要成果物
1. **npmパッケージ**: `@piper-tts/wasm`
2. **Unity Package**: `com.piper-tts.webgl`
3. **デモアプリケーション**: Web版、Unity WebGL版

## 2. 技術的アプローチ

### 2.1 アーキテクチャ概要
```
ブラウザ/Unity WebGL
    ↓
JavaScript/TypeScript API
    ↓
WebAssembly モジュール
├── MeCab (形態素解析)
├── OpenJTalk (音素化)
└── ONNX Runtime Web (音声合成)
    ↓
Web Audio API (音声出力)
```

### 2.2 技術スタック
| コンポーネント | 技術選定 | 理由 |
|---------------|---------|------|
| 音素化 | OpenJTalk + MeCab | 日本語精度の高さ |
| 音声合成 | ONNX Runtime Web | WebGPU対応、高速推論 |
| 音声出力 | AudioWorklet | 低レイテンシ再生 |
| ビルドツール | Emscripten 3.1.61+ | 最新のWebAssembly機能 |

### 2.3 主要な技術課題と対策

#### 辞書サイズ問題
- **課題**: 現状103MB → Web配信には大きすぎる
- **対策**: 段階的最適化
  - Phase 1: 10MB（基本語彙10,000語）
  - Phase 2: 5MB（頻出5,000語）
  - Phase 3: 2-3MB（コア3,000語）

#### Unity WebGLメモリ制限
- **課題**: 256MBヒープ制限
- **対策**: 
  - メモリプール実装
  - AudioClipキャッシング
  - Web Worker活用

## 4. リスク管理

### 高リスク項目

#### 1. OpenJTalk移植の複雑さ
- **影響**: スケジュール遅延
- **確率**: 中
- **対策**: 
  - Phase 0での早期検証
  - 既存実装の調査
  - 代替案（簡易音素化）の準備

#### 2. 辞書サイズ目標未達
- **影響**: ユーザビリティ低下
- **確率**: 中
- **対策**:
  - 段階的目標設定
  - 品質とサイズのトレードオフ明確化
  - CDNフォールバック実装

#### 3. Unity WebGLメモリ超過
- **影響**: 動作不能
- **確率**: 低
- **対策**:
  - 継続的プロファイリング
  - メモリプール実装
  - 軽量モード提供
